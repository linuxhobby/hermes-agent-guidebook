## 1. 首次运行：Quick Setup 向导

### 1.1 启动 hermes 命令

安装完成后，在终端输入：

```bash
hermes
```

如果这是你的第一次运行，Hermes 会自动启动 Quick Setup 向导。这个向导就像是新手机的「开机设置」，会引导你完成最核心的几项配置。

> **Tips**
>
> 你也可以随时通过 `hermes setup` 命令重新运行配置向导。比如你换了新的 API Key，或者想切换默认模型，都可以用这个命令重新配置。

**向导启动前的准备：**

在运行 `hermes setup` 之前，建议先准备好以下信息，这样配置过程会更加顺畅：

1. **已决定使用的 LLM 提供商** —— 如果不确定，先选 DeepSeek 或 OpenRouter
2. **对应提供商的 API Key** —— 提前注册并复制好 Key
3. **Agent 的名称** —— 给你的 Agent 起个名字（如「小墨」「码匠」等）
4. **期望的对话语言** —— 中文、英文或其他语言

准备这些只需要 5 分钟，但能让你在向导中一气呵成地完成配置，而不是中途暂停去注册账号。

#### 步骤 4：获取和输入 API Key

根据你选择的提供商，向导会要求你输入 API Key。下面是最常用平台的获取教程：

**DeepSeek API Key 获取：**

1. 访问 https://platform.deepseek.com/
2. 点击右上角「登录」，用手机号或邮箱注册
3. 登录后进入「API Keys」页面
4. 点击「创建 API Key」
5. 复制生成的 Key（以 `sk-` 开头）

```
🔑 输入 DeepSeek API Key
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

请粘贴你的 API Key（不会显示在屏幕上）：

> ******************************

✓ 格式验证通过
✓ 连接测试成功
✓ 余额查询: ¥15.00（赠金）
```

> **注意**
>
> 粘贴 API Key 时请务必检查是否有**尾随空格**！这是官方排障指南中排名前三的问题。你可以在粘贴后按 `Ctrl+A`（全选）查看是否有额外空格。如果不确定，可以先粘贴到文本编辑器中检查，确认无误后再复制到终端。

**OpenRouter API Key 获取：**

1. 访问 https://openrouter.ai/
2. 用 GitHub 或 Google 账号登录
3. 进入 Settings > Keys
4. 点击「Create Key」
5. 复制 Key（以 `sk-or-v1-` 开头）

> **Tips**
>
> OpenRouter 的优势在于「一 Key 多用」。你只用这一个 Key，就可以调用 Google Gemini、Mistral、Anthropic Claude、Meta Llama 等数十个模型的 API。而且它提供部分模型的免费额度（虽然速率有限制），非常适合入门体验。

**Nous Portal 订阅：**

1. 访问 https://portal.nousresearch.com/
2. 注册账号并选择订阅计划
3. 订阅后系统会生成一个统一 API Key
4. 用这个 Key 即可访问 Portal 上所有接入的模型

**Ollama 本地模型配置：**

如果你选择本地模型，不需要 API Key，但需要先安装 Ollama 并下载模型：

```bash
# 安装 Ollama
brew install ollama  # macOS
# 或访问 https://ollama.com/download 下载安装包

# 下载 Qwen 2.5 Coder 32B（推荐）
ollama pull qwen2.5-coder:32b

# 验证模型可用
ollama list
```

然后在 Hermes 配置中选择：

```
本地模型提供者: ollama
模型名称: qwen2.5-coder:32b
```

> **注意**
>
> Qwen 2.5 Coder 32B 需要约 20GB 显存（或同等内存）。如果你的电脑没有独立显卡，运行速度会很慢。对于没有 GPU 的 Mac 用户，可以使用 7B 或 14B 版本，但效果会打折扣。

### 1.3 配置完成后的验证测试

配置完成后，建议运行以下测试命令来确认一切正常：

```bash
# 1. 版本检查
hermes --version

# 2. 系统诊断（所有检查项应为绿色通过）
hermes doctor

# 3. 开始第一次对话
hermes
```

第一次对话时，试着问一个简单的问题：

```
> 你好，请简单介绍一下你自己
```

如果 Hermes 能正常回复，说明配置完全成功！

**首次对话测试清单：**

- [ ] Hermes 能正常启动并显示欢迎信息
- [ ] 输入消息后能收到回复（不是错误或超时）
- [ ] 回复内容合理、通顺
- [ ] 尝试让 Hermes 读取一个本地文件（如 `cat ~/.hermes/config.yaml`）
- [ ] 尝试让 Hermes 执行一个简单的 shell 命令（如 `ls -la`）
- [ ] 尝试使用 `/model` 切换模型
- [ ] 尝试使用 `/memory` 添加一条记忆
- [ ] 输入 `exit` 能正常退出

如果以上所有测试都通过，恭喜你，你的 Hermes 已经完全就绪，可以开始日常使用了！

> **常见问题**
>
> **Q：第一次对话时 Hermes 回复很慢，甚至超时，怎么办？**
>
> A：这通常是网络问题或模型 API 响应慢导致的。解决方法：
> 1. 检查网络连接：`ping google.com`
> 2. 测试 API 连通性：`hermes doctor --models`
> 3. 如果是国内用户访问海外 API，检查代理是否生效
> 4. 尝试切换到响应更快的模型（如 DeepSeek）
> 5. 如果使用的是本地模型，检查 Ollama 是否在运行：`ollama list`

## 2. 对话模式详解

### 2.1 基本对话入门

运行 `hermes` 命令后，你会进入一个交互式对话界面：

```bash
$ hermes

╔══════════════════════════════════════════╗
║       Hermes Agent v1.2.3                ║
║       Model: deepseek-chat               ║
╚══════════════════════════════════════════╝

输入消息开始对话，输入 /help 查看可用命令，输入 exit 退出。

You: 
```

此时光标停留在 `You:` 后面，你可以直接输入问题，按回车发送。

**示例对话：**

```
You: 你好，请问你能帮我做什么？

Hermes: 你好！我是你的 AI 助手，可以帮助你完成以下任务：

• 💬 **对话问答** —— 回答各类知识性问题
• 💻 **编程辅助** —— 写代码、Debug、代码审查
• 📝 **内容创作** —— 写文章、邮件、报告
• 🔍 **信息检索** —— 搜索网页、分析文档
• 📁 **文件管理** —— 读写文件、整理资料
• ⚡ **自动化任务** —— 执行脚本、定时任务

直接告诉我你需要什么帮助吧！

You: 
```

> **Tips**
>
> 在对话中，你可以随时输入 `/help` 查看所有可用的斜杠命令，或者输入 `exit` 或按 `Ctrl+D` 退出对话。

### 2.2 多轮对话的上下文管理

Hermes 会自动维护对话的上下文。这意味着你可以像和朋友聊天一样，进行多轮对话，而不需要每次都重复背景信息。

**上下文示例：**

```
You: 我正在做一个 Python 项目，需要处理 CSV 文件

Hermes: 好的，处理 CSV 文件在 Python 中有很多方式。你可以使用...

You: 数据量比较大，有 100 万行，哪种方式最快？

Hermes: 对于 100 万行的大数据集，推荐使用 pandas 的...
```

注意第二轮对话中，你不需要再提「Python」和「CSV」，Hermes 已经记住了上下文。

**上下文管理的进阶技巧：**

**技巧 1：主动重置上下文**

当你想切换到全新话题时，可以使用 `/clear` 命令清除当前上下文。这就像是把黑板擦干净，准备写新的内容。

```
You: /clear

✓ 上下文已清除。

You: 现在我们来讨论一个完全不同的主题...
```

**技巧 2：利用上下文延续复杂任务**

对于需要多步骤完成的任务，上下文让对话变得异常流畅：

```
You: 帮我写一个 Flask 博客的后端 API

Hermes: [给出完整的 Flask API 代码]

You: 好的，现在帮我写对应的前端页面，用 Vue.js

Hermes: [基于刚才的 API 设计，给出 Vue.js 前端代码]

You: 最后帮我写个 Docker 配置，把前后端一起打包

Hermes: [给出 Dockerfile 和 docker-compose.yml]
```

**技巧 3：在上下文中修正错误**

如果 Hermes 的某一步回答有误，你不需要从头开始，直接指出错误即可：

```
You: 帮我写一个计算阶乘的函数

Hermes: [给出了递归实现]

You: 这个递归实现对于大数会栈溢出，改成迭代版本

Hermes: [给出迭代实现]
```

> **注意**
>
> 上下文不是无限存储的！默认情况下，Hermes 会保留最近 50 条对话记录。当对话很长时，早期的消息可能会被自动丢弃。如果需要保留重要信息，可以：
> 1. 使用 `/memory` 命令让 Hermes 记住关键信息
> 2. 使用 `/compact` 压缩上下文（详见第 4 章）
> 3. 重要的配置和背景信息写入 SOUL.md 或 MEMORY.md
> 4. 对于特别长的任务，可以分段进行，每完成一段用 `/memory` 保存关键结论

### 2.3 对话中使用工具

Hermes 最强大的功能之一就是**工具调用**。在对话过程中，当你提出的需求需要外部工具时，Hermes 会自动调用相应的工具来完成任务。

**示例 1：文件操作**

```
You: 帮我读取 ~/.hermes/config.yaml 的内容

Hermes: 我来帮你读取这个配置文件。

[工具调用: file_read]

### 2.4 对话中的高效提问技巧

和 Hermes 对话，就像和一位聪明的助手交流。提问的方式直接决定了回答的质量。以下是几个让 Hermes 回答更好的提问技巧：

**技巧 1：明确任务类型**

❌ 不好的提问：「帮我看看这个」
✅ 好的提问：「帮我审查这段 Python 代码的安全性，特别关注 SQL 注入和 XSS 漏洞」

**技巧 2：提供足够的上下文**

❌ 不好的提问：「为什么报错？」
✅ 好的提问：「我在运行 `python main.py` 时遇到以下错误：
```
Traceback (most recent call last):
  File "main.py", line 42, in <module>
    result = fetch_data(user_input)
TypeError: expected str, got int
```
`fetch_data` 函数的代码如下：...」

**技巧 3：指定输出格式**

❌ 不好的提问：「总结一下」
✅ 好的提问：「请用 bullet points 总结以下文章的核心观点，每条不超过 20 个字，并标注每条的重要性（高/中/低）」

**技巧 4：分步骤提问复杂问题**

对于复杂任务，不要一次性抛出所有需求。先让 Hermes 理解整体目标，再逐步细化：

```
You: 我想做一个个人博客网站，用什么技术栈比较好？

Hermes: [给出技术栈建议]

You: 好，我决定用 Next.js + Tailwind。帮我设计一下数据库表结构，
需要支持文章、标签、评论三个功能。

Hermes: [给出数据库设计]

You: 很好。现在帮我写用户认证模块的代码，要求支持邮箱+密码和 GitHub OAuth。
```

**技巧 5：利用角色设定**

在 SOUL.md 中定义好角色后，你可以直接要求它按角色视角回答：

```
You: 以你的投研分析师身份，评估一下这个加密货币项目的风险
```

> **Tips**
>
> 好的提问 = 明确的任务 + 足够的上下文 + 期望的输出格式。把这三点做到位，Hermes 的回答质量会提升一个档次。记住，Agent 再聪明，也无法读取你的心思——你提供的信息越完整，它给你的答案就越精准。

### 2.5 对话历史的存储与检索

Hermes 会自动保存所有对话历史到数据库中。你可以随时查看和检索过去的对话。

```bash
# 列出最近的会话
hermes chat list

# 查看某个会话的详细内容
hermes chat show <session_id>

# 搜索历史对话中的内容
hermes chat search "Python CSV"

# 导出某个会话为 Markdown 文件
hermes chat export <session_id> --format md --output ~/chat-history.md
```

> **注意**
>
> 对话历史默认存储在 `~/.hermes/database/hermes.db` 中。这个 SQLite 数据库文件包含了你的所有聊天记录。如果你担心隐私问题，可以定期清理历史，或者将敏感对话设置为不保存（在对话开始时输入 `/private` 进入隐私模式）。

### 2.5 退出与恢复对话

**退出当前对话：**

```
You: exit

对话已保存。会话 ID: 2026-04-21-abc123
再见！
```

**恢复上次的对话：**

```bash
# 方式 1：直接运行 hermes，会自动恢复上次会话
hermes

# 方式 2：指定会话 ID 恢复
hermes chat resume 2026-04-21-abc123

# 方式 3：从列表中选择
hermes chat list
hermes chat resume <session_id>
```

> **Tips**
>
> Hermes 支持同时维护多个对话会话。比如你可以有一个「工作会话」处理项目代码，一个「学习会话」讨论技术书籍，一个「生活会话」安排日程。通过不同的会话 ID，你可以随时在不同的话题之间切换。

## 3. 常用命令速查大全

Hermes 提供了丰富的 CLI 命令。掌握这些命令，你就能像老司机一样熟练操控你的 Agent。

### 3.1 hermes（开启对话）

`hermes` 是最常用的命令，用于启动交互式对话。它就像打开微信聊天窗口一样简单。

```bash
# 基础用法 —— 启动交互式对话
hermes

# 带初始提示启动（适合一次性任务）
hermes -c "帮我写一个 Python 爬虫"

# 使用特定模型启动（覆盖默认配置）
hermes --model deepseek-reasoner

# 使用特定 provider 启动
hermes --provider openrouter --model anthropic/claude-3.5-sonnet

# 不加载历史会话，全新开始
hermes --fresh

# 进入隐私模式（不保存对话历史）
hermes --private

# 指定工作目录（Agent 以此目录为上下文）
hermes --workspace ~/my-project

# 组合使用多个参数
hermes --model claude-3.5-sonnet --workspace ~/project --private
```

**使用场景示例：**

| 场景 | 推荐命令 | 说明 |
|-----|---------|------|
| 日常闲聊/问答 | `hermes` | 默认启动，自动恢复上次会话 |
| 临时任务 | `hermes -c "任务描述"` | 执行完自动退出 |
| 代码审查 | `hermes --workspace .` | 以当前项目为上下文 |
| 私密对话 | `hermes --private` | 不保存到数据库 |
| 切换模型测试 | `hermes --model gpt-5` | 临时用其他模型 |
| 全新话题 | `hermes --fresh` | 不携带历史上下文 |

# 技巧 1：批量导出所有会话为 Markdown
hermes chat list --format json | jq -r '.[].id' | while read id; do
  hermes chat export "$id" --format md --output "~/backups/$id.md"
done

# 技巧 2：测试所有配置的模型连接
hermes model list --format json | jq -r '.[].name' | while read model; do
  echo "Testing $model..."
  hermes model test "$model"
done

# 技巧 3：一键诊断并修复常见问题
hermes doctor --fix && hermes config validate && echo "All good!"

# 技巧 4：快速切换模型进行 A/B 测试
hermes -c "用 Python 写快速排序" --model deepseek-chat > /tmp/result_a.txt
hermes -c "用 Python 写快速排序" --model claude-3.5-sonnet > /tmp/result_b.txt
diff /tmp/result_a.txt /tmp/result_b.txt

# 技巧 5：批量更新所有技能
hermes skills list --outdated | xargs -I {} hermes skills update {}
```

> ```bash
> alias hdoc='hermes doctor'
> alias hchat='hermes chat list'
> alias hmodel='hermes model list'

> ```

## 4. 斜杠命令系统

在对话模式中，输入以 `/` 开头的命令可以触发特殊功能。斜杠命令就像是 Hermes 的「快捷键」，让你不用退出对话就能执行各种操作。

### 4.1 /help 帮助

```
You: /help

可用命令：
  /help        显示此帮助信息
  /model       切换模型
  /memory      查看或编辑记忆
  /skills      管理技能
  /compact     压缩上下文
  /clear       清除当前上下文
  /export      导出对话
  /private     进入隐私模式
  /exit        退出对话

输入 /help <命令名> 查看详细说明。
```

### 4.2 /model 切换模型

在对话过程中随时切换模型：

```
You: /model

可用模型：
  1. deepseek-chat (当前)
  2. deepseek-reasoner
  3. anthropic/claude-3.5-sonnet
  4. openai/gpt-5

> 3

✓ 已切换到 anthropic/claude-3.5-sonnet
```

> **Tips**
>
> 在对话中切换模型是 Hermes 的杀手级功能之一。比如你先让便宜的 DeepSeek 处理大部分对话，遇到复杂的代码审查时切到 Claude 3.5 Sonnet，完成后再切回 DeepSeek。这样既能保证质量，又能控制成本。

### 4.3 /memory 查看记忆

```
You: /memory

当前全局记忆（来自 MEMORY.md）：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

• 用户偏好简洁回答
• 用户是 Python 开发者
• 用户项目目录: ~/projects/my-startup

You: /memory add 用户最近在学 Rust

✓ 已添加到全局记忆。
```

### 4.4 /skills 管理技能

```
You: /skills

已加载的技能：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

✓ web_search      网页搜索
✓ file_operations 文件读写
✓ code_execution  代码执行
✗ stock_analysis  股票分析 (已禁用)

You: /skills enable stock_analysis

✓ 已启用 stock_analysis
```

### 4.5 /compact 压缩上下文

当对话变得很长时，上下文可能会占用大量 token。`/compact` 命令会让 Hermes 自动总结之前的对话内容，用简洁的摘要替代原始消息，从而释放上下文空间。

```
You: /compact

正在压缩上下文...
✓ 上下文已从 42 条消息压缩为 3 条摘要。
节省 token: ~8,500
```

> **注意**
>
> 压缩上下文会丢失部分细节。如果对话中有重要的具体信息（如特定的代码片段、精确的数字），建议在压缩前用 `/memory` 命令将其保存到全局记忆中。

### 4.6 /clear 清除上下文

```
You: /clear

确定要清除当前对话上下文吗？（y/n）> y

✓ 上下文已清除。当前会话变为全新状态。
```

### 4.7 /export 导出对话

```
You: /export

导出格式：
  1. Markdown
  2. JSON
  3. 纯文本

> 1

导出文件名（默认: session-2026-04-21.md）> 

✓ 对话已导出到 ~/session-2026-04-21.md
```

### 4.8 其他斜杠命令

| 命令 | 功能 | 示例 |
|-----|------|------|
| `/private` | 进入隐私模式（不保存本次对话） | `/private` |
| `/public` | 退出隐私模式 | `/public` |
| `/undo` | 撤销上一条消息 | `/undo` |
| `/retry` | 重新生成上一条回复 | `/retry` |
| `/system` | 发送系统消息 | `/system 请用中文回答` |
| `/temp` | 临时切换模型（仅下一条消息） | `/temp claude-3.5-sonnet` |

**斜杠命令使用技巧：**

**技巧 1：快速切换模型进行 A/B 测试**

```
You: 帮我写一个快速排序的 Python 实现

Hermes: [DeepSeek 给出实现]

You: /temp claude-3.5-sonnet
用类型注解和文档字符串重写上面的代码

Hermes: [Claude 给出更完善的实现]
```

**技巧 2：用 /system 纠正 Agent 的行为**

```
You: /system 从现在开始，所有代码示例都要包含单元测试

You: 写一个判断闰年的函数

Hermes: [代码 + 单元测试]
```

**技巧 3：用 /compact + /memory 组合管理长对话**

```
You: /compact

Hermes: ✓ 上下文已压缩

You: /memory add 我们刚才讨论的核心结论是：使用异步 IO 可以提升 3 倍性能

Hermes: ✓ 已添加到全局记忆
```

> **Tips**
>
> 斜杠命令是 Hermes 交互效率的倍增器。刚开始你可能记不住所有命令，没关系，只要记住 `/help`、`/model` 和 `/memory` 这三个最常用命令，就能覆盖 80% 的场景。其他的命令可以慢慢在实践中熟悉。

### 5.4 模型切换的实际效果对比

同一个问题，不同模型的回答风格和深度可能差异很大。下面是一个实际对比：

**问题：解释什么是递归函数，并用 Python 举例。**

**DeepSeek (deepseek-chat)：**

> 递归函数就是在函数内部调用自己的函数。就像俄罗斯套娃，每个娃娃里面还有一个更小的自己。
>
> 经典例子是阶乘：
> ```python
> def factorial(n):
>     if n <= 1:
>         return 1
>     return n * factorial(n - 1)
> ```
>
> 但要注意，递归必须有终止条件，否则会无限循环导致栈溢出。

**Claude 3.5 Sonnet：**

> 递归是一种编程范式，函数通过调用自身来解决问题。其核心思想是将大问题分解为结构相同的子问题。
>
> Python 示例 —— 阶乘：
> ```python
> def factorial(n: int) -> int:
>     """计算 n 的阶乘。"""
>     if n < 0:
>         raise ValueError("n must be non-negative")
>     if n <= 1:
>         return 1
>     return n * factorial(n - 1)
> ```
>
> 更实用的例子 —— 遍历嵌套目录：
> ```python
> import os
>
> def list_files(path: str, indent: int = 0):
>     """递归列出目录下所有文件。"""
>     for item in os.listdir(path):
>         full = os.path.join(path, item)
>         print("  " * indent + item)
>         if os.path.isdir(full):
>             list_files(full, indent + 1)
> ```
>
> 递归 vs 迭代的权衡：递归代码更简洁直观，但函数调用有栈深度限制（Python 默认约 1000 层）。生产环境中，对于深度未知的问题，建议改用迭代或尾递归优化。

可以看到，DeepSeek 的回答简洁直白，适合快速理解；Claude 的回答更全面深入，包含了类型注解、错误处理、实际应用场景和性能权衡。

## 6. 分层模型配置（省钱秘笈）

如果你只用一种模型完成所有任务，很快就会面临一个困境：用 Claude 3.5 Sonnet 处理所有事情太贵，用 DeepSeek 处理复杂任务又不够用。分层模型配置就是解决这个问题的「省钱秘笈」。

### 6.1 config.yaml 模型配置详解

分层模型的核心思想是：**不同难度的任务用不同档次的模型**。就像你不会用兰博基尼去买菜，也不会用自行车跑高速一样。

在 `config.yaml` 中，分层模型通过以下配置实现：

```yaml
# ~/.hermes/config.yaml

models:
  # === 第一层：推理模型 ===
  # 负责处理复杂问题、需要深度思考的任务
  reasoning:
    provider: "anthropic"
    model: "claude-3.5-sonnet"
    # 触发条件：任务复杂度评分 > 0.7
    # 适用场景：架构设计、复杂 Debug、数学证明、深度分析

  # === 第二层：执行模型 ===
  # 负责处理常规任务、日常对话
  execution:
    provider: "deepseek"
    model: "deepseek-chat"
    # 适用场景：日常问答、简单代码、文件操作、信息检索

  # === 第三层：Fallback 链 ===
  # 当首选模型不可用时，按顺序尝试备选模型
  fallback:
    - provider: "openrouter"
      model: "google/gemini-flash-1.5"
    - provider: "deepseek"
      model: "deepseek-chat"
    - provider: "openai"
      model: "gpt-4o-mini"

  # === 本地模型（轻量任务）===
  local:
    provider: "ollama"
    model: "qwen2.5-coder:32b"
    # 适用场景：离线环境、隐私敏感、简单代码补全
```

> **注意**
>
> 分层模型的自动切换依赖于 Hermes 内置的「任务复杂度评估器」。它会根据你的输入自动判断任务难度：
> - 简单问题（如"Python 怎么写循环？"）→ 执行模型
> - 复杂问题（如"帮我设计一个高并发的微服务架构"）→ 推理模型
> - 代码审查、长文档分析 → 推理模型
> - 日常闲聊、简单文件操作 → 执行模型

### 6.2 核心逻辑用强模型、基础执行用便宜模型

### 6.3 Fallback 链机制详解

Fallback 链是分层模型配置中的「保险丝」。当首选模型因为网络问题、余额不足或服务商故障而无法使用时，Hermes 会自动按顺序尝试 Fallback 列表中的模型。

**Fallback 触发条件：**

1. API 返回 429（速率限制）
2. API 返回 401/403（认证失败 / 余额不足）
3. API 返回 5xx（服务商内部错误）
4. 请求超时（默认 60 秒）
5. 网络连接失败

**Fallback 配置示例：**

```yaml
models:
  execution:
    provider: "deepseek"
    model: "deepseek-chat"

  fallback:
    # 第一备选：OpenRouter 上的 Gemini（免费额度）
    - provider: "openrouter"
      model: "google/gemini-flash-1.5"
      
    # 第二备选：OpenAI 的便宜模型
    - provider: "openai"
      model: "gpt-4o-mini"
      
    # 第三备选：本地模型（最后一道防线）
    - provider: "ollama"
      model: "qwen2.5-coder:7b"
```

> **注意**
>
> Fallback 链中的模型应该尽量选择**不同类型的服务商**。比如你的主模型是 DeepSeek（国内），第一备选选 OpenRouter（海外），第二备选选本地 Ollama。这样可以避免单一服务商故障导致所有模型都不可用的情况。

## 7. SOUL.md 人设配置

### 7.1 什么是 SOUL.md

SOUL.md 是 Hermes 最具特色的功能之一。它是 Agent 的「灵魂文件」，定义了你的 Agent 的性格、身份、说话方式、专业领域和行为准则。

你可以把 SOUL.md 理解为：

- **对 Hermes 来说** = 它的角色剧本和表演指南
- **对你来说** = 一次编写，永久生效的「系统提示词」
- **对对话来说** = 决定 Agent 语气、风格和能力的底层配置

> **注意**
>
> SOUL.md 与单次对话中的「系统消息」不同。系统消息只在当前会话有效，而 SOUL.md 会在**每次启动 Hermes 时自动加载**，永久影响 Agent 的行为。这意味着你不需要每次对话都告诉它「请用中文回答」或「请简洁一些」——把这些写进 SOUL.md 即可。

### 7.2 编写有效的 SOUL.md（模板 + 示例）

### 7.5 常见人设配置错误与纠正

#### 错误 1：写得太多太杂

**错误示例：**

```markdown
## 专业领域

- Python
- JavaScript
- Java
- C++
- Go
- Rust
- 数据分析
- 机器学习
- 深度学习
- 区块链
- 金融
- 医学
- 法律
...
```

**问题：** 列了太多领域，Agent 反而不知道你真正需要它擅长什么。

**纠正：** 聚焦 3-5 个核心领域，其他领域可以写「了解但不精通」。

#### 错误 2：指令互相矛盾

**错误示例：**

```markdown
## 说话风格

- 回答要简洁，不超过 3 句话
- 每个回答都要包含完整的代码示例和详细解释
```

**问题：** 「不超过 3 句话」和「完整代码示例 + 详细解释」不可能同时满足。

**纠正：** 按优先级排序，或者分场景说明。

#### 错误 3：过度限制

**错误示例：**

```markdown
## 禁忌

- 不使用任何标点符号
- 不使用数字
- 不使用英文
- 不提及任何技术名词
```

**问题：** 过度限制会让 Agent 的回答质量严重下降。

**纠正：** 只限制真正影响体验的行为，保留 Agent 的灵活性。

#### 错误 4：不写「身份」部分

**问题：** 没有身份定义，Agent 不知道自己是谁，回答缺乏一致性。

**纠正：** 始终包含「身份」部分，哪怕只有一句话。

### 7.6 SOUL.md 的调试与验证技巧

写了 SOUL.md 之后，怎么知道它有没有生效？如果 Agent 的表现不符合预期，又该如何调试？这里有几个实用的技巧。

#### 验证 SOUL.md 是否生效

最简单的方法是问 Agent 一个能体现人设的问题：

```
You: 你是谁？请描述一下你的角色和专长。
```

如果 SOUL.md 生效了，Agent 会按照你定义的身份回答，而不是默认的「我是一个 AI 助手」。

#### 常见问题和修复

**问题 1：Agent 仍然用默认语气回答**

原因：SOUL.md 中的指令不够具体，或者被其他配置覆盖。

修复：在「说话风格」部分使用更明确的指令，例如：

```markdown
## 说话风格

- **必须**使用中文回答，无论用户用什么语言提问
- **必须**先给出结论，再展开解释
- **禁止**使用「作为一个人工智能助手」作为回答开头
```

用「必须」「禁止」等强约束词比「尽量」「建议」更有效。

**问题 2：SOUL.md 太长了，Agent 表现不稳定**

原因：SOUL.md 内容过多，导致核心指令被稀释。

修复：精简内容，只保留最关键的 5-10 条指令。SOUL.md 不是越详细越好——Agent 的上下文窗口有限，太长的 SOUL.md 反而会让重要信息被忽略。

#### SOUL.md 迭代优化法

写 SOUL.md 不是一锤子买卖，建议用以下方法持续优化：

1. **记录不满意的对答** —— 当 Agent 的回答不符合预期时，把对话保存下来
2. **分析原因** —— 是 SOUL.md 缺少相关指令，还是指令不够明确？
3. **修改 SOUL.md** —— 添加或调整对应条款
4. **重启 Hermes 验证** —— 用同样的提问测试，看是否改善
5. **重复 1-4** —— 一般经过 3-5 轮迭代，SOUL.md 就会趋于稳定

> **Tips**
>
> 好的 SOUL.md 就像好的公司章程——不需要面面俱到，但核心原则必须清晰明确。初学者最容易犯的错误是写得太长、太琐碎。建议先写一个 20 行的精简版，用一周时间观察效果，再根据实际需要逐步扩充。记住：少即是多，精准的约束胜过冗长的描述。

## 9. 新手常见误区与最佳实践

### 9.1 误区 1：把它当搜索引擎用

**错误做法：**

```
You: 2026 年诺贝尔奖得主是谁？
```

**问题：** Hermes 不是搜索引擎，它的知识有截止日期，对于实时信息可能会给出错误答案。

**正确做法：**

```
You: 搜索一下 2026 年诺贝尔奖的最新消息
```

**区别：** 明确使用「搜索」指令，Hermes 会调用 `web_search` 工具获取最新信息，而不是依赖训练数据中的知识。

> **注意**
>
> 大语言模型的「幻觉」问题（一本正经地胡说八道）在回答事实性问题时最明显。对于时效性强的事实查询，一定要让 Agent 联网搜索，不要直接问。

### 9.2 误区 2：不给足够上下文

**错误做法：**

```

```

**问题：** 没有提供代码，Agent 无从改起。

**正确做法：**

目标是让处理 100 万条数据的时间从 30 秒降到 5 秒以内

> **Tips**

**错误做法：**

### 9.4 更多常见误区

#### 误区 4：忽视 SOUL.md 的作用

**错误做法：** 从不修改默认的 SOUL.md，每次对话都手动说明自己的偏好。

**问题：** 浪费大量 token 在重复说明上，而且 Agent 无法形成稳定的「人格记忆」。

**正确做法：** 花 20 分钟写一份好的 SOUL.md，之后每次启动 Hermes 都会自动加载。

#### 误区 5：所有任务都用最强模型

**错误做法：** 不管问什么问题都用 Claude 3.5 Sonnet 或 GPT-5。

**问题：** 简单的「今天天气怎么样」也用 $0.03 的模型回答，一个月下来费用惊人。

**正确做法：** 配置分层模型，让 Hermes 自动选择最适合当前任务的模型。

#### 误区 6：不及时清理对话历史

**错误做法：** 从不删除旧会话，数据库文件膨胀到几个 GB。

**问题：** 占用磁盘空间，搜索历史变慢，隐私风险增加。

**正确做法：** 每月运行一次 `hermes chat cleanup --older-than 30d`，同时备份重要会话。

#### 误区 7：在公共环境中暴露 API Key

**错误做法：** 在共享服务器、公共仓库或截图中暴露 API Key。

**问题：** API Key 一旦泄露，可能被他人盗用，导致巨额账单。

**正确做法：** 始终通过环境变量管理 API Key，`.gitignore` 中排除 `.env` 文件，分享截图前打码敏感信息。

#### 误区 8：不信任工具执行的安全提示

**错误做法：** 看到 Hermes 要执行 `rm -rf` 或修改系统文件时，不看详情直接确认。

**问题：** 虽然 Hermes 有安全机制，但用户确认是最后一道防线。误操作可能导致数据丢失。

**正确做法：** 每次工具执行前都仔细查看即将执行的命令，特别是涉及删除、修改系统配置的操作。

14. **为不同项目创建独立工作区**

```bash
# 项目 A
cd ~/project-a && hermes --workspace .

# 项目 B
cd ~/project-b && hermes --workspace .
```

每个工作区的上下文隔离，避免项目间的信息混淆。

15. **定期 Review 和优化 SOUL.md**

每个月花 10 分钟回顾 SOUL.md，根据实际使用体验调整：

```bash
# 查看 SOUL.md 当前内容
cat ~/.hermes/SOUL.md

# 让 Hermes 帮你评估 SOUL.md 的效果
hermes -c "评估我的 SOUL.md 配置，给出优化建议"
```

### 工作流 2：$5 VPS 上的 Telegram 机器人

这是一个低成本、高价值的应用场景：在最便宜的 VPS 上部署一个 24/7 运行的 Hermes Telegram 机器人。

**硬件要求：**

- 任意 $3-5/月的 VPS（Hetzner CPX11、DigitalOcean Basic Droplet 等）
- 1 vCPU + 1GB RAM 完全够用

**部署步骤：**

```bash
# 1. 在 VPS 上安装 Hermes（参考第五册 VPS 部署章节）
curl -fsSL https://hermes-agent.nousresearch.com/install.sh | sh

# 2. 配置 Hermes
hermes setup

# 3. 创建 Telegram Bot
# 在 Telegram 中搜索 @BotFather
# 发送 /newbot，按提示创建机器人
# 记下 Bot Token（格式: 123456789:ABCdefGHIjklMNOpqrsTUVwxyz）

# 4. 配置 Telegram 网关
hermes gateway config telegram --token "你的BotToken"

# 5. 设置安全允许列表（重要！）
# 只允许特定用户使用你的机器人，防止被滥用
```

**配置允许列表：**

```yaml
# 在 ~/.hermes/config.yaml 中添加
gateway:
  telegram:
    token: "${TELEGRAM_BOT_TOKEN}"
    allowed_users:
      - 123456789    # 你的数字用户 ID
      - 987654321    # 朋友的数字用户 ID
```

> **注意**
>
> Telegram 的安全设置中，**允许列表需要填写数字用户 ID，不是 @handle（用户名）！**
> 你可以通过以下方式获取自己的数字 ID：
> 1. 在 Telegram 中搜索 @userinfobot
> 2. 点击 Start，机器人会回复你的用户信息，其中 "Id" 就是数字用户 ID
> 3. 或者在 Telegram 中发送任意消息给 @getidsbot
> 
> 填写错误的格式（如 @username）会导致消息过滤不生效，任何人都能使用你的机器人，消耗你的 API 额度！

**启动机器人：**

```bash
# 启动网关（前台运行测试）
hermes gateway start

# 测试正常后，使用 systemd 后台运行
sudo systemctl enable hermes
sudo systemctl start hermes
```

**使用示例：**

在 Telegram 中向你的机器人发送消息：

```

    def fetch_data(url):
        return requests.get(url).json()

机器人: 好的，改成异步版本：

```python
import aiohttp

async def fetch_data(url):
    async with aiohttp.ClientSession() as session:
        async with session.get(url) as response:
            return await response.json()
```

记得在调用处使用 `await fetch_data(url)`。
```

### 工作流 3：每日 Cron 简报（自动收集信息并发邮件）

这是一个自动化信息收集场景：每天早上自动抓取 Hacker News 热门、GitHub 通知、天气信息，然后生成一份简报发送到你的邮箱。

**配置步骤：**

```bash
# 1. 确保 Hermes 已安装并配置好模型

# 2. 编辑定时任务配置
nano ~/.hermes/cron/jobs.yaml
```

**任务配置：**

```yaml
# ~/.hermes/cron/jobs.yaml
jobs:
  - name: "morning-briefing"
    description: "每日早间简报"
    schedule: "0 8 * * *"  # 每天早上 8 点
    command: |
      请帮我生成一份今日早报，包含以下内容：
      
      1. 搜索 Hacker News 今日热门文章 Top 5，总结每条的核心内容
      2. 检查我的 GitHub 通知，总结需要我关注的事项
      3. 查询今天上海的天气
      4. 查看今天是否有重要的技术会议或产品发布
      
      将所有内容整理成一份结构清晰的 Markdown 简报，
      标题为「早报 - YYYY年MM月DD日」，保存到 ~/briefings/ 目录
    enabled: true

  - name: "send-daily-email"
    description: "发送早报邮件"
    schedule: "30 8 * * *"  # 每天早上 8:30（简报生成后）
    command: |
      读取 ~/briefings/ 目录下最新的早报文件，
      将其内容作为邮件正文，发送到 myemail@example.com
      邮件主题：「Hermes 早报 - 今日日期」
    enabled: true
```

**启用定时任务：**

```bash
# 验证配置格式
hermes cron validate

# 查看任务列表
hermes cron list

# 手动测试运行
hermes cron run morning-briefing

# 确认任务已启用
hermes cron status
```

> **Tips**
>
> 这个工作流展示了 Hermes 的「自动化」能力。一旦配置完成，它每天会自动执行，不需要你手动干预。你可以根据自己的需求定制简报内容，比如添加股票监控、RSS 订阅源、竞品动态等。Cron 任务的语言支持自然语言描述，不需要写复杂的脚本——这是 Hermes 相比传统自动化工具的独特优势。
