# 第七册：三层记忆系统详解

## 1. 为什么记忆系统是 Hermes 的核心竞争力

### 1.1 Agent 没有记忆会怎样——上下文爆炸的痛苦

这就是没有记忆系统的 AI Agent 的真实写照。

让我们看一个具体的例子。假设你要让 Agent 帮你做一个「季度投研报告」，这个任务需要：

1. 先搜索某行业的最新新闻（10条结果）
2. 用 Python 分析这些新闻的情感倾向
3. 结合财务数据生成图表
4. 把图表插入到报告模板中
5. 输出最终的 PDF 文件

如果没有记忆系统，会发生什么呢？

当你在第3步说「用刚才爬取的数据做分析」时，Agent 可能会一脸茫然地问：「什么数据？我刚才没保存任何数据啊。」因为它在前两步的工具调用结果，如果没有被妥善记录到上下文中，就会像写在沙滩上的字一样被潮水冲走。

即使你的大模型有 128K 的上下文窗口（相当于能记住约 10 万字的文本），在长时间、多步骤的复杂任务中，Token 消耗也会像滚雪球一样越滚越大。每一次工具调用的结果、每一条系统提示、每一个中间变量，都在无情地吞噬着有限的上下文空间。当上下文接近上限时，模型要么开始「失忆」（遗忘早期信息），要么响应速度急剧下降，要么干脆报错退出。

这种「上下文爆炸」的痛苦，是每个深度使用 AI Agent 的用户都经历过的。它直接限制了 Agent 能够处理任务的复杂度，也严重影响了用户体验的连贯性。

> **注意**
>
> 很多新手会误以为「上下文窗口越大越好」。确实，从 4K 到 128K 甚至 200K 的窗口扩展，让单次对话能处理的信息量变大了。但窗口大不等于记忆好——它只是「短期记忆」的容量变大了。如果没有智能的记忆管理系统，再大的窗口也会在复杂任务中被填满，而且成本会呈指数级增长。

### 1.2 传统 Agent 的记忆方案及其局限

在 Hermes 出现之前，开源社区对 Agent 记忆问题的解决方案大致可以分为以下几类：

**方案一：纯上下文记忆（即无记忆方案）**

这是最原始的方式，也是很多早期 Agent 框架（如最初的 AutoGPT）采用的方式。所有信息都依赖大模型的上下文窗口来承载。优点是简单直接，缺点是前面已经详细描述过了——上下文爆炸、Token 成本高、无法跨会话保留信息。

**方案二：向量数据库（Vector DB）记忆**

这是目前最主流的方案，包括使用 Pinecone、Weaviate、Milvus、Chroma 等向量数据库来存储和检索记忆。工作原理是将文本转为向量（Embedding），通过向量相似度来检索相关内容。

这种方案的问题在于：

- **检索精度依赖 Embedding 质量**：如果向量模型不能准确理解语义，检索结果就会偏差很大。比如用户说「上次我提到的那个项目」，Agent 可能检索到完全不相关的内容。
- **无法精确匹配关键词**：向量相似度擅长找「意思相近」的内容，但不擅长找「包含某个确切词语」的内容。如果你说「把上次生成的那个叫『报表_v2』的文件发给我」，向量检索很可能找不到精确匹配。
- **冷启动问题**：新数据刚写入向量数据库时，检索效果往往不好，需要积累一定量数据后才能发挥作用。
- **运维成本高**：需要额外部署和维护向量数据库服务，增加了系统复杂度。

**方案三：简单的文件持久化**

一些轻量级方案会把重要信息写入本地文件（如 JSON、TXT），下次启动时读取。这种方式虽然实现了跨会话记忆，但缺乏智能的检索和筛选机制——要么一次性加载所有内容（可能文件很大），要么只能按固定规则读取（不够灵活）。

**方案四：外部记忆服务**

如 Mem0、Zep 等专门的记忆服务。它们通常提供云端 API，负责记忆的存储、检索和更新。这种方案的问题是：

- **数据离开本地**：敏感信息需要上传到第三方服务器
- **网络依赖**：没有网络就无法工作
- **额外成本**：大多数服务需要付费使用
- **延迟较高**：每次记忆操作都需要网络请求

### 1.3 Hermes 记忆系统的设计哲学：让 AI 像人一样记忆

Hermes 的设计团队在研究了大量用户反馈和现有方案的局限后，提出了一个核心问题：**人类是怎么记忆的？**

与很多 Agent 框架追求「无限记忆」不同，Hermes 走了一条截然相反的路——**有界记忆（Bounded Memory）**。官方的设计理念非常明确：无限记忆听起来诱人，但会让 Agent 变成一个「垃圾杂物抽屉」，什么都往里塞，最后连自己有什么都记不住。Hermes 强制整合：当 MEMORY.md 满时，Agent 必须主动决定什么值得保留、什么应该丢弃。这种「纪律」保持了工作上下文的精简，让模型始终专注在真正重要的事情上。

正如 Hermes Atlas 官方文档所说：「Claude Code 的 memory 存储关于你的事实——偏好要点。而 Hermes 存储的是程序——产生你想要的要点的研究-过滤-格式化工作流。事实告知模型，程序直接被复用。」这句话精准地概括了 Hermes 记忆系统的本质：它不仅记住「是什么」，更记住「怎么做」。

人类记忆系统不是单一的结构，而是由多个层次组成的精妙体系：

- **工作记忆**（短期记忆）：你正在思考的事情，容量极小（约 4-7 个信息单元），但访问速度极快。
- **情景记忆**（中期记忆）：最近发生的事情，比如今天早上吃了什么、昨天和谁聊了天。容量中等，需要主动回忆才能提取。
- **语义记忆**（长期记忆）：你学会的知识和技能，比如知道 2+2=4、知道北京是中国的首都。容量几乎无限，但需要线索才能激活相关部分。
- **外部记忆**：你写在笔记本上的备忘、存在电脑里的文档、书架上的书籍。容量不受限，但需要主动查找。

Hermes 的记忆系统正是借鉴了这种分层架构，创造了「四层温度记忆模型」。它的设计哲学可以总结为三条核心原则：

**原则一：分层存储，各尽其用**

不同温度（时效性）的记忆，存放在不同层级的存储介质中。热数据放在内存中快速访问，温数据放在文件中持久保存，冷数据放在数据库中按需检索。就像你不会把全家老小的照片都贴在工作电脑的桌面上，而是会分类存放在不同的位置。

**原则二：按需加载，而非全量加载**

传统方案最大的问题之一是「启动时加载所有记忆」。想象一下，你每天早上上班，秘书都要把过去十年的所有会议记录、邮件往来、项目文档都摆在你面前——这不仅浪费你的时间，还会让你被海量信息淹没，找不到真正需要的东西。

Hermes 采用的是「按需检索」策略：Agent 在新对话开始时，不会加载全部历史记录，而是根据当前话题智能检索最相关的片段。这大大降低了 Token 消耗，也避免了信息过载。

**原则三：纯本地优先，隐私至上**

所有记忆数据默认存储在本地 SQLite 数据库和 Markdown 文件中。你的对话内容、行为习惯、个人偏好，都不会离开你的电脑。这对于处理敏感信息（如企业内网数据、个人隐私数据）至关重要。

### 1.4 三层记忆 vs 传统方案的优势对比

为了让你更直观地理解 Hermes 记忆系统的优势，我们来做一个横向对比：

| 对比维度 | 纯上下文记忆 | 向量数据库方案 | 简单文件持久化 | **Hermes 四层记忆** |
|---------|-----------|------------|-----------|-----------------|
| 跨会话记忆 | 无 | 有 | 有 | **有** |
| 检索精度 | 无检索 | 语义相似度 | 无检索 | **FTS5 全文 + LLM 摘要** |
| 启动速度 | 快 | 中等 | 中等 | **快（按需加载）** |
| Token 成本 | 极高 | 中等 | 低 | **极低（按需检索）** |
| 本地隐私 | 有 | 依赖部署 | 有 | **完全本地** |
| 运维复杂度 | 低 | 高 | 低 | **低（SQLite 内置）** |
| 用户画像 | 无 | 无 | 无 | **自动建模（USER.md）** |
| 自我压缩 | 无 | 无 | 无 | **/compact 智能压缩** |
| 外部扩展 | 无 | 有 | 无 | **Honcho/Mem0/OpenViking** |

从这个对比表中可以清楚地看到，Hermes 的四层记忆模型在几乎所有维度上都做到了优秀或良好，而且是一个开箱即用的「全家桶」方案——你不需要单独部署向量数据库、不需要写复杂的检索逻辑、不需要担心数据隐私，一切都内置在 Hermes 中，只需要简单的配置就能工作。

> **Tips**
>
> 如果你是技术人员，可能会问：「向量检索在某些场景下确实很有用，Hermes 不支持吗？」答案是支持的！通过外部记忆扩展（如 Mem0），你可以无缝集成向量检索能力。但 Hermes 的默认方案选择 FTS5 全文索引作为主力检索引擎，是因为在大多数实际场景中，精确的关键词匹配比语义相似度更可靠、更可预测。

## 2. 总览：四层温度记忆模型

### 2.1 热记忆（即时）—— 当前会话上下文

热记忆是最「热」的记忆，就像你大脑中正在思考的内容一样，访问速度最快，但容量最小，而且会随着会话结束而消失。

在 Hermes 中，热记忆就是当前会话的上下文窗口。它包含了：

- 当前对话中你和 Agent 的所有消息
- 最近一次工具调用的结果
- 系统提示（System Prompt）和 SOUL.md 的内容
- 当前激活的 Skill 的上下文

热记忆的特点可以用三个词概括：**快、小、暂存**。

- **快**：所有数据都在内存中，访问延迟几乎为零
- **小**：受限于大模型的上下文窗口（通常是 4K-200K Tokens）
- **暂存**：会话结束或 /compact 执行后，部分内容会被压缩或丢弃

### 2.2 温记忆（持久）—— MEMORY.md + USER.md

温记忆就像你写在便利贴上的重要提醒，或者是你手机备忘录里的关键信息。它们不像热记忆那样随时在眼前，但比翻查旧笔记本要快得多。

Hermes 的温记忆由两个核心文件组成：

**MEMORY.md**：存储与任务相关的持久化信息。比如：

- 「用户要求所有 Python 代码都使用 Type Hints」
- 「当前项目的 Git 仓库地址是 https://github.com/user/project」
- 「上次分析的那只股票代码是 600519」

**USER.md**：存储用户画像和行为模式。比如：

- 「用户喜欢用中文交流，但技术术语保留英文」
- 「用户在上午 9-11 点效率最高，适合安排复杂任务」
- 「用户对深度学习话题感兴趣，对区块链话题不感兴趣」

这两个文件都是纯文本的 Markdown 格式，存储在 Hermes 的配置目录中。它们的容量被严格限制（MEMORY.md 约 2200 字符，USER.md 约 1375 字符），这是经过精心设计的——如果文件太大，每次加载都会消耗大量 Token；如果太小，又无法记录足够有用的信息。

### 2.3 冷记忆（检索）—— SQLite + FTS5 + LLM 摘要

冷记忆就像你书架上排列整齐的档案柜，或者是电脑里分门别类的文件夹。里面存储了海量的历史信息，但你不会一次性把它们全部搬到桌面上，而是需要时去查找。

Hermes 的冷记忆系统由三个核心组件构成：

**SQLite 数据库**：轻量级的本地关系型数据库，存储所有的历史对话、工具调用结果、文件操作记录等。一个 SQLite 文件就是一个完整的数据库，不需要单独的服务进程。

**FTS5 全文索引**：SQLite 内置的全文搜索引擎。你可以把它理解为一个小型的「本地 Google」——输入关键词，它能瞬间从海量文本中找到包含这些词的记录。

**LLM 摘要**：利用大语言模型的能力，自动为每次会话生成简明扼要的摘要。这些摘要也会被索引到 FTS5 中，让你在搜索时能更快地定位到相关会话。

冷记忆的核心设计思想是「按需检索」。当你开启一个新会话时，Hermes 不会把过去一年的所有对话都加载到上下文中，而是：

1. 分析你当前输入的内容，提取关键词
2. 用 FTS5 在历史记录中检索相关片段
3. 只把最相关的几条记录（通常 3-5 条）提取出来
4. 将这些相关片段注入到当前上下文中

这种方式让 Agent 拥有了「回想」能力，同时又不会因为记忆太多而「大脑过载」。

### 2.4 外部记忆（扩展）—— Honcho、Mem0 等

外部记忆是四层模型的最外层，相当于你的个人图书馆、云端笔记、或者是请了一位专门帮你整理资料的研究助理。它们不是 Hermes 内置的，但可以通过插件或 API 无缝集成。

目前支持的外部记忆方案包括：

- **Honcho**：专门用于推导用户行为模式的记忆模块
- **Mem0**：强大的记忆管理平台，支持向量检索
- **OpenViking**：面向企业级场景的记忆扩展
- 以及更多社区开发的记忆插件

外部记忆的优势在于专业化和扩展性。比如 Honcho 在用户行为建模方面比 Hermes 内置的 USER.md 更深入；Mem0 在跨设备同步方面有独到之处。但它们的缺点是需要额外配置，且部分服务涉及数据外传。

### 2.5 各层的容量、速度、成本对比表

| 记忆层级 | 存储介质 | 典型容量 | 访问速度 | Token 成本 | 持久性 | 检索方式 |
|---------|---------|---------|---------|----------|-------|---------|
| 热记忆 | 内存（上下文窗口） | 4K-200K Tokens | 毫秒级 | 高（全部计费） | 会话级 | 直接访问 |
| 温记忆 | 本地 Markdown 文件 | ~3.5K 字符 | 毫秒级 | 低（固定大小） | 永久 | 全量加载 |
| 冷记忆 | SQLite + FTS5 | 无上限 | 百毫秒级 | 极低（只加载相关片段） | 永久 | 全文检索 |
| 外部记忆 | 网络服务/外部 DB | 依赖服务 | 百毫秒-秒级 | 依赖服务 | 依赖服务 | 依赖服务 |

### 2.6 类比：就像人的短期记忆、长期记忆、笔记本、图书馆

为了让四层记忆模型更加直观，我用一个生活中的类比来总结：

**热记忆**就像你此刻脑中正在构思的内容。你正在想「第一页 PPT 要讲团队介绍」，这个念头在你的意识中非常清晰，但如果你想别的事情，这个念头可能就会被挤掉。这就是工作记忆的容量限制。

**温记忆**就像你贴在显示器边缘的便利贴。上面写着「演讲时间：15分钟」「重点强调 Q3 业绩」「别忘提到新客户案例」。你不需要去翻笔记本，一眼就能看到。但如果便利贴太多，你反而会找不到重点，所以温记忆的容量被刻意限制了。

**冷记忆**就像你的笔记本和电脑文件夹。过去半年里，你做的每一次演讲、每一份报告、每一条客户反馈，都记录在里面。你不会在准备演讲时把过去半年的所有文件都摊开，而是会根据「这次演讲的主题」去搜索相关资料——找到其中 3-5 份最相关的文档，提取关键内容。

**外部记忆**就像你公司里的资料室，或者是国家图书馆。里面有海量的专业书籍、行业报告、历史档案。你需要的时候可以去查阅，获取更深入、更全面的信息。但去图书馆需要时间和交通成本，所以你通常先在笔记本里找，找不到再去图书馆。

> **常见问题**
>
> **Q：为什么温记忆的容量这么小？只有几千字符？**
>
> A：这是经过深思熟虑的设计。温记忆（MEMORY.md + USER.md）每次会话都会完整加载到上下文中。如果它们太大，会挤占宝贵的上下文空间，导致热记忆可用容量减少。限制容量的另一个原因是「逼迫 Agent 做选择」——不是什么都记，而是只记最重要、最精炼的信息。这就像一个优秀的秘书，不会把你说的每句话都记下来，而是只记关键点。

## 3. 第一层：会话记忆（热记忆）

### 3.1 会话上下文的工作原理

会话记忆（热记忆）是 Hermes 记忆系统的最前线，也是你作为用户最直接感受到的部分。理解它的工作原理，能帮助你更高效地与 Agent 协作。

当你启动 Hermes 并开始一段对话时，系统会构建一个「上下文窗口」（Context Window）。这个窗口里装的不是简单的聊天记录，而是一个精心编排的「信息包」，按照固定的顺序排列：

```
[系统提示 / System Prompt]
  ↓
[SOUL.md 内容]
  ↓
[激活的 Skill 说明]
  ↓
[MEMORY.md 内容（温记忆注入）]
  ↓
[检索到的相关历史片段（冷记忆注入）]
  ↓
[当前对话历史]
  ↓
[用户的最新输入]
```

这个顺序是有讲究的。越靠前的内容，在模型处理时的「注意力权重」通常越高（具体取决于模型架构，但一般来说位置越靠前越不容易被遗忘）。所以 Hermes 把最重要的系统指令放在最前面，把动态的对话历史放在后面。

让我们看一个具体的例子。假设你正在使用 Hermes 分析一份财报，上下文窗口的内容可能如下（简化版）：

```
System: 你是一个专业的财务分析助手...
SOUL.md: [用户的个人偏好和角色设定]
Skill: financial_analysis v2.1 [激活的财务分析技能]
MEMORY.md: 用户关注白酒行业，上次分析的是贵州茅台(600519)
冷记忆片段: [上次对话中关于ROE分析的结论]

User: 帮我分析一下五粮液今年的三季报
Assistant: 好的，我来获取五粮液(000858)的最新三季报数据...
[工具调用结果：获取到的财务数据]

User: 和茅台对比一下毛利率
Assistant: [分析中...]
```

每次你发送新消息，或者 Agent 执行工具调用后，这个上下文窗口都会被更新。窗口里的内容越多，发送给大模型的 Token 数就越多，响应时间越长，成本也越高。

> **Tips**
>
> 如果你使用的是按 Token 计费的商业模型（如 GPT-4、Claude），上下文窗口里的每一条信息都在计费范围内——包括系统提示、SOUL.md、MEMORY.md、历史对话。所以理解上下文构成，能帮助你优化成本。使用 `hermes status` 命令可以查看当前会话的 Token 使用情况。

### 3.2 Token 窗口管理策略

Token 窗口管理是热记忆的核心挑战。想象你的上下文窗口是一个容量固定的背包（比如能装 1000 个 Tokens），每次对话都在往里面塞东西。当背包快满的时候，你该怎么办？

Hermes 采用了一套智能的 Token 管理策略，主要包含以下几个机制：

**机制一：固定保留区 + 动态对话区**

上下文窗口被划分为两个区域：

- **固定保留区**：系统提示、SOUL.md、温记忆（MEMORY.md + USER.md）。这部分内容每次请求都必须包含，占用固定的 Token 预算。
- **动态对话区**：实际的对话历史和工具调用结果。这部分是弹性变化的，当空间不足时需要被压缩或截断。

假设你使用的是一个 32K 上下文的模型：

```
总容量：32,000 Tokens
固定保留区：~2,000 Tokens（系统提示 + SOUL.md + 温记忆）
动态对话区可用：~30,000 Tokens
```

**机制二：FIFO（先进先出）截断**

当动态对话区超出容量时，Hermes 会采用「先进先出」的策略，截断最早的对话消息。也就是说，你最早说的话会被逐渐「遗忘」，最近的消息会被保留。

但这里有个重要的例外：**工具调用和工具调用结果通常会被优先保留**。因为它们是 Agent 实际执行任务的证据，如果丢失了，Agent 可能会重复执行已经做过的事情。

**机制三：智能压缩（/compact）**

这是 Hermes 最具特色的功能之一。当对话越来越长时，你可以主动执行 `/compact` 命令，让 Agent 对当前会话的内容进行智能压缩。具体过程如下：

1. Agent 回顾整个对话历史
2. 识别出关键信息（决策、结论、重要的中间结果）
3. 用简洁的语言把这些关键信息总结成一段话
4. 将总结替换掉原来的长对话，释放大量 Token 空间
5. 把这次压缩的摘要存入 SQLite，方便以后检索

`/compact` 的效果通常非常显著。一个 50 轮对话、消耗 15,000 Tokens 的会话，经过压缩后可能只需要 500 Tokens 就能保留全部关键信息。

> **注意**
>
> `/compact` 是一个「有损压缩」。它会丢弃对话中的细节（比如具体的思考过程、不重要的闲聊），只保留结论和关键决策。如果你需要保留完整的对话记录用于审计或复盘，建议在执行 `/compact` 之前先用 `hermes export` 导出会话。

### 3.3 /compact 命令：智能压缩上下文

`/compact` 是 Hermes 最强大的热记忆管理工具之一。让我们深入了解它的使用方法和内部逻辑。

**基本用法：**

在 Hermes 的交互界面中，任何时候你都可以输入：

```bash
/compact
```

Agent 会立即开始压缩过程。你会看到类似这样的输出：

```
正在压缩会话上下文...
✓ 已识别 12 个关键决策点
✓ 已提取 5 条重要结论
✓ 已总结 3 个工具调用结果
压缩完成：15,420 Tokens → 486 Tokens（节省 96.8%）
摘要已保存至数据库，ID: conv_20260421_001
```

**什么时候应该用 /compact？**

| 场景 | 建议 |
|------|------|
| 对话已经超过 30 轮 | 强烈建议使用 |
| 工具调用超过 10 次 | 建议使用 |
| 上下文 Token 超过总容量的 70% | 必须使用 |
| 感觉 Agent 开始「遗忘」早期内容 | 立即使用 |
| 正在执行一个长期、多步骤的任务 | 每完成一个阶段就使用一次 |
| 对话即将结束，想保留关键信息 | 结束前使用 |

**自动压缩配置：**

你可以在 `config.yaml` 中设置自动压缩的触发条件：

```yaml
# config.yaml 中的记忆相关配置
memory:
  # 当上下文 Token 数超过此阈值时自动触发压缩
  auto_compact_threshold: 12000
  
  # 是否启用自动压缩
  auto_compact_enabled: true
  
  # 压缩时保留的最近消息数（这些消息不会被压缩）
  preserve_recent_messages: 5
```

> **常见问题**
>
> **Q：/compact 会删除我的原始对话记录吗？**
>
> A：不会。`/compact` 只会影响当前加载到上下文窗口中的内容。完整的原始对话仍然保存在 SQLite 数据库中，你可以随时用 `hermes history` 查看完整记录。压缩只是让当前工作内存更精简。
>
> **Q：压缩后 Agent 还能回答关于细节的问题吗？**
>
> A：如果细节在压缩过程中被丢弃了，Agent 就无法直接回答。但它可以通过检索冷记忆（SQLite 中的完整记录）来查找原始细节。你可以明确告诉它「去数据库里查一下详细记录」。

### 3.4 上下文溢出时的处理机制

当上下文窗口达到极限时，Hermes 有一套优雅的处理机制，确保系统不会崩溃，同时尽量保留最重要的信息。

**溢出检测：**

每次发送请求前，Hermes 会计算当前上下文的 Token 数。如果加上即将发送的消息后超过模型容量，就会触发溢出处理。

**处理流程（按优先级）：**

```
1. 检查是否可以使用 /compact
   ↓ 是 → 自动执行压缩
   ↓ 否（压缩后仍然溢出）→ 下一步

2. 截断动态对话区（FIFO）
   - 从最早的对话开始删除
   - 保留工具调用和结果
   - 保留最近的 3-5 轮对话
   ↓ 仍然溢出 → 下一步

3. 缩减冷记忆注入
   - 减少检索到的历史片段数量
   - 缩短每个片段的长度
   ↓ 仍然溢出 → 下一步

4. 缩减温记忆注入
   - 临时不加载 USER.md（只保留 MEMORY.md）
   ↓ 仍然溢出（极罕见）→ 下一步

5. 报错提示用户
   - "当前会话上下文已达到极限，建议开启新会话"
```

这种分层降级策略确保了在绝大多数情况下，系统都能自动恢复，而不需要用户手动干预。

### 3.5 最佳实践：如何让单次会话效率最高

经过前面的讲解，你应该已经理解了热记忆的工作原理。下面是经过社区大量实践总结出的最佳实践：

**实践一：定期使用 /compact**

不要让对话无限增长。建议每完成一个任务阶段就压缩一次。就像你写完一个章节就保存一次文档一样，这是良好的工作习惯。

```bash
# 示例：一个多阶段任务中的压缩策略
User: 帮我分析一下 Q3 的财报数据
[Agent 执行分析...]
User: /compact  # 第一阶段完成，压缩

User: 基于刚才的分析，生成一份 PPT 大纲
[Agent 生成大纲...]
User: /compact  # 第二阶段完成，压缩

User: 把大纲导出为 Markdown 文件
[Agent 导出...]
```

**实践二：善用 MEMORY.md 记录跨会话信息**

如果你知道某个信息在后续会话中还会用到，直接告诉 Agent 把它写入 MEMORY.md：

```bash
User: 把这个项目的 Git 地址记住：github.com/mycompany/project-alpha
Agent: 已记录到 MEMORY.md

User: 以后提到 "alpha 项目" 就是指这个仓库
Agent: 已更新 MEMORY.md
```

这样下次会话开始时，Agent 就会自动知道这些信息，不需要你重复交代。

**实践三：避免在单一会话中处理过多不相关的任务**

如果你刚让 Agent 分析了财报，接下来想让它帮你写一封邮件，建议开启新会话。因为财报分析的上下文对写邮件没有帮助，反而会挤占 Token 空间。你可以这样：

```bash
# 不推荐：在同一个会话中混杂不相关任务
User: 分析财报...
[20轮对话后...]
User: 对了，帮我写封邮件给李总  # 浪费Token

# 推荐：任务切换时开启新会话
User: /new  # 开启新会话
User: 帮我写封邮件给李总...
```

**实践四：明确指定需要保留的信息**

在复杂任务中，如果你知道某些中间结果特别重要，可以明确告诉 Agent：

```bash
User: 这个 API 的认证令牌很重要，请把它记录到 MEMORY.md 中
Agent: 已记录：API认证令牌 = sk-abc123...

User: 刚才生成的那个 SQL 查询语句，摘要里一定要保留
Agent: 已将该 SQL 查询标记为重要，压缩时会优先保留
```

**实践五：关注 Token 使用情况**

定期使用 `hermes status` 查看当前 Token 消耗：

```bash
$ hermes status

会话状态
========
会话ID: session_20260421_001
模型: claude-3-7-sonnet-20250219
上下文窗口: 32,000 Tokens
已使用: 18,450 Tokens (57.7%)
固定保留区: 2,100 Tokens
动态对话区: 16,350 Tokens
建议: 上下文使用率较高，建议在 80% 前执行 /compact
```

当使用率超过 70% 时，就应该考虑压缩或结束当前任务了。

> **Tips**
>
> 很多用户有个误区：认为「单轮对话越长越好」。实际上，Agent 的效率不在于单轮对话的长度，而在于信息的组织方式。把一个大任务拆分成多个小任务，每个任务在独立的压缩单元中完成，整体效率往往更高。

## 4. 第二层：持久记忆（温记忆）

### 4.1 MEMORY.md 文件详解

MEMORY.md 是 Hermes 持久记忆系统的核心文件之一。它的作用就像是 Agent 的「随身记事本」——里面记录着与当前任务、当前项目、当前环境相关的关键信息。每次会话开始时，这个记事本的内容都会被加载到上下文中，让 Agent 能够「记得」之前交代过的重要事项。

#### 4.1.1 文件结构与格式

MEMORY.md 采用纯 Markdown 格式存储，这有两个好处：一是人类可以直接阅读和编辑，二是 Markdown 的结构性让 Agent 更容易理解其中的内容层次。

默认情况下，MEMORY.md 位于：

```
~/.hermes/memory/MEMORY.md
```

文件的基本结构如下：

```markdown
# Memory

## Project Context
- 项目名称：智能客服系统升级
- 技术栈：Python 3.11 + FastAPI + PostgreSQL
- Git 仓库：github.com/company/customer-service-v2

## Key Decisions
- 使用 JWT 进行身份认证，Token 有效期 24 小时
- 数据库迁移工具选用 Alembic，不选 Django ORM
- API 响应格式统一为 JSON:API 规范

## Important Notes
- 生产环境数据库密码不要在对话中明文展示
- 测试服务器地址：http://test-api.company.internal
- 部署前必须先跑完整测试套件

## Current Focus
- 正在实现订单查询接口的缓存层（Redis）
- 预计本周五完成压力测试
```

你可以看到，这个文件分为几个逻辑区块（用 `##` 二级标题分隔），每个区块下面用列表形式记录具体事项。这种结构既清晰又节省空间。

> **注意**
>
> MEMORY.md 的容量被限制在约 2200 字符（具体值可能因版本略有差异）。这不是一个硬性的技术限制，而是一个「软约束」——当内容接近上限时，Agent 会主动压缩或合并旧条目。如果你的 MEMORY.md 经常写满，说明记录的信息不够精炼，需要学习如何提取「真正重要的」信息。

#### 4.1.2 自动写入机制

MEMORY.md 最智能的地方在于：Agent 会自动判断哪些信息值得记录，哪些不值得。

当你与 Agent 对话时，系统会在后台运行一个「记忆评估器」。这个评估器会分析对话内容，根据以下标准判断信息的重要性：

**高优先级（几乎一定会记录）：**

- 用户明确说「记住这个」「记下来」「以后都要用」
- 关键配置信息（API 密钥、服务器地址、数据库连接串）
- 重要的决策和结论（「我们决定用方案 A 而不是方案 B」）
- 项目相关的上下文（项目名称、技术栈、特殊要求）

**中优先级（根据上下文判断是否记录）：**

- 用户的偏好设置（「我喜欢深色主题」「用中文回复」）
- 中间计算结果（如果后续步骤会用到）
- 错误和解决方案（「上次是因为权限不足导致的」）

**低优先级（通常不记录）：**

- 临时性的查询（「今天北京天气怎么样」）
- 已经被执行的命令的具体输出（保留命令本身，不保留输出）
- 闲聊内容

自动写入的触发时机通常是在：

1. **用户明确指示记录时**：即时写入
2. **任务成功完成时**：总结关键信息并写入
3. **发生重要决策时**：即时写入
4. **会话结束时**：回顾整个对话，补录遗漏的重要信息

#### 4.1.3 手动编辑方法

虽然自动写入很方便，但有时候你需要手动编辑 MEMORY.md。Hermes 提供了几种方式：

**方式一：通过命令行直接编辑**

```bash
# 使用你喜欢的编辑器打开 MEMORY.md
nano ~/.hermes/memory/MEMORY.md
# 或者
vim ~/.hermes/memory/MEMORY.md
# 或者
code ~/.hermes/memory/MEMORY.md  # VS Code
```

**方式二：在对话中让 Agent 帮你编辑**

```bash
User: 帮我在 MEMORY.md 里加一条：测试环境的 Redis 端口是 6380
Agent: 已添加。

User: 把 MEMORY.md 里关于 JWT 的那条删掉，我们改用 Session 了
Agent: 已删除 JWT 相关条目，并添加了新条目：身份认证改为 Session-Cookie 方案。

User: 给我看看现在的 MEMORY.md
Agent: [展示当前 MEMORY.md 的完整内容]
```

**方式三：使用 /memory 命令系列（详见第7章）**

```bash
/memory add "生产服务器 IP: 192.168.1.100"
/memory remove "JWT"  # 删除包含"JWT"的条目
/memory show  # 显示当前内容
```

#### 4.1.4 容量限制（~2200字符）与管理策略

2200 字符大约是 700-800 个汉字，或者 300-400 个英文单词。这个容量看起来不大，但如果使用得当，足以记录一个中等复杂度项目的所有关键信息。

**高效利用容量的技巧：**

1. **使用列表而非段落**：列表更节省空间，也更容易被 Agent 解析

```markdown
# 好：简洁的列表
- API 基地址: https://api.example.com/v2
- 认证方式: Bearer Token
- 超时设置: 30s

# 差：冗长的段落
API 的基地址是 https://api.example.com/v2，这是生产环境的地址。
认证方式我们选用了 Bearer Token，因为相比 Basic Auth 它更安全。
请求的超时时间设置为 30 秒，这是考虑到网络延迟和服务器响应速度...
```

2. **定期清理过期信息**：项目结束后，把不再相关的条目删掉

3. **合并相似条目**：如果有三条都是关于数据库配置的，合并成一条

4. **使用缩写和代码**：技术术语用标准缩写，节省空间

```markdown
# 好
- DB: PostgreSQL 15, host=db.internal, port=5432
- ORM: SQLAlchemy 2.0 + Alembic

# 差
- 数据库使用的是 PostgreSQL 的第 15 个版本...
- 对象关系映射工具选用了 SQLAlchemy...
```

5. **分级管理**：如果同时处理多个项目，可以用不同的 MEMORY.md 文件

```bash
# 为不同项目使用不同的记忆文件
~/.hermes/memory/MEMORY.md          # 默认
~/.hermes/memory/project_a_MEMORY.md # 项目A
~/.hermes/memory/project_b_MEMORY.md # 项目B
```

在对话中切换：

```bash
User: 切换到项目 A 的上下文
Agent: 已加载 project_a_MEMORY.md
```

#### 4.1.5 实际 MEMORY.md 示例

以下是一个真实的 MEMORY.md 示例（来自一个实际使用 Hermes 进行 Web 开发的开发者）：

```markdown

### 4.2 USER.md 文件详解

如果说 MEMORY.md 是「项目记事本」，那 USER.md 就是「用户画像档案」。它记录的不是项目信息，而是你——作为用户的偏好、习惯、行为模式和个性化需求。

#### 4.2.1 用户画像的自动建模

Hermes 最令人惊叹的功能之一，就是它能够「观察」你的行为，自动构建用户画像。这个过程不需要你做任何特殊操作，完全在后台静默进行。

自动建模的数据来源包括：

1. **显式反馈**：你直接说「我喜欢用中文」「请用 TypeScript」
2. **隐式行为**：
   - 你经常查询什么类型的信息（技术文档 > 新闻 > 娱乐）
   - 你倾向什么编程语言（Python 比 JavaScript 用得多）
   - 你通常在什么时间段使用 Agent（上午 vs 深夜）
   - 你对 Agent 的回复做什么修改（经常删减冗余内容 = 喜欢简洁风格）
3. **错误模式**：你经常纠正 Agent 的某些错误类型（比如混淆两个相似的概念）
4. **任务偏好**：你更倾向于让 Agent 自主决策，还是喜欢逐步确认

这些信息会被提炼成简洁的条目，写入 USER.md。

#### 4.2.2 记录哪些信息

USER.md 中常见的信息类型包括：

```markdown
# User Profile

## Communication
- Language: 中文为主，技术术语保留英文

这些信息的价值在于：它们让 Agent 能够「像你的老搭档一样」与你协作。当你说「帮我写个脚本」，Agent 会自动用 Python（你的首选语言）；当你问一个技术问题，Agent 知道你的水平层次，不会从最基础的概念讲起；当你让它审查代码，它会自动检查类型注解和测试覆盖率。

#### 4.2.3 容量限制（~1375字符）

USER.md 的容量限制（约 1375 字符）比 MEMORY.md 更严格。这是因为：

1. 用户画像的变化频率比项目信息低，不需要频繁更新
2. 用户画像应该高度精炼——只记录最核心的特征
3. 限制容量可以避免画像过于「臃肿」，导致加载时消耗过多 Token

当 USER.md 接近容量上限时，Agent 会启动「画像合并」机制：

- 把两个相似的偏好合并成一条更通用的
- 删除已经不再适用的旧偏好
- 把低频偏好降级或删除

### 4.3 持久记忆的读取时机与优先级

了解了 MEMORY.md 和 USER.md 的内容后，我们来看看它们是如何被加载到会话中的。

**读取时机：**

1. **会话开始时**：每次启动新会话，Hermes 都会读取这两个文件
2. **文件被修改后**：如果你在对话中让 Agent 修改了记忆文件，修改会立即生效
3. **执行 /refresh 命令时**：强制重新加载所有记忆

**加载优先级（高到低）：**

```
1. 系统提示（System Prompt）
2. SOUL.md
3. 激活的 Skill 说明
4. USER.md（用户画像）
5. MEMORY.md（项目记忆）
6. 冷记忆检索结果
7. 当前对话历史
8. 用户输入
```

这个优先级顺序的设计理由：

- **系统提示和 SOUL.md 在最前面**：它们是 Agent 的「基本行为准则」，必须最先被看到
- **USER.md 在 MEMORY.md 前面**：你的个人偏好比项目信息更根本。比如你喜欢简洁回复，这个偏好应该优先于某个项目要求详细文档的需求
- **温记忆在冷记忆前面**：当前项目的直接记忆，比历史检索到的片段更相关
- **对话历史在最后面**：动态变化的内容放在最后，符合 FIFO 截断策略（如果溢出，最先丢失的是最早的对话）

> **注意**
>
> 如果你发现 Agent 在某些情况下「记不住」你的偏好，可能是因为 MEMORY.md 中的某条项目级设置覆盖了 USER.md 中的用户级设置。例如，USER.md 说你喜欢简洁回复，但 MEMORY.md 中写着「这个项目需要详细的文档说明」，Agent 可能会为了项目需求而提供更详细的回复。这是预期行为——项目上下文优先于个人偏好。

## 5. 第三层：检索记忆（冷记忆）

### 5.1 SQLite 数据库架构详解

冷记忆是 Hermes 记忆系统中技术含量最高的部分，也是它与大多数其他 Agent 框架拉开差距的地方。让我们从最底层的存储架构开始，一层一层地向上理解。

#### 5.1.1 表结构设计

Hermes 使用 SQLite 作为冷记忆的存储引擎。SQLite 的选择是深思熟虑的：

- **零配置**：不需要安装独立的数据库服务，一个 `.db` 文件就是完整的数据库
- **跨平台**：Windows、macOS、Linux 都能用
- **高性能**：对于本地单用户场景，SQLite 的性能完全够用
- **事务安全**：支持 ACID，数据不会损坏
- **内置 FTS5**：SQLite 从 3.19.0 版本开始内置 FTS5 全文搜索扩展

数据库文件默认位于：

```
~/.hermes/memory/hermes_memory.db
```

核心表结构如下（简化版）：

```sql
-- 会话表：记录每一次对话会话
CREATE TABLE conversations (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id TEXT UNIQUE NOT NULL,    -- 会话唯一标识
    started_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    ended_at TIMESTAMP,
    summary TEXT,                        -- LLM 生成的会话摘要
    tags TEXT,                           -- 标签，用逗号分隔
    total_messages INTEGER DEFAULT 0,    -- 消息总数
    total_tokens INTEGER DEFAULT 0       -- Token 总数
);

-- 消息表：存储每一条消息
CREATE TABLE messages (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    conversation_id INTEGER NOT NULL,
    role TEXT NOT NULL,                  -- 'user' 或 'assistant'
    content TEXT NOT NULL,
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    tokens INTEGER,                      -- 该消息的 Token 数
    FOREIGN KEY (conversation_id) REFERENCES conversations(id)
);

-- 工具调用表：记录 Agent 的每一次工具使用
CREATE TABLE tool_calls (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    conversation_id INTEGER NOT NULL,
    tool_name TEXT NOT NULL,             -- 工具名称
    arguments TEXT,                      -- 调用参数（JSON 格式）
    result TEXT,                         -- 调用结果
    success BOOLEAN DEFAULT 1,
    duration_ms INTEGER,                 -- 执行耗时（毫秒）
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (conversation_id) REFERENCES conversations(id)
);

-- 文件操作表：记录 Agent 的文件读写操作
CREATE TABLE file_operations (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    conversation_id INTEGER NOT NULL,
    operation TEXT NOT NULL,             -- 'read', 'write', 'edit', 'delete'
    file_path TEXT NOT NULL,
    content_hash TEXT,                   -- 内容哈希（用于检测变更）
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (conversation_id) REFERENCES conversations(id)
);

-- 记忆片段表：存储提取出的关键记忆
CREATE TABLE memory_fragments (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    source_conversation_id INTEGER,
    content TEXT NOT NULL,               -- 记忆内容
    importance_score REAL,               -- 重要性评分（0-1）
    category TEXT,                       -- 分类：'fact', 'decision', 'preference', etc.
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    last_accessed TIMESTAMP,
    access_count INTEGER DEFAULT 0,      -- 被检索到的次数
    FOREIGN KEY (source_conversation_id) REFERENCES conversations(id)
);
```

这个设计的精妙之处在于：**不仅存储了「说什么」，还存储了「做什么」**。工具调用记录和文件操作记录，让 Agent 能够回答这样的问题：「上次我是怎么解决那个问题的？」「我修改过这个文件的哪几行？」

#### 5.1.2 存储的数据类型

Hermes 的冷记忆数据库不仅存储对话文本，还记录了丰富的元数据：

**对话类数据：**

- 完整的用户输入和 Agent 回复
- 每次对话的时间戳和持续时间
- 对话中使用的模型和参数
- 会话级别的 Token 统计

**工具类数据：**

- 工具名称和调用参数
- 工具执行结果（成功/失败）
- 执行耗时
- 调用链（一个工具调用触发了另一个工具）

**文件类数据：**

- 读写的文件路径
- 操作类型（创建/修改/删除）
- 内容哈希值（用于追踪变更）
- 操作时间

**知识类数据：**

- 从对话中提取的关键事实
- 用户明确记录的备忘
- 自动生成的会话摘要
- 用户偏好和行为的推断结果

#### 5.1.3 数据量增长估算

很多用户关心：这个数据库会不会无限膨胀？我需要手动清理吗？

让我们做一个估算。假设你是一个「中等强度」的用户：

- 每天与 Agent 对话 20 轮
- 每轮平均 200 Tokens
- 每天使用 5 次工具调用
- 每天产生 3 次文件操作

一天的数据量：

```
消息：20 轮 × 2 条（用户+Agent）= 40 条消息
工具调用：5 次
文件操作：3 次
文本总量：20 × 200 = 4,000 Tokens ≈ 12,000 字符
```

一年的数据量：

```
消息：40 × 365 = 14,600 条
工具调用：5 × 365 = 1,825 次
文件操作：3 × 365 = 1,095 次
文本总量：12,000 × 365 ≈ 4,380,000 字符 ≈ 4.4 MB
```

加上索引和元数据，一年的数据库大小大约在 **10-20 MB** 左右。这个体积在现代电脑上完全微不足道——你的微信聊天记录一个月可能就几百 MB。

即使你是「重度用户」（每天 100 轮对话），一年的数据量也就在 50-100 MB 级别。SQLite 能轻松处理 GB 级别的数据库，所以你完全不需要担心存储空间的问题。

> **Tips**
>
> 虽然存储空间不是问题，但检索速度会随着数据量增加而有所下降。Hermes 内置了自动维护机制（详见第7章），包括索引优化和旧数据归档。你也可以手动设置保留策略（详见第10章）。

### 5.2 FTS5 全文索引

FTS5（Full-Text Search 5）是 SQLite 内置的全文搜索引擎。它是 Hermes 冷记忆检索能力的核心支柱。理解 FTS5，就理解了 Hermes 如何做到「瞬间从海量历史中找到相关内容」。

#### 5.2.1 什么是 FTS5（通俗解释）

我们先不用技术术语，用一个类比来解释 FTS5。

想象你有一个巨大的图书馆，里面有 10 万本书。如果你要找一本「讲 Python 编程的书」，传统的方式是一本一本地翻看——这显然太慢了。

于是你做了一个「目录卡片」系统：每本书你都在卡片上记录书名、作者、关键词。所有卡片按字母顺序排列在一个巨大的抽屉柜里。当你要找 Python 的书时，直接去 "P" 开头的抽屉里找就行了。这就是「索引」。

但 FTS5 比这个更强大。它不仅记录「这本书里有 Python 这个词」，还记录「Python 这个词在书的第 3 页第 2 段出现了」。更进一步，它能理解「Python 编程」「Python 开发」「Python 语言」都是相关的，甚至能处理拼写错误（比如你把 Python 拼成了 Pthon）。

在 Hermes 的语境中，FTS5 做的事情是：

1. 把每一条历史消息、每一个工具调用结果、每一个文件操作记录，都「拆开」成单词
2. 建立一个巨大的「倒排索引」——记录每个词出现在哪些记录中
3. 当你搜索时，瞬间找到包含这些词的所有记录
4. 按照相关度排序，把最相关的结果返回给你

#### 5.2.2 索引建立过程

FTS5 索引的建立是自动的，你不需要手动操作。但为了让你理解背后的机制，我们来看看索引是如何构建的：

```sql
-- 创建 FTS5 虚拟表（Hermes 内部自动完成）
CREATE VIRTUAL TABLE messages_fts USING fts5(
    content,                    -- 要索引的文本内容
    conversation_id UNINDEXED,  -- 不索引，仅存储关联ID
    timestamp UNINDEXED         -- 不索引，仅存储
);

-- 当新消息写入时，自动添加到索引（触发器自动完成）
INSERT INTO messages_fts(content, conversation_id, timestamp)
VALUES ('用户输入的内容...', 123, '2026-04-21 10:30:00');
```

索引建立的过程：

1. **分词（Tokenization）**：将文本拆分成单词。对于中文，需要使用专门的分词器（如 `jieba` 或 SQLite 的 `icu` 分词器）。Hermes 默认使用 `porter` 分词器处理英文，`simple` 分词器处理中文。

2. **标准化（Normalization）**：
   - 统一大小写（"Python" 和 "python" 视为同一个词）
   - 去除标点符号
   - 词干提取（"running" 和 "runs" 都归为 "run"）

3. **去停用词**：过滤掉「的」「是」「the」「and」等无意义的词，减少索引体积

4. **构建倒排索引**：记录每个词在哪些文档中出现，以及出现的位置

#### 5.2.3 检索速度与效果

FTS5 的检索速度非常快。在普通笔记本电脑上，从 100 万条记录中检索关键词，通常只需要 **几毫秒到几十毫秒**。

让我们看一个实际的检索场景：

假设你在半年前的一个会话中讨论过「如何用 Docker 部署 PostgreSQL」。现在你忘了具体的命令，只记得「Docker」和「PostgreSQL」两个关键词。你开启新会话，输入：

```bash
User: 上次部署 PostgreSQL 的 Docker 命令是什么来着？
```

Hermes 的内部检索过程：

```sql
-- 1. 提取关键词
-- 分析输入："部署 PostgreSQL 的 Docker 命令"
-- 提取关键词：PostgreSQL, Docker, 部署, 命令
-- 过滤停用词：保留 PostgreSQL, Docker, 命令

-- 2. FTS5 检索
SELECT 
    content,
    conversation_id,
    rank  -- FTS5 内置的相关度评分
FROM messages_fts
WHERE messages_fts MATCH 'PostgreSQL AND Docker AND 命令'
ORDER BY rank
LIMIT 5;
```

检索结果可能是：

```
结果1 (rank: 1.2): "docker run -d --name pg -e POSTGRES_PASSWORD=secret postgres:15"
结果2 (rank: 2.5): "我用 Docker 部署了 PostgreSQL，配置如下..."
结果3 (rank: 3.1): "Docker Compose 文件里 PostgreSQL 的部分..."
```

然后 Hermes 会把这些相关片段注入到当前上下文中，Agent 就能回答你的问题：

```
Agent: 根据之前的记录，你使用的 Docker 命令是：

docker run -d \
  --name pg \
  -e POSTGRES_PASSWORD=secret \
  -p 5432:5432 \
  postgres:15

这是你在 2025 年 10 月 15 日的会话中使用的配置。
```

#### 5.2.4 与传统全文搜索的对比

如果你做过信息检索相关的工作，可能会问：FTS5 和 Elasticsearch、Solr 等传统全文搜索引擎相比如何？

| 特性 | FTS5 (SQLite) | Elasticsearch | 传统 SQL LIKE |
|------|--------------|---------------|-------------|
| 部署复杂度 | 零（内置） | 高（需独立服务） | 零 |
| 检索速度 | 毫秒级 | 毫秒级 | 秒级（大数据量时） |
| 相关性排序 | 支持 | 支持（更强大） | 不支持 |
| 模糊匹配 | 基本支持 | 强大 | 不支持 |
| 分布式 | 不支持 | 支持 | 不支持 |
| 数据量上限 | GB 级 | TB 级 | 无索引时很慢 |
| 维护成本 | 极低 | 高 | 低 |

对于 Hermes 的使用场景（个人或小型团队的本地 Agent），FTS5 在绝大多数情况下已经绰绰有余。它的「零部署」优势尤其重要——你只需要安装 Hermes，全文检索能力就自动可用了。

> **注意**
>
> FTS5 的一个局限是中文分词能力相对较弱（默认的 `simple` 分词器对中文只是按字拆分）。Hermes 通过预处理优化了这个问题，但在某些复杂的中文查询场景下，检索效果可能不如专门的搜索引擎。如果你主要用中文工作且对检索精度要求极高，可以考虑通过 Mem0 等外部方案增强检索能力。

### 5.3 按需检索机制

按需检索是 Hermes 冷记忆系统最具创新性的设计。传统的记忆方案往往在会话开始时就把所有历史记录加载到上下文中，而 Hermes 采用了一种更智能的方式。

#### 5.3.1 新对话时不再全量加载

让我们对比一下两种策略：

**传统策略（全量加载）：**

```
会话开始
  ↓
加载过去 100 个会话的所有消息（50,000 Tokens）
  ↓
加上系统提示和当前输入（2,000 Tokens）
  ↓
总上下文：52,000 Tokens
  ↓
模型处理（慢、贵、容易混乱）
```

**Hermes 策略（按需检索）：**

```
会话开始
  ↓
只加载固定保留区（系统提示 + SOUL.md + 温记忆）（2,000 Tokens）
  ↓
分析用户输入，提取关键词
  ↓
FTS5 检索，只取最相关的 3-5 条记录（~500 Tokens）
  ↓
总上下文：~2,500 Tokens
  ↓
模型处理（快、便宜、聚焦）
```

差异是惊人的：传统策略可能一次请求就消耗 50,000+ Tokens，而 Hermes 的策略通常只消耗 2,000-3,000 Tokens。按照 GPT-4 的定价（输入 $0.03/1K Tokens），每轮对话的成本差异是 $1.5 vs $0.075——相差 20 倍！

#### 5.3.2 根据当前话题智能检索

Hermes 的检索不是简单的关键词匹配，而是一个多阶段的智能过程：

**阶段一：查询理解**

当用户输入一条消息时，Hermes 首先分析这条消息的意图和主题：

```
用户输入: "帮我看看上次那个支付接口的 bug 修好了没"

提取关键词: 支付接口, bug, 修好
推断主题: code_review, bug_fix, payment
时间范围: 最近（"上次"暗示近期）
```

**阶段二：多维度检索**

根据理解的结果，从不同维度并行检索：

```sql
-- 维度1：消息内容匹配
SELECT * FROM messages_fts 
WHERE messages_fts MATCH '支付 AND (bug OR 修复 OR 修好)'
ORDER BY rank LIMIT 3;

-- 维度2：工具调用记录
SELECT * FROM tool_calls
WHERE tool_name IN ('code_review', 'file_edit')
  AND (result LIKE '%支付%' OR arguments LIKE '%支付%')
ORDER BY timestamp DESC LIMIT 2;

-- 维度3：文件操作记录
SELECT * FROM file_operations
WHERE file_path LIKE '%payment%' OR file_path LIKE '%pay%'
ORDER BY timestamp DESC LIMIT 2;
```

**阶段三：相关度融合与排序**

将多个维度的结果合并，按综合相关度排序：

```
相关度评分 = 
  FTS 排名分数 × 0.4 +       -- 文本匹配度
  时间衰减因子 × 0.3 +       -- 越新的越重要
  工具成功率 × 0.2 +         -- 成功的操作更重要
  用户反馈权重 × 0.1         -- 用户标注重要的优先
```

**阶段四：片段提取与注入**

从选中的记录中提取最相关的片段（不是整段对话，而是包含关键词的那几句话），然后格式化为上下文注入：

```
[相关历史片段 #1]
来源：2026-04-15 的会话
内容：用户报告支付接口在并发请求时出现 500 错误。Agent 建议使用连接池并增加了重试逻辑。

[相关历史片段 #2]
来源：2026-04-16 的会话
内容：修复了 payment/service.py 中的 race condition，提交了 commit a3f7d2e。
```

#### 5.3.3 只提取相关片段

这是一个关键设计：不是把整个历史会话都加载进来，而是只提取「相关的那几句话」。

这样做的好处：

1. **极大节省 Token**：一条完整的会话可能 500 Tokens，但相关片段可能只有 50 Tokens
2. **减少噪声**：不相关的历史信息不会干扰当前任务
3. **提高精度**：模型只看到了真正需要的信息，不会被无关内容分散注意力

提取算法会围绕关键词向外扩展，取包含关键词的句子和前后各一句，形成一个「语义完整的片段」。如果关键词出现在工具调用结果中，会同时提取工具名称、参数和结果摘要。

#### 5.3.4 API 调用成本大幅降低

按需检索机制带来的最大实际好处，就是 API 调用成本的大幅降低。

假设你使用 Claude 3.7 Sonnet（$0.03/1K input tokens），每天进行 50 轮对话：

| 策略 | 每轮平均 Tokens | 每日 Tokens | 每日成本 |
|------|----------------|------------|---------|
| 全量加载 | 25,000 | 1,250,000 | $37.50 |
| Hermes 按需检索 | 3,000 | 150,000 | $4.50 |
| **节省** | **88%** | **88%** | **$33/天** |

一个月下来，Hermes 的按需检索机制可以为你节省约 **$990** 的 API 费用。这不仅仅是省钱——更重要的是，更低的成本让你可以更频繁、更深度地使用 Agent，而不必心疼账单。

> **Tips**
>
> 如果你想进一步降低成本，可以在 `config.yaml` 中调整检索参数：
> ```yaml
> memory:
>   retrieval:
>     max_fragments: 3        # 减少检索到的片段数量（默认5）
>     fragment_max_tokens: 80 # 限制每个片段的长度（默认150）
>     time_decay_days: 30     # 只检索最近30天的记录（默认90）
> ```
> 这些调整会在检索精度和成本之间做权衡。如果你主要处理近期任务，把时间范围缩小可以显著减少检索开销。

### 5.4 LLM 摘要生成

除了 FTS5 全文检索，Hermes 还利用大语言模型本身的能力，为每次会话生成智能摘要。这些摘要既服务于人类用户（快速回顾），也服务于 Agent（高效检索）。

#### 5.4.1 自动生成会话摘要

每次会话结束时（或执行 `/compact` 时），Hermes 会触发摘要生成流程：

```
1. 获取当前会话的完整对话记录
2. 构建摘要提示模板：
   "请为以下对话生成一个简洁的摘要（100字以内）。
    摘要应包括：主要任务、关键决策、重要结论、遗留问题。"
3. 调用 LLM 生成摘要
4. 将摘要保存到 conversations 表的 summary 字段
5. 同时为摘要建立 FTS5 索引
```

生成的摘要示例：

```
会话摘要（2026-04-21）:
用户要求分析 Q1 财报数据。使用 Python 抓取并清洗了 3 家公司的 
财务数据，生成对比图表。关键结论：A 公司毛利率最高（42%）。
遗留：需要补充行业均值数据。
```

#### 5.4.2 摘要的存储与索引

摘要会被存储在两个地方：

1. **conversations 表的 summary 字段**：作为元数据，便于快速浏览
2. **FTS5 索引**：让摘要也能被检索到

当用户搜索时，FTS5 会同时搜索：

- 原始消息内容
- 会话摘要
- 工具调用结果
- 文件操作记录

这意味着，即使用户记不住具体的对话内容，只记得「大概是什么主题」，也能通过摘要找到相关会话。

#### 5.4.3 摘要质量的影响因素

摘要的质量直接影响检索的效果。以下因素会影响摘要质量：

**正向因素：**

- 对话主题明确、任务清晰
- 有明确的决策点和结论
- 用户和 Agent 的互动质量高

**负向因素：**

- 对话过于零散、话题频繁切换
- 大量闲聊和无关内容
- 任务中途被打断

你可以通过以下方式帮助 Agent 生成更好的摘要：

1. **在任务完成时做总结**：「总结一下我们今天做了什么」
2. **给会话打标签**：「给这个会话打个标签：投研-Q1-白酒」
3. **手动修正摘要**：如果你觉得自动摘要不够准确，可以手动编辑

```bash
User: 修改刚才那个会话的摘要，加上「最终输出了一份 Excel 报告」
Agent: 已更新会话摘要。
```

## 6. 外部记忆扩展

虽然 Hermes 内置的记忆系统已经非常强大，但在某些特定场景下，外部记忆扩展可以提供更专业的能力。本节介绍几种主流的外部记忆方案及其集成方法。

### 6.1 Honcho 模块详解

Honcho 是一个专注于「用户行为模式推导」的记忆模块，由 a16z 支持开发。它的核心理念是：通过分析用户的历史行为，推断出用户的隐性偏好和需求，从而让 Agent 的响应更加个性化。

#### 6.1.1 Honcho 的设计理念

Honcho 的设计灵感来自心理学中的「认知建模」理论。它认为，一个真正智能的 Agent 不应该只是「记住用户说过什么」，而应该「理解用户是什么样的人」。

具体来说，Honcho 会：

1. **分析对话模式**：你在什么情况下会打断 Agent？你通常如何纠正 Agent 的错误？
2. **推断认知风格**：你是「先整体后细节」的人，还是「先细节后整体」的人？
3. **识别情感触发点**：哪些类型的话题会让你表现出兴奋、困惑或不耐烦？
4. **建模知识水平**：你在不同领域的能力层次（专家/熟练/新手）

基于这些分析，Honcho 生成一个「用户心理模型」，供 Agent 在后续交互中参考。

#### 6.1.2 行为模式推导机制

Honcho 的行为模式推导分为三个层次：

**第一层：表面模式（Surface Patterns）**

从原始对话中提取可直接观察的模式：

```json
{
  "communication_patterns": {
    "average_response_length": 45,
    "preferred_language": "zh-CN",
    "uses_technical_jargon": true,
    "asks_clarifying_questions": false
  },
  "timing_patterns": {
    "peak_activity_hours": [9, 10, 14, 15],
    "average_session_duration_minutes": 23,
    "typical_task_complexity": "medium"
  }
}
```

**第二层：推断特征（Inferred Traits）**

基于表面模式，推断更深层的用户特征：

```json
{
  "cognitive_style": "analytical",
  "risk_tolerance": "medium",
  "learning_preference": "example-driven",
  "decision_making": "data-driven",
  "expertise_areas": ["backend_development", "system_design"],
  "novice_areas": ["frontend_design", "mobile_development"]
}
```

**第三层：预测模型（Predictive Model）**

基于历史数据，预测用户在未来场景中的行为：

```json
{
  "predicted_behaviors": {
    "when_introducing_new_tech": "will_ask_for_examples",
    "when_facing_errors": "prefers_step_by_step_debugging",
    "when_given_options": "tends_to_choose_performance_over_convenience"
  }
}
```

#### 6.1.3 安装与配置方法

在 Hermes 中集成 Honcho：

```bash
# 1. 安装 Honcho Python 包
pip install honcho-ai

# 2. 在 config.yaml 中启用 Honcho
```yaml
# ~/.hermes/config.yaml
memory:
  external:
    honcho:
      enabled: true
      api_key: "your_honcho_api_key"  # 可选，本地模式可不填
      mode: "local"  # 或 "cloud"
      inference_frequency: "per_session"  # per_message / per_session / per_day
      depth: "full"  # surface / inferred / full
```

# 3. 重启 Hermes
hermes restart
```

如果你选择 `mode: "local"`，所有 Honcho 的数据处理都在本地完成，不会上传到 Honcho 的服务器。但某些高级推断功能可能需要 `cloud` 模式。

#### 6.1.4 实际效果展示

启用 Honcho 前后对比：

**启用前：**

```
User: 帮我写个脚本处理日志文件
Agent: 好的，这是一个 Python 脚本...
[生成一个通用脚本]
```

**启用后（Honcho 已推断出用户偏好）：**

```
User: 帮我写个脚本处理日志文件
Agent: 好的，基于你之前的偏好（喜欢用 Go 写工具脚本、重视错误处理、
      偏好结构化输出），这是一个 Go 脚本...
[生成一个带 error handling、JSON 输出的 Go 脚本]
```

Honcho 的推断不是基于你明确说过的偏好，而是基于你过去的行为模式。比如你可能从来没说过「我喜欢 Go 语言」，但如果你过去 10 次让 Agent 写脚本时 8 次都选了 Go，Honcho 就会推断出这个偏好。

> **注意**
>
> Honcho 是一个相对较新的项目，其推断算法的准确性还在持续改进中。建议把它作为「增强」而非「替代」Hermes 内置的记忆系统。在使用过程中，如果你发现 Honcho 的推断有偏差，可以通过 `honcho feedback` 命令提供纠正反馈。

### 6.2 Mem0 集成

Mem0 是目前最流行的外部记忆平台之一，被描述为「AI 应用的记忆层」。它提供了比 Hermes 内置记忆更强大的检索和管理能力，特别适合需要跨设备同步记忆的场景。

**Mem0 的核心特性：**

1. **混合检索**：同时支持向量检索（语义相似度）和关键词检索（精确匹配）
2. **记忆层级**：自动区分短期记忆、长期记忆和核心事实
3. **多模态**：支持文本、图像、音频等多种类型记忆
4. **跨平台**：记忆可以在不同设备、不同应用间同步
5. **记忆冲突解决**：当新旧记忆矛盾时，自动判断哪个更可靠

**在 Hermes 中集成 Mem0：**

```bash
# 1. 安装 Mem0
pip install mem0ai

# 2. 获取 API 密钥（在 mem0.ai 注册）
# 3. 配置 Hermes
```yaml
memory:
  external:
    mem0:
      enabled: true
      api_key: "m0-your-api-key"
      organization_id: "your-org-id"
      project_id: "your-project-id"
      
      # 检索配置
      retrieval:
        method: "hybrid"  # semantic / keyword / hybrid
        top_k: 5
        
      # 记忆更新配置
      update:
        auto_add: true
        importance_threshold: 0.7
```

# 4. 重启 Hermes
```

**Mem0 与内置记忆的协作模式：**

Hermes 支持「分层记忆」模式：内置记忆处理日常任务，Mem0 处理需要高级检索能力的场景。具体协作方式：

```
用户输入
  ↓
同时查询 Hermes 内置记忆 和 Mem0
  ↓
融合两个来源的结果
  ↓
按相关度排序，去重
  ↓
注入上下文
```

> **Tips**
>
> 如果你同时使用 Mem0 和内置记忆，建议在 `config.yaml` 中设置优先级：
> ```yaml
> memory:
>   retrieval_priority: ["local", "mem0"]  # 先查本地，再查 Mem0
>   # 或
>   retrieval_priority: ["mem0", "local"]  # 先查 Mem0，再查本地
> ```

### 6.3 OpenViking 集成

OpenViking 是一个面向企业级场景的记忆扩展方案，由 Viking Analytics 开发。它专注于解决以下企业痛点：

1. **团队协作记忆**：多个用户共享项目记忆，新成员加入时能快速了解项目上下文
2. **权限控制**：不同角色可以访问不同层级的记忆（公开/团队/个人/机密）
3. **合规审计**：所有记忆操作都有完整的审计日志，满足企业合规要求
4. **数据隔离**：不同客户/项目的数据严格隔离，防止交叉污染

**OpenViking 的架构：**

```
OpenViking Server
├── Project Memory Pool (共享项目记忆)
├── Team Memory Pool (团队知识库)
├── Personal Memory Pool (个人记忆)
└── Audit Log (审计日志)

Hermes Agent ←→ OpenViking Client ←→ OpenViking Server
```

**集成方式：**

```yaml
memory:
  external:
    openviking:
      enabled: true
      server_url: "https://openviking.company.internal"
      api_key: "vk-xxx"
      
      # 企业级配置
      project_id: "project-alpha"
      team_id: "backend-team"
      access_level: "team"  # personal / team / project / admin
      
      # 合规配置
      audit_enabled: true
      data_retention_days: 365
      encryption: "aes-256-gcm"
```

OpenViking 更适合以下场景：

- 中大型开发团队使用 Hermes 协作
- 需要满足数据合规要求的企业
- 项目记忆需要在团队成员间共享
- 需要精细的权限控制

对于个人用户和小团队，Hermes 内置的记忆系统通常已经足够。

### 6.4 其他 5 种外部记忆方案

除了 Honcho、Mem0 和 OpenViking，社区还开发了多种其他记忆扩展方案。根据 Hermes Atlas 官方文档，目前已支持 **8 种社区记忆扩展**，包括 Honcho、Mem0、Hindsight、Supermemory 等。以下是值得了解的方案：

**方案一：Hindsight —— 事后反思增强记忆**

Hindsight 是一款专注于「事后反思」的记忆扩展。它的核心理念是：Agent 不应该只记录「发生了什么」，还应该记录「为什么会这样」以及「下次如何做得更好」。每次会话结束后，Hindsight 会触发一个反思流程，自动生成「经验卡片」，记录成功模式、失败教训和优化建议。

Hindsight 特别适合需要持续改进的复杂工作流场景，比如代码审查、投研分析、内容创作等。

```yaml
memory:
  external:
    hindsight:
      enabled: true
      reflection_depth: "detailed"  # brief / detailed / deep
      auto_reflect: true
      retention: "lessons_only"     # 只保留提炼出的经验教训
```

**方案二：Supermemory —— 超级个人记忆库**

Supermemory 是一个主打「捕获一切」的外部记忆方案。它不仅可以存储对话历史，还能主动抓取你浏览的网页、收藏的推文、阅读的文章，建立统一的个人知识库。通过浏览器插件和 API，Supermemory 能让 Hermes 访问你的整个数字生活轨迹。

适合希望 Agent 拥有「完整上下文」的深度用户。但需注意隐私影响——它确实需要访问大量个人数据。

```yaml
memory:
  external:
    supermemory:
      enabled: true
      api_key: "sm-your-key"
      capture_web_browsing: true
      capture_bookmarks: true
```

**方案三：Chroma + LangChain Memory**

使用 Chroma 向量数据库作为后端，通过 LangChain 的 ConversationBufferMemory 接口集成。适合需要语义检索的场景。

```yaml
memory:
  external:
    chroma:
      enabled: true
      persist_directory: "~/.hermes/chroma_db"
      collection_name: "hermes_memories"
      embedding_model: "sentence-transformers/all-MiniLM-L6-v2"
```

**方案二：Pinecone 云端向量记忆**

Pinecone 是一个托管的向量数据库服务，提供高性能的语义检索。适合不需要本地部署、愿意使用云服务的用户。

优点：检索速度快、支持大规模数据、自动扩缩容
缺点：数据上云、有额外成本、需要网络连接

**方案三：Zep Memory**

Zep 是专门为 LLM 应用设计的长期记忆服务。它的特点是「渐进式摘要」——随着对话进行，自动将短期记忆提炼成长期记忆。

Zep 的独特之处在于它理解「时间」的概念。比如它能回答「用户上周对什么话题最感兴趣？」这种需要时间维度分析的问题。

**方案四：Zep Memory**

Zep 是专门为 LLM 应用设计的长期记忆服务。它的特点是「渐进式摘要」——随着对话进行，自动将短期记忆提炼成长期记忆。Zep 的独特之处在于它理解「时间」的概念。比如它能回答「用户上周对什么话题最感兴趣？」这种需要时间维度分析的问题。

**方案五：本地文件系统记忆（增强版）**

如果你不想用任何外部服务，可以用增强的本地文件方案：

```
~/.hermes/memory/
├── daily/              # 按日期归档
│   ├── 2026-04-21.md
│   ├── 2026-04-20.md
│   └── ...
├── projects/           # 按项目归档
│   ├── project-alpha/
│   ├── project-beta/
│   └── ...
├── topics/             # 按主题归档
│   ├── python-tips.md
│   ├── docker-notes.md
│   └── ...
└── index.json          # 所有文件的索引
```

配合一个简单的检索脚本，就能实现基本的记忆管理。这种方式完全离线，但检索能力较弱。

**方案六：Obsidian / Notion 集成**

如果你已经在使用 Obsidian 或 Notion 作为个人知识库，可以把它们作为 Hermes 的外部记忆：

```yaml
memory:
  external:
    obsidian:
      enabled: true
      vault_path: "~/Documents/Obsidian Vault"
      # Hermes 会索引你的 Obsidian 笔记
      # 并在检索时纳入搜索结果
```

这种方式的最大好处是：你的 Agent 记忆和你的个人知识库合二为一。你在 Obsidian 中记录的读书笔记、会议记录、项目文档，Agent 都能检索到。

### 6.5 选择建议：什么场景用什么方案

面对这么多记忆方案，如何选择？以下是针对不同场景的推荐：

| 使用场景 | 推荐方案 | 理由 |
|---------|---------|------|
| 个人开发者，日常使用 | Hermes 内置记忆 | 零配置、完全本地、有界设计保持专注 |
| 需要语义检索（找「意思相近」的内容） | Mem0 或 Chroma | 向量检索擅长语义匹配 |
| 需要事后反思和经验积累 | Hindsight | 自动生成经验卡片，持续改进 |
| 希望 Agent 拥有完整数字生活上下文 | Supermemory | 捕获浏览历史、收藏、阅读记录 |
| 跨设备同步记忆 | Mem0 Cloud | 云端同步，多设备一致 |
| 企业团队协作用 | OpenViking | 权限控制、审计合规 |
| 深度用户行为分析 | Honcho | 行为模式推断 |
| 已有 Obsidian/Notion 知识库 | 对应集成方案 | 统一知识管理 |
| 极度隐私敏感（如处理医疗/金融数据） | 纯内置记忆 | 数据绝不出本地 |
| 超大规模数据（百万级会话） | Pinecone / 定制方案 | 专业向量数据库的扩展性 |

> **Tips**
>
> 对于绝大多数用户，我的建议是：**先用好 Hermes 内置的记忆系统**。它覆盖了 80% 的使用场景，而且零配置、零成本。当你遇到内置系统无法解决的特定问题时，再考虑引入外部扩展。不要为了「技术炫酷」而增加不必要的复杂度。

## 7. 记忆管理实战

### 7.1 /memory 命令系列详解

Hermes 提供了一套完整的 `/memory` 命令，让你能够像操作数据库一样管理记忆。掌握这些命令，是高效使用 Hermes 的必备技能。

**命令总览：**

```bash
/memory show              # 查看当前记忆状态
/memory add <内容>         # 手动添加一条记忆
/memory remove <关键词>     # 删除包含关键词的记忆条目
/memory edit <id> <新内容>  # 编辑指定记忆
/memory search <关键词>     # 搜索记忆
/memory export [文件名]     # 导出记忆
/memory import <文件名>     # 导入记忆
/memory compact            # 压缩记忆文件
/memory clear              # 清空所有记忆（谨慎使用！）
```

#### /memory show —— 查看当前记忆状态

这是你最常用的命令之一。它会展示当前加载的所有记忆内容：

#### /memory add —— 手动添加记忆

有时候你需要手动添加一些信息到记忆中：

```bash
# 添加到 MEMORY.md
User: /memory add "测试服务器密码: TestPass123! (仅内部使用)"
Agent: 已添加到 MEMORY.md。

# 添加到 USER.md（使用 --user 标志）
User: /memory add --user "最近在学习 Kubernetes"
Agent: 已添加到 USER.md。

# 添加到特定分类
User: /memory add --category "credentials" "AWS Access Key: AKIA..."
Agent: 已添加到 MEMORY.md 的 credentials 分类下。
```

添加时的一些技巧：

1. **使用关键词**：添加的内容要包含你未来可能搜索的关键词
2. **保持简洁**：一条记忆只记录一件事
3. **定期回顾**：添加后过段时间看看是否真的有用

#### /memory remove —— 删除记忆

当某些记忆过期或错误时，你需要删除它们：

```bash
# 按关键词删除（删除包含该关键词的所有条目）
User: /memory remove "测试服务器"
Agent: 已删除 2 条包含"测试服务器"的记忆：
  - "测试服务器地址: 192.168.1.50"
  - "测试服务器密码: TestPass123!"

# 精确匹配删除
User: /memory remove --exact "JWT"
Agent: 已删除 1 条精确匹配的记忆。

# 删除整个分类
User: /memory remove --category "sprint-2026-Q1"
Agent: 已删除分类 "sprint-2026-Q1" 下的 15 条记忆。
```

> **注意**
>
> `/memory remove` 删除的是温记忆（MEMORY.md / USER.md）中的条目。如果你想删除冷记忆（SQLite 中的历史记录），需要使用 `hermes history delete` 命令。两者的删除范围不同，不要混淆。

#### /memory edit —— 编辑记忆

有时候记忆的内容需要更新，但不需要完全删除重写：

```

这个命令让你在温记忆中快速找到需要的内容：

#### /memory import —— 导入记忆

与导出对应，导入让你可以恢复备份或迁移记忆：

```bash

Agent: 正在导入记忆...

✓ USER.md: 已导入 12 条记忆

Agent: ⚠ 这将覆盖所有现有记忆。确认吗？(y/N)
```

#### /memory compact —— 压缩记忆文件

当 MEMORY.md 或 USER.md 接近容量上限时，可以手动压缩：

```bash
User: /memory compact

MEMORY.md:
  压缩前: 2,180 字符
  压缩后: 1,560 字符

压缩算法会：

1. 合并语义相似的条目
2. 删除明确过期的信息
3. 缩短过长的描述
4. 移除冗余的修饰词

### 7.6 记忆清理与维护

随着时间推移，记忆中会积累一些「垃圾数据」——过时的信息、重复的内容、测试时留下的临时数据。定期清理可以保持记忆系统的高效运作。

**自动清理配置：**

```yaml
memory:
  maintenance:
    auto_cleanup: true
    cleanup_schedule: "0 2 * * 0"  # 每周日凌晨2点
    
    # 清理规则
    rules:
      # 规则1：删除30天前的临时会话
      - type: "delete_old_sessions"
        age_days: 30
        conditions:
          - "message_count < 5"  # 少于5条消息的会话
          - "NOT tagged 'important'"  # 且没有被标记为重要
      
      # 规则2：合并重复的记忆条目
      - type: "merge_duplicates"
        similarity_threshold: 0.85
      
      # 规则3：归档旧数据
      - type: "archive"
        age_days: 90
        destination: "~/.hermes/memory/archive/"
```

**手动清理：**

```bash
# 查看可以清理的内容
$ hermes memory cleanup --dry-run

预计可清理:
  - 临时会话: 23 个 (可释放 2.1 MB)
  - 重复记忆条目: 7 条 (可释放 350 字符)
  - 已删除文件的引用: 12 条
  总计可释放: ~2.5 MB

# 执行清理
$ hermes memory cleanup
正在清理...
✓ 已删除 23 个临时会话
✓ 已合并 7 条重复记忆
✓ 已清理 12 条失效引用
已释放 2.5 MB 空间

# 重建索引（如果检索变慢）
$ hermes memory rebuild-index
正在重建 FTS5 索引...
✓ 索引重建完成 (耗时 3.2s)
```

## 8. SOUL.md 深度配置

### 8.1 SOUL.md 与记忆系统的关系

SOUL.md 是 Hermes 中最特殊的配置文件。它不是传统意义上的「记忆」（它不记录历史信息），但它直接影响记忆系统的工作方式。可以把 SOUL.md 理解为 Agent 的「人格设定」和「工作守则」，它决定了 Agent 如何对待记忆、如何使用记忆、以及如何在记忆中筛选信息。

SOUL.md 与记忆系统的关系可以用一个类比来理解：

- **SOUL.md** 是公司的《员工手册》，规定了员工应该如何工作
- **MEMORY.md** 是员工手边的「项目笔记本」，记录具体项目的注意事项
- **USER.md** 是员工的「客户档案」，记录服务对象的偏好
- **SQLite** 是公司的「档案室」，存放所有历史资料

没有 SOUL.md，Agent 就不知道「我是谁」「我应该怎么工作」「什么信息值得记住」。

SOUL.md 默认位于：

```
~/.hermes/SOUL.md
```

### 8.2 高效 SOUL.md 模板大全

根据不同的使用场景，SOUL.md 的内容应该有所不同。以下提供 5 个经过社区验证的高效模板。

#### 模板一：投研助手

```markdown
# SOUL — 投研助手

## 身份
你是专业的投研分析师，擅长从海量信息中提取投资逻辑，
生成结构化的研究报告。

## 工作准则
1. 所有数据来源必须标注，不可编造数据
2. 财务指标计算必须展示公式和过程
3. 风险提示必须用单独章节呈现
4. 观点必须区分「事实」「推断」「猜测」

## 记忆策略
- 记住用户关注的行业和个股
- 记录每次研究的关键结论和置信度
- 保留重要的财务数据和估值模型
- 建立「观点追踪」：记录观点的形成时间和修正历史

## 输出格式
- 标题层级不超过 3 级
- 关键数据用表格呈现
- 图表用 Mermaid 或 ASCII 绘制
- 报告末尾附「信息来源」和「免责声明」

## 语言风格
- 专业、客观、冷静
- 避免使用「肯定」「绝对」等确定性词汇
- 多用「数据显示」「从...来看」「如果...则」

## 特殊规则
- 涉及具体股票时，必须提醒「不构成投资建议」
- 使用 MEMORY.md 记录：关注列表、估值模型参数、研究进度
- 使用 USER.md 了解用户的风险偏好和投资周期
```

### 8.3 让 Agent 自动优化 SOUL.md 的技巧

SOUL.md 不是一成不变的。随着你使用 Hermes 的时间增长，你会发现某些规则有效、某些规则需要调整。Hermes 支持让 Agent 自动优化 SOUL.md。

## 9. 隐私与安全

### 9.1 纯本地存储的安全优势

Hermes 记忆系统最核心的安全特性就是：**所有数据默认存储在本地**。

这意味着：

1. **数据不出境**：你的对话内容、工作文件、个人偏好，永远不会离开你的电脑
2. **无网络依赖**：即使没有网络，记忆系统也能正常工作
3. **无第三方访问**：没有云服务提供商能看到你的数据
4. **完全可控**：你可以随时查看、修改、删除任何数据

对比一下云端方案：

| 维度 | Hermes 本地存储 | 云端记忆服务 |
|------|---------------|------------|
| 数据位置 | 你的电脑 | 服务商的服务器 |
| 网络依赖 | 无 | 必须有网络 |
| 隐私控制 | 完全自主 | 依赖服务商政策 |
| 数据泄露风险 | 低（物理安全） | 中（依赖服务商安全） |
| 合规审计 | 容易 | 困难 |
| 跨设备同步 | 需手动 | 自动 |

对于处理敏感信息的用户（如企业内网开发、医疗数据处理、金融分析），纯本地存储几乎是必选方案。

### 9.2 数据加密选项

虽然本地存储已经很安全，但如果你有多人共用电脑、或者担心电脑被盗，可以启用数据加密。

**SQLite 数据库加密：**

SQLite 本身不内置加密，但可以通过 SQLCipher 扩展实现：

```bash
# 安装 SQLCipher
$ brew install sqlcipher  # macOS
$ apt-get install sqlcipher  # Ubuntu

# 在 Hermes 配置中启用
```yaml
memory:
  encryption:
    enabled: true
    provider: "sqlcipher"
    # 密码从环境变量读取，不会写入配置文件
    password_env: "HERMES_DB_PASSWORD"
```

# 设置环境变量
$ export HERMES_DB_PASSWORD="your-strong-password"
```

启用加密后，即使有人拿到了你的 `.db` 文件，没有密码也无法读取内容。

**温记忆文件加密：**

MEMORY.md 和 USER.md 是纯文本文件，可以用 GPG 加密：

```bash
# 加密
$ gpg --symmetric --cipher-algo AES256 ~/.hermes/memory/MEMORY.md
# 会生成 MEMORY.md.gpg

# 解密（Hermes 启动时自动执行）
$ gpg --decrypt ~/.hermes/memory/MEMORY.md.gpg > ~/.hermes/memory/MEMORY.md
```

Hermes 支持自动加解密配置：

```yaml
memory:
  encryption:
    warm_memory:
      enabled: true
      method: "gpg"
      key_id: "your-gpg-key-id"
```

### 9.3 敏感信息过滤机制

即使数据存储在本地，你仍然可能不小心在对话中透露敏感信息（如密码、API 密钥、身份证号）。Hermes 提供了敏感信息过滤机制：

**自动检测与替换：**

```yaml
security:
  sensitive_data_filter:
    enabled: true
    
    # 检测规则
    patterns:
      - name: "api_key"
        regex: "(sk-|ak-|pk-)[a-zA-Z0-9]{20,}"
        action: "hash_and_store"  # 哈希存储，不存明文
      
      - name: "password"
        regex: "(?i)(password|passwd|pwd)\s*[:=]\s*[^\s]+"
        action: "redact"  # 替换为 [REDACTED]
      
      - name: "credit_card"
        regex: "\b(?:\d{4}[-\s]?){3}\d{4}\b"
        action: "block"  # 阻止保存
      
      - name: "id_card"
        regex: "\d{17}[\dXx]|\d{15}"
        action: "block"
    
    # 自定义规则
    custom_patterns:
      - name: "internal_ip"
        regex: "10\.\d{1,3}\.\d{1,3}\.\d{1,3}"
        action: "redact"
```

**效果示例：**

```
用户输入: "我的 API Key 是 sk-abc123xyz789..."
存储到数据库前:
  → 检测到 api_key 模式
  → 替换为: "我的 API Key 是 [API_KEY_HASH:a3f7d2e...]"
  → 原始密钥不存储，只存哈希值

用户输入: "密码是 Secret123!"
存储到数据库前:
  → 检测到 password 模式
  → 替换为: "密码是 [REDACTED]"
```

> **注意**
>
> 敏感信息过滤只在存储到冷记忆（SQLite）时生效。如果你让 Agent 把敏感信息写入 MEMORY.md，它仍然会以明文形式存储在文件中（虽然文件在本地）。建议不要把真正的密码写入 MEMORY.md，而是使用环境变量或密码管理器。

### 9.4 数据备份加密方案

前面我们讲了如何备份记忆。如果备份文件也需要加密（比如你要把备份存到云盘）：

```bash
# 方法1：使用 GPG 加密备份
$ tar czf - ~/.hermes/memory/ | gpg --symmetric --cipher-algo AES256 > backup_20260421.tar.gz.gpg

# 解密恢复
$ gpg --decrypt backup_20260421.tar.gz.gpg | tar xzf -

# 方法2：使用 age（更现代的加密工具）
$ tar czf - ~/.hermes/memory/ | age -p > backup_20260421.tar.gz.age

# 方法3：使用 Hermes 内置的备份加密
$ hermes memory backup --full --encrypt --password-file ~/.hermes/.backup_pass
```

### 9.5 与云端记忆方案的隐私对比

最后，我们用一张详细的对比表来总结 Hermes 本地记忆与云端记忆的隐私差异：

| 隐私维度 | Hermes 本地 | 云端记忆服务 |
|---------|-----------|------------|
| 数据存储位置 | 用户设备 | 服务商服务器 |
| 数据传输 | 无（纯本地） | 必须通过HTTPS上传 |
| 服务商能否读取数据 | 不能 | 技术上可以（依赖信任） |
| 政府/法律调取 | 需物理获取设备 | 服务商可能被迫提供 |
| 数据删除彻底性 | 用户自行删除，可控 | 依赖服务商执行 |
| 加密密钥控制 | 用户持有 | 服务商持有或托管 |
| 审计能力 | 完全透明 | 黑盒 |
| 合规认证（SOC2/ISO27001） | 不适用 | 需查验服务商证书 |

**结论：**

- 如果你处理的是**公开信息**或**非敏感数据**，云端记忆的便利性可能更重要
- 如果你处理的是**企业机密**、**个人隐私**或**受监管数据**，Hermes 的纯本地存储是更安全的选择
- 最佳实践是：**敏感数据用本地存储，非敏感数据可选择云端扩展**

## 10. 高级配置与调优

### 10.1 记忆相关 config.yaml 参数详解

`config.yaml` 是 Hermes 的核心配置文件，其中与记忆系统相关的参数众多。以下是完整参数说明：

```yaml
# ~/.hermes/config.yaml

memory:
  # ═══════════════════════════════════════
  # 温记忆配置
  # ═══════════════════════════════════════
  
  warm:
    # MEMORY.md 的最大字符数（软限制）
    max_memory_md_chars: 2200
    
    # USER.md 的最大字符数（软限制）
    max_user_md_chars: 1375
    
    # 自动压缩阈值（超过此比例自动触发 compact）
    auto_compact_threshold: 0.85
    
    # 温记忆文件路径
    memory_md_path: "~/.hermes/memory/MEMORY.md"
    user_md_path: "~/.hermes/memory/USER.md"
    
    # 自动写入的触发条件
    auto_write:
      enabled: true
      triggers:
        - explicit_request      # 用户明确要求记录
        - task_completion       # 任务完成时
        - important_decision    # 重要决策时
        - session_end           # 会话结束时
  
  # ═══════════════════════════════════════
  # 冷记忆配置
  # ═══════════════════════════════════════
  
  cold:
    # SQLite 数据库路径
    db_path: "~/.hermes/memory/hermes_memory.db"
    
    # FTS5 配置
    fts5:
      enabled: true
      tokenizer: "porter"      # 分词器: porter / simple / icu
      
    # 检索配置
    retrieval:
      # 每次检索返回的最大片段数
      max_fragments: 5
      
      # 每个片段的最大 Token 数
      fragment_max_tokens: 150
      
      # 检索的时间范围（天）
      time_decay_days: 90
      
      # 检索时使用的字段权重
      field_weights:
        message_content: 1.0
        tool_result: 0.8
        file_operation: 0.6
        summary: 1.2
      
      # 相似度阈值（低于此值的结果丢弃）
      min_relevance_score: 0.3
    
    # 会话摘要配置
    summarization:
      enabled: true
      # 摘要模型（使用主模型或轻量模型）
      model: "same_as_main"    # 或指定如 "gpt-4o-mini"
      max_summary_length: 200
      
    # 数据保留策略
    retention:
      # 保留策略类型: forever / age_based / count_based
      policy: "age_based"
      
      # 按时间保留：超过此天数的会话归档
      max_age_days: 365
      
      # 按数量保留：最多保留多少条会话
      max_conversations: 2000
      
      # 归档路径（不删除，只移动）
      archive_path: "~/.hermes/memory/archive/"
  
  # ═══════════════════════════════════════
  # 热记忆配置
  # ═══════════════════════════════════════
  
  hot:
    # 上下文管理策略
    context_management:
      # 自动 compact 触发阈值（上下文使用比例）
      auto_compact_threshold: 0.75
      
      # 压缩时保留的最近消息数
      preserve_recent: 5
      
      # 溢出时的处理策略: compact / truncate / error
      overflow_strategy: "compact"
      
      # 系统提示和 SOUL.md 是否计入上下文限制
      count_system_in_limit: true
  
  # ═══════════════════════════════════════
  # 外部记忆配置
  # ═══════════════════════════════════════
  
  external:
    honcho:
      enabled: false
      # 详见第6章
      
    mem0:
      enabled: false
      # 详见第6章
      
    openviking:
      enabled: false
      # 详见第6章
  
  # ═══════════════════════════════════════
  # 维护配置
  # ═══════════════════════════════════════
  
  maintenance:
    # 自动维护任务
    auto_cleanup:
      enabled: true
      schedule: "0 2 * * 0"    # Cron 表达式
    
    # 备份配置
    backup:
      enabled: true
      schedule: "0 3 * * 0"
      retention: 10
      encrypt: false
      destination: "~/.hermes/backups"
    
    # 索引优化
    index_optimization:
      enabled: true
      schedule: "0 4 1 * *"    # 每月1日凌晨4点
```

### 10.2 记忆保留策略调整

根据你的使用习惯，可以调整记忆的保留策略：

**场景一：短期项目（1-3个月）**

```yaml
memory:
  cold:
    retention:
      policy: "age_based"
      max_age_days: 90        # 只保留3个月
      archive_path: "~/.hermes/memory/archive/"
  
  warm:
    auto_compact_threshold: 0.70  # 更频繁地压缩
```

**场景二：长期知识积累（1年以上）**

```yaml
memory:
  cold:
    retention:
      policy: "count_based"
      max_conversations: 5000   # 保留大量会话
      max_age_days: 730         # 最多2年
  
  warm:
    auto_compact_threshold: 0.90  # 允许温记忆更满
```

**场景三：极简模式（只保留关键信息）**

```yaml
memory:
  cold:
    retention:
      policy: "age_based"
      max_age_days: 30          # 只保留1个月
    
    summarization:
      enabled: true             # 但必须生成摘要
  
  warm:
    auto_compact_threshold: 0.60  # 频繁压缩
    max_memory_md_chars: 1500     # 更小的温记忆
    max_user_md_chars: 1000
```

### 10.3 检索精度 vs 召回率的平衡

在信息检索领域，有两个核心指标：

- **精度（Precision）**：检索到的结果中，真正相关的比例。精度高 = 垃圾结果少
- **召回率（Recall）**：所有相关结果中，被检索到的比例。召回率高 = 漏掉的少

这两个指标通常此消彼长。在 Hermes 中，你可以通过参数调整这个平衡：

**偏向精度（适合明确知道要找什么的场景）：**

```yaml
memory:
  cold:
    retrieval:
      max_fragments: 3           # 只取最相关的结果
      min_relevance_score: 0.6   # 提高门槛
      fragment_max_tokens: 100   # 更短的片段
```

**偏向召回率（适合探索性查询，宁可多不要漏）：**

```yaml
memory:
  cold:
    retrieval:
      max_fragments: 10          # 取更多结果
      min_relevance_score: 0.1   # 降低门槛
      fragment_max_tokens: 200   # 更长的片段
      time_decay_days: 180       # 更广的时间范围
```

### 10.4 大数据量下的性能优化

当数据库增长到几百 MB 甚至 GB 级别时，检索速度可能会下降。以下是优化策略：

**策略一：分区存储**

```yaml
memory:
  cold:
    # 按时间分区，每个季度的数据一个数据库文件
    partitioning:
      enabled: true
      granularity: "quarterly"   # monthly / quarterly / yearly
      
    # 检索时只查最近的分区，除非用户明确要求
    retrieval:
      default_partitions: 2      # 默认查最近2个分区
```

**策略二：索引优化**

```bash
# 手动执行索引优化
$ hermes memory optimize-index

# 这个命令会：
# 1. 重建 FTS5 索引（消除碎片）
# 2. 运行 ANALYZE 优化查询计划
# 3. 清理已删除数据占用的空间（VACUUM）
```

**策略三：读写分离**

如果你需要同时进行大量写入和查询：

```yaml
memory:
  cold:
    # 写入到主库，查询从副本
    replication:
      enabled: true
      sync_interval: 60  # 每60秒同步一次
```

**策略四：硬件加速**

```yaml
memory:
  cold:
    # 使用 SSD（如果还没用的话）
    # SQLite 在 SSD 上性能提升数倍
    
    # 调整缓存大小
    pragma:
      cache_size: 10000        # 页面数（约 40 MB）
      temp_store: "memory"     # 临时表存内存
      journal_mode: "wal"      # Write-Ahead Logging，提高并发
```

## 11. 常见问题与排障

### 11.1 记忆不生效？

**症状：** 你明明告诉 Agent 记住了某件事情，但下次会话时它似乎「忘了」。

**排查步骤：**

**步骤1：检查记忆是否真的被写入**

```bash
User: /memory show
```

看看你要记住的内容是否在 MEMORY.md 或 USER.md 中。如果没有，说明写入失败了。

**步骤2：检查写入时机**

如果你是在会话中途让 Agent 记住的，但随后会话崩溃了（比如网络断开），写入可能没有完成。建议在让 Agent 记住重要信息后，再确认一次：

```bash
User: 记住了吗？给我看看 MEMORY.md
Agent: [展示内容]
```

**步骤3：检查容量是否已满**

如果 MEMORY.md 或 USER.md 已经达到容量上限，新内容可能无法写入：

```bash
User: /memory show
# 查看使用率是否 100%
```

如果是满的，执行 `/memory compact` 或手动删除不重要的条目。

**步骤4：检查检索是否被触发**

有时候记忆确实存储了，但检索时没有匹配到。可能是因为：

- 你的查询关键词和记忆中存储的关键词不匹配
- 检索的时间范围太窄（只查最近7天，但记忆是30天前的）
- 相关度阈值太高，结果被过滤掉了

尝试扩大检索范围：

```yaml
memory:
  cold:
    retrieval:
      time_decay_days: 365    # 扩大时间范围
      min_relevance_score: 0.1  # 降低门槛
```

**步骤5：检查外部记忆冲突**

如果你同时启用了外部记忆（如 Mem0），可能存在数据不一致：

```bash
User: 当前使用哪些记忆来源？
Agent: 正在使用：本地 SQLite、Mem0 Cloud。
  本地记忆：156 条会话
  Mem0 记忆：89 条记录
  
  注意：Mem0 中的部分记忆与本地记忆有冲突（3条）。
```

如果有冲突，优先使用本地记忆，或手动同步两个来源。

### 11.2 记忆太多太杂？

**症状：** MEMORY.md 里塞满了各种信息，Agent 经常被不相关的内容干扰。

**解决方案：**

**方案1：分类管理**

把 MEMORY.md 按项目/主题分成多个文件：

```
~/.hermes/memory/
├── MEMORY.md              # 通用记忆
├── MEMORY-project-alpha.md # 项目A
├── MEMORY-project-beta.md  # 项目B
└── MEMORY-personal.md      # 个人事务
```

在对话中切换：

```bash
User: 切换到 project-alpha 的记忆
Agent: 已加载 MEMORY-project-alpha.md。
```

**方案2：定期审计**

每月让 Agent 帮你审查 MEMORY.md：

```bash
User: 审查一下 MEMORY.md，删除过时的、合并重复的
Agent:
审查结果:
  建议删除（已过时）:
    - "测试服务器地址: 192.168.1.50" (服务器已下线)
    - "项目 deadline: 2026-03-01" (已过期)
  
  建议合并（内容重复）:
    - 3 条关于 API 地址的条目 → 合并为 1 条
  
  建议保留:
    - 所有关于技术栈的信息
    - Git 仓库地址

是否执行这些修改？

User: 执行
Agent: 已优化 MEMORY.md。
  删除: 4 条
  合并: 3 条 → 1 条
  当前使用率: 68% (原为 94%)
```

**方案3：使用标签系统**

给记忆条目打标签，便于分类和筛选：

```markdown
## Project [tag:work, tag:active]
- 项目名称: Alpha
- 状态: 进行中

## Personal [tag:personal]
- 生日提醒: 妈妈的生日是 5 月 15 日

## Archived [tag:work, tag:archived]
- 旧项目 Beta 的信息...
```

### 11.3 检索结果不准确？

**症状：** 你搜索某个关键词，返回的结果完全不相关；或者明明记得某件事情，却检索不到。

**原因和解决方案：**

**原因1：分词问题（尤其是中文）**

FTS5 的默认分词器对中文支持不够好。如果你搜索「数据分析」，它可能只匹配「数」「据」「分」「析」单独出现的情况。

**解决：** 确保启用了适合中文的分词器：

```yaml
memory:
  cold:
    fts5:
      tokenizer: "icu zh_CN"   # 使用 ICU 分词器，中文支持更好
```

或者安装 `jieba` 分词器：

```bash
pip install jieba
```

```yaml
memory:
  cold:
    fts5:
      tokenizer: "jieba"
```

**原因2：关键词不匹配**

你可能搜索「Docker」，但记忆中写的是「容器化技术」。

**解决：** 使用更宽泛的搜索词，或让 Agent 帮你找：

```bash
User: 帮我找找关于容器部署的记录
Agent: 找到以下内容（包含 Docker、Kubernetes、container 等关键词）...
```

**原因3：时间范围过滤**

默认只检索最近 90 天的记录，你要找的内容可能是半年前的。

**解决：** 扩大时间范围：

```bash
User: /search "Docker" --after 2025-01-01
```

**原因4：相关度阈值过高**

如果 `min_relevance_score` 设置得太高（如 0.8），很多相关但匹配度不够高的结果会被过滤掉。

**解决：** 暂时降低阈值：

```yaml
memory:
  cold:
    retrieval:
      min_relevance_score: 0.1
```

### 11.4 数据库损坏修复

SQLite 数据库虽然稳定，但在极端情况下（如电脑突然断电、磁盘空间不足、文件系统损坏）可能会损坏。

**症状：**

- Hermes 启动时报数据库错误
- 检索功能完全失效
- `hermes status` 显示数据库异常

**修复步骤：**

**步骤1：备份当前数据库**

```bash
$ cp ~/.hermes/memory/hermes_memory.db ~/.hermes/memory/hermes_memory.db.bak
```

**步骤2：检查数据库完整性**

```bash
$ sqlite3 ~/.hermes/memory/hermes_memory.db "PRAGMA integrity_check;"
```

如果返回 `ok`，说明数据库结构完整。如果返回错误信息，继续下一步。

**步骤3：尝试修复**

```bash
# 方法1：使用 SQLite 的恢复模式
$ sqlite3 ~/.hermes/memory/hermes_memory.db ".recover" | sqlite3 ~/.hermes/memory/hermes_memory_fixed.db

# 方法2：使用 Hermes 内置修复工具
$ hermes memory repair --force
```

**步骤4：如果修复失败，从备份恢复**

```bash
# 使用最近的自动备份
$ hermes memory restore --latest

# 或手动恢复
$ cp ~/.hermes/backups/full_backup_20260414/hermes_memory.db ~/.hermes/memory/
```

**步骤5：如果备份也损坏**

最坏的情况下，你可以重建数据库：

```bash
$ hermes memory reset
# 这会删除所有冷记忆，但保留温记忆（MEMORY.md 和 USER.md）
```

> **注意**
>
> 数据库损坏的情况非常罕见（SQLite 被设计为在断电时也能保持数据完整）。真正发生损坏时，通常是磁盘硬件问题。建议定期检查磁盘健康状态，并保持定期备份的习惯。

### 11.5 迁移时记忆丢失？

**症状：** 你把 Hermes 从一台电脑迁移到另一台电脑，发现记忆「丢失」了。

**原因：** 记忆文件没有正确迁移，或者路径配置不一致。

**正确的迁移步骤：**

**步骤1：在旧机器上导出完整备份**

```bash
$ hermes memory backup --full
备份已保存到: ~/.hermes/backups/full_backup_20260421.tar.gz
```

**步骤2：复制配置文件和记忆目录**

```bash
# 需要复制的文件和目录
~/.hermes/config.yaml
~/.hermes/SOUL.md
~/.hermes/memory/           # 整个记忆目录
~/.hermes/skills/           # 如果有自定义技能
```

**步骤3：在新机器上安装 Hermes**

```bash
# 按照安装指南安装 Hermes
# 但不要启动，等配置文件复制完成后再启动
```

**步骤4：复制文件到新机器**

```bash
# 使用 scp
$ scp -r ~/.hermes/ user@new-machine:~/

# 或使用云存储同步
# 将 ~/.hermes/ 放入 Dropbox / iCloud / OneDrive 同步
```

**步骤5：验证路径配置**

```yaml
# 检查 config.yaml 中的路径是否匹配新机器
memory:
  cold:
    db_path: "~/.hermes/memory/hermes_memory.db"  # 使用 ~ 会自动解析为当前用户目录
```

如果使用绝对路径，需要确保新机器上的路径一致。

**步骤6：启动 Hermes 并验证**

```bash
$ hermes status --memory
# 检查会话数、消息数是否与旧机器一致

$ /memory show
# 检查 MEMORY.md 和 USER.md 内容是否完整

$ /search "测试"
# 检查冷记忆检索是否正常
```

**常见问题：**

- **权限问题**：确保新机器上的文件权限正确（`chmod 600` 对于敏感文件）
- **路径问题**：Windows 和 macOS/Linux 的路径格式不同，注意转换
- **版本兼容**：如果新旧机器的 Hermes 版本差异很大，可能需要升级数据库格式：`hermes memory migrate`
