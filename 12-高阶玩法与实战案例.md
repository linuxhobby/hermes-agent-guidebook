这就是高阶玩家的 Hermes —— 不是你在"用"它，而是它在"为你工作"。

## 2. 沙箱配置与安全调优

### 2.1 执行环境的安全分级

在让 Hermes 执行系统命令或运行代码之前，安全是你必须考虑的第一件事。想象一下，如果你的 Agent 不小心执行了 `rm -rf /`，或者运行了一个包含恶意代码的脚本，后果会怎样？

Hermes 的沙箱系统就是为了解决这个问题而设计的。它把代码执行环境分为四个安全等级，就像一个军营有不同的安全区域：

| 安全等级 | 名称 | 权限范围 | 适用场景 |
|----------|------|----------|----------|
| Level 0 | 完全隔离 | 只读文件系统，禁止网络访问 | 运行完全不可信的代码 |
| Level 1 | 受限执行 | 可读写临时目录，限制网络访问 | 运行来源不明的脚本 |
| Level 2 | 标准沙箱 | 可读写工作目录，允许外部网络 | 日常任务执行 |
| Level 3 | 信任模式 | 完整系统访问，无限制 | 自己编写的可信代码 |

> **类比**
>
> 安全等级就像机场的安检级别。Level 0 是最高级别——你要脱鞋、解皮带、全身扫描，什么东西都不能带。Level 3 就是 VIP 通道——快速通过，基本不检查。不同的航班（任务）需要不同的安检级别。

### 2.2 沙箱（Sandbox）配置详解

Hermes 的沙箱基于 Docker 容器和系统级隔离技术实现。下面是详细的配置方法。

**基础配置（在 `config.yaml` 中）**

```yaml
# ============================================
# 沙箱安全配置文件 - config.yaml
# ============================================

sandbox:
  # 默认安全等级：所有未明确指定的任务使用此等级
  default_level: 2
  
  # Docker 镜像配置
  docker:
    # 基础镜像，包含 Python、Node.js、常用工具
    image: "hermes/sandbox:latest"
    # 是否自动拉取最新镜像
    auto_pull: true
    # 容器资源限制
    resources:
      # CPU 核心数限制
      cpus: 2.0
      # 内存限制（MB）
      memory: 1024
      # 临时存储限制（MB）
      disk: 512
  
  # 网络访问控制
  network:
    # 是否允许容器访问外部网络
    enabled: true
    # 允许访问的域名白名单（为空表示允许所有）
    allowed_hosts:
      - "api.openai.com"
      - "api.anthropic.com"
      - "github.com"
      - "registry.npmjs.org"
    # 禁止访问的域名黑名单
    blocked_hosts:
      - "localhost"
      - "127.0.0.1"
      - "0.0.0.0"
  
  # 文件系统权限
  filesystem:
    # 只读目录（容器内无法修改）
    read_only:
      - "/app/templates"
      - "/app/config"
    # 可读写目录
    read_write:
      - "/app/workspace"
      - "/tmp"
    # 禁止访问的路径
    forbidden:
      - "/etc/passwd"
      - "/etc/shadow"
      - "/root/.ssh"
      - "~/.aws"
  
  # 命令执行超时（秒）
  timeout: 300
  
  # 环境变量隔离
  env_isolation: true
```

**按任务类型设置安全等级**

你可以在 Skill 或工作流配置中为特定任务指定安全等级：

```yaml
# skills/web_search.yaml
name: "web_search"
description: "执行网络搜索"

# 此技能只需要访问外部网络，不需要本地文件操作
sandbox:
  level: 2
  network:
    enabled: true
  filesystem:
    read_write:
      - "/tmp/search_results"

---

# skills/code_execution.yaml
name: "code_execution"
description: "执行用户提供的代码"

# 用户代码不可信，使用严格隔离
sandbox:
  level: 0
  docker:
    resources:
      cpus: 1.0
      memory: 512
      disk: 256
  timeout: 60
```

> **Tips**
>
> 一个实用的经验法则：**凡是涉及用户输入的代码执行，都至少用 Level 1；完全不可信的来源用 Level 0；你自己写的自动化脚本可以用 Level 3。** 宁可在安全上过度配置，也不要为了方便留下隐患。

### 2.3 命令审批机制的自定义

Hermes 有一个非常重要的安全功能：**命令审批（Command Approval）**。当 Agent 要执行某些敏感操作时，会先暂停并征求你的确认。

**配置审批规则**

```yaml
# config.yaml 中的审批配置
approval:
  # 全局开关
  enabled: true
  
  # 审批模式：manual（手动）/ auto（自动）/ conditional（条件）
  mode: "conditional"
  
  # 条件审批规则
  rules:
    # 规则1：文件删除操作必须审批
    - name: "file_deletion"
      pattern: "rm\\s+-[rf]*\\s+.*"
      action: "require_approval"
      message: "⚠️ 检测到文件删除命令，请确认是否执行：{command}"
    
    # 规则2：系统级命令必须审批
    - name: "system_commands"
      pattern: "(sudo|chmod|chown|mkfs|fdisk)\\s+.*"
      action: "require_approval"
      message: "⚠️ 检测到系统级命令，需要管理员权限，请确认：{command}"
    
    # 规则3：网络请求自动通过（在允许域名内）
    - name: "network_requests"
      pattern: "(curl|wget|fetch)\\s+.*"
      action: "auto_approve"
      condition: "host_in_whitelist"
    
    # 规则4：数据库操作必须审批
    - name: "database_ops"
      pattern: "(DROP|DELETE|TRUNCATE|UPDATE)\\s+.*"
      action: "require_approval"
      message: "⚠️ 检测到数据修改命令，请确认：{command}"
    
    # 规则5：大规模文件操作提醒
    - name: "bulk_operations"
      pattern: "find\\s+.*-delete|rm\\s+-rf\\s+/.*"
      action: "block"
      message: "🚫 危险命令已拦截：{command}"
```

**审批通知方式**

```yaml
approval:
  # 通知渠道配置
  notifications:
    # 终端交互式审批（默认）
    - channel: "terminal"
      enabled: true
      timeout: 60  # 等待用户输入的超时时间（秒）
    
    # Telegram 审批（适合远程/无人值守场景）
    - channel: "telegram"
      enabled: true
      bot_token: "${TELEGRAM_BOT_TOKEN}"
      chat_id: "${TELEGRAM_CHAT_ID}"
      timeout: 300  # 给用户更多时间响应
    
    # 邮件审批（适合企业环境）
    - channel: "email"
      enabled: false
      smtp_server: "smtp.gmail.com"
      to: "admin@company.com"
      timeout: 3600
```

> **注意**
>
> 如果你把 Hermes 部署在服务器上并启用了 Web 界面或第三方平台接入，务必配置 Telegram 或邮件审批。否则当 Agent 需要审批时，你会在终端看不到提示，任务就会一直卡住。

### 2.4 信任级别设置

Hermes 允许你为不同的 Skill、工具或工作流设置信任级别，实现更细粒度的安全控制。

```yaml
# trust_levels.yaml
# 信任级别配置文件

trust_levels:
  # 内置工具和官方 Skill —— 完全信任
  builtin:
    sources:
      - "hermes.builtin.*"
      - "hermes.official.*"
    permissions:
      sandbox_level: 3
      auto_approve: true
      network_access: true
  
  # 社区贡献的 Skill —— 有限信任
  community:
    sources:
      - "hermes.community.*"
    permissions:
      sandbox_level: 1
      auto_approve: false
      network_access: true
      filesystem:
        read_only:
          - "/app/data"
        read_write:
          - "/tmp"
  
  # 用户自己编写的 Skill —— 根据用户选择
  user:
    sources:
      - "user.*"
    permissions:
      sandbox_level: 2  # 默认标准沙箱
      auto_approve: false
      network_access: true
  
  # 从未知来源安装的 Skill —— 最高警惕
  untrusted:
    sources:
      - "*"
    permissions:
      sandbox_level: 0
      auto_approve: false
      network_access: false
      filesystem:
        read_only: []
        read_write:
          - "/tmp/isolated"
```

### 2.6 案例：安全地运行未知代码

假设你收到了一段别人发来的 Python 脚本，说是可以帮你"自动整理桌面文件"。你想让 Hermes 运行它，但又担心有安全隐患。这时候应该怎么做？

**步骤1：创建隔离沙箱配置**

```yaml
# isolated_run.yaml
name: "safe_execution"
description: "安全执行未知来源代码"

sandbox:
  level: 0  # 最高隔离级别
  docker:
    image: "hermes/sandbox:minimal"
    resources:
      cpus: 0.5
      memory: 256
      disk: 100
  network:
    enabled: false  # 完全断网
  filesystem:
    read_only:
      - "/app/code"  # 只读挂载待执行的代码
    read_write:
      - "/tmp/output"  # 只允许写到临时目录
  timeout: 30  # 30秒超时
```

**步骤2：使用 Hermes CLI 安全执行**

```bash
# 将未知代码放入隔离目录
mkdir -p ./isolated_run
cp unknown_script.py ./isolated_run/

# 使用 Hermes 的隔离执行命令
$ hermes exec --sandbox isolated_run.yaml --file ./isolated_run/unknown_script.py

# 查看执行结果（只能从指定输出目录读取）
cat ./isolated_run/output/result.json
```

**步骤3：分析代码行为（不执行）**

如果你连隔离执行都不放心，可以让 Hermes 先静态分析代码：

```bash
# 让 Hermes 分析代码的功能和潜在风险
$ hermes analyze --file unknown_script.py --depth detailed

# 预期输出示例：
# [ANALYSIS] 检测到以下行为：
# - 文件读取操作：读取 ~/Desktop 目录
# - 文件写入操作：在 ~/Desktop 创建整理后的文件夹
# - 网络操作：无
# - 系统调用：无
# - 风险评级：低（仅操作桌面文件，无敏感路径访问）
```

> **Tips**
>
> 对于任何来源不明的代码，养成"先分析、再隔离执行、最后才正常执行"的习惯。Hermes 的 `analyze` 命令会检查代码中的文件操作、网络请求、系统调用等行为，给你一个风险评级。这个习惯可能某天真会救你一命。

## 3. 语音交互高阶玩法

### 3.1 语音输入（STT）高级配置

语音输入（Speech-to-Text，简称 STT）让你可以用说话代替打字，跟 Hermes 交互。这对于双手忙碌的场景（开车、做饭、运动）特别有用。

Hermes 支持多种 STT 引擎，从完全免费的本地模型到高质量的云端 API。

**支持的 STT 引擎**

| 引擎 | 类型 | 质量 | 延迟 | 成本 | 适用场景 |
|------|------|------|------|------|----------|
| Whisper (本地) | 本地 | 中高 | 中等 | 免费 | 日常对话、隐私敏感 |
| Whisper API | 云端 | 高 | 低 | $0.006/分钟 | 高质量要求 |
| DeepSpeech | 本地 | 中 | 低 | 免费 | 资源受限设备 |
| Google STT | 云端 | 高 | 低 | $0.024/分钟 | 多语言支持 |
| Azure STT | 云端 | 高 | 低 | $1.0/小时 | 企业环境 |

**基础配置**

```yaml
# config.yaml 中的 STT 配置
stt:
  # 默认引擎
  engine: "whisper_local"
  
  # 本地 Whisper 配置
  whisper_local:
    # 模型大小：tiny/base/small/medium/large
    # 越大越准，但也越慢、越吃资源
    model: "base"
    # 语言设置：auto 自动检测，或指定如 "zh", "en"
    language: "auto"
    # 设备：cpu / cuda（需要 NVIDIA 显卡）
    device: "cpu"
    # 是否翻译为英文（Whisper 特有功能）
    translate: false
  
  # Whisper API 配置
  whisper_api:
    api_key: "${OPENAI_API_KEY}"
    model: "whisper-1"
    language: "auto"
  
  # Google STT 配置
  google_stt:
    credentials_path: "${GOOGLE_CREDENTIALS_PATH}"
    language_code: "zh-CN"
    # 是否启用自动标点
    enable_automatic_punctuation: true
  
  # 音频输入设置
  audio:
    # 采样率
    sample_rate: 16000
    # 录音时长限制（秒）
    max_duration: 60
    # 静音检测阈值（用于自动停止录音）
    silence_threshold: 0.02
    # 静音持续多久自动停止（秒）
    silence_duration: 2.0
```

**高级配置：语音活动检测（VAD）**

VAD（Voice Activity Detection）让 Hermes 能够自动检测你什么时候开始说话、什么时候说完，不需要手动按"录音"按钮。

```yaml
stt:
  # VAD 配置
  vad:
    enabled: true
    # VAD 引擎：webrtc / silero / energy
    engine: "silero"
    
    silero:
      # 模型路径（自动下载）
      model: "silero_vad"
      # 检测阈值（0-1，越高越严格）
      threshold: 0.5
      # 最小语音长度（毫秒）
      min_speech_duration_ms: 250
      # 最大语音长度（毫秒）
      max_speech_duration_s: 60
      # 语音前预留的静音（毫秒）
      speech_pad_ms: 100
```

**唤醒词配置**

你可以设置一个唤醒词（Wake Word），就像"Hey Siri"或"OK Google"一样。只有检测到唤醒词后，Hermes 才会开始录音和识别。

```yaml
stt:
  wake_word:
    enabled: true
    # 唤醒词
    phrase: "嘿 Hermes"
    # 唤醒词引擎：porcupine / snowboy / custom
    engine: "porcupine"
    # 灵敏度（0-1）
    sensitivity: 0.7
    # 唤醒后的监听超时（秒）
    listen_timeout: 10
```

> **类比**
>
> 唤醒词就像是办公室的敲门声。平时 Hermes 在"专注做自己的事"（不监听），只有听到"敲门声"（唤醒词），它才会抬头听你说话。这样既省电（不处理无意义的背景音），又保护隐私（不会误录你的私人对话）。

### 3.2 语音输出（TTS）高级配置

语音输出（Text-to-Speech，简称 TTS）让 Hermes 可以用说话的方式回复你，而不是显示文字。

**支持的 TTS 引擎**

| 引擎 | 类型 | 质量 | 延迟 | 成本 | 特点 |
|------|------|------|------|------|------|
| Piper | 本地 | 中高 | 低 | 免费 | 开源、轻量、支持多种语言 |
| Coqui TTS | 本地 | 高 | 中 | 免费 | 质量优秀，支持语音克隆 |
| Edge TTS | 云端(免费) | 高 | 低 | 免费 | 微软 Edge 浏览器同款 |
| OpenAI TTS | 云端 | 很高 | 中 | $0.015/1K字符 | 音质极佳，多种声音 |
| ElevenLabs | 云端 | 顶级 | 中 | $0.18/1K字符 | 最自然的语音 |

**基础配置**

```yaml
# config.yaml 中的 TTS 配置
tts:
  # 默认引擎
  engine: "edge_tts"
  
  # Edge TTS 配置（免费）
  edge_tts:
    # 语音选择
    # 中文女声：zh-CN-XiaoxiaoNeural
    # 中文男声：zh-CN-YunjianNeural
    # 英文女声：en-US-JennyNeural
    voice: "zh-CN-XiaoxiaoNeural"
    # 语速：0.5-2.0
    rate: "+0%"
    # 音量
    volume: "+0%"
  
  # OpenAI TTS 配置
  openai_tts:
    api_key: "${OPENAI_API_KEY}"
    model: "tts-1"  # tts-1（快）或 tts-1-hd（高质）
    voice: "alloy"  # alloy/echo/fable/onyx/nova/shimmer
    speed: 1.0
  
  # Piper 本地 TTS 配置
  piper:
    model: "zh_CN-huayan-medium"
    # 语速
    length_scale: 1.0
    # 音调
    noise_scale: 0.667
  
  # 输出设置
  output:
    # 播放设备：auto / 设备名称
    device: "auto"
    # 是否自动播放
    auto_play: true
    # 缓存语音文件（避免重复生成）
    cache_enabled: true
    cache_dir: "./cache/tts"
```

**流式 TTS 配置（降低延迟）**

传统的 TTS 是"生成完整文本的语音，然后再播放"，延迟较高。流式 TTS 可以在文本生成的过程中就开始合成和播放，实现"边想边说"的效果。

```yaml
tts:
  # 流式输出配置
  streaming:
    enabled: true
    # 句子分隔符（遇到这些符号就切分并合成）
    sentence_delimiters: ".。!！?？;；\n"
    # 最小句子长度（字符数）
    min_sentence_length: 10
    # 最大缓冲区（句子数）
    max_buffer_size: 3
    # 是否启用打断（用户说话时停止播放）
    allow_barge_in: true
```

> **Tips**
>
> 如果你追求"极致自然"的语音体验，推荐 OpenAI TTS 的 `tts-1-hd` 模型配合 `shimmer` 声音。如果你追求"免费且好用"，Edge TTS 是最好的选择——它和微软 Edge 浏览器的"大声朗读"功能用的是同一个引擎，质量相当高。如果你需要完全离线运行（比如车载场景），Piper 或 Coqui TTS 是首选。

### 3.3 多语言语音支持

Hermes 的语音系统支持多种语言的输入和输出，你可以根据场景灵活切换。

```yaml
# 多语言语音配置
stt:
  # 默认语言
  default_language: "zh"
  
  # 语言切换命令（说出这些词可以临时切换语言）
  language_switch:

      "说中文": "zh"
      "说英语": "en"
      "说日语": "ja"
    # 切换后的持续时间（秒），0 表示永久切换
    duration: 0
  
  # 自动语言检测
  auto_detect:
    enabled: true
    # 可能检测到的语言列表
    languages:
      - "zh"
      - "en"
      - "ja"
      - "ko"

tts:
  # 按语言配置不同的 TTS 引擎和声音
  language_mapping:
    zh:
      engine: "edge_tts"
      voice: "zh-CN-XiaoxiaoNeural"
    en:
      engine: "edge_tts"
      voice: "en-US-JennyNeural"
    ja:
      engine: "edge_tts"
      voice: "ja-JP-NanamiNeural"
    # 自动检测语言后，使用对应的声音
    auto_switch: true
```

## 4. 实战案例：AI 投研流水线

### 4.1 需求分析：每日自动追踪 AI/Web3 热点

作为一个关注 AI 和 Web3 领域的投资者或研究人员，你每天早上都需要花 1-2 小时浏览各种信息源——Hacker News、GitHub Trending、Twitter、Reddit、技术博客……然后手动整理成一份可读性强的报告。

这个过程有几个痛点：

1. **信息过载**：每天产生的内容太多，你不可能全部看完
2. **信息噪声**：很多内容是重复、低质或不相关的
3. **整理耗时**：看完信息后，还需要花时间提炼要点、格式化
4. **容易遗漏**：手动浏览总有遗漏，可能错过重要信号
5. **缺乏关联**：单独看每条信息看不出趋势，需要跨源关联分析

**解决方案**：让 Hermes 搭建一条"投研流水线"，每天自动完成信息收集、清洗、分析、报告生成和推送的全过程。

**预期效果**：
- 每天早上 8:00，你的 Telegram/飞书收到一份 curated 好的行业日报
- 报告包含：热点摘要、趋势分析、关键数据、值得关注的项目
- 每日运行成本约 $0.1-0.5（主要是 API 调用费用）
- 你自己只需要花 5 分钟浏览报告，而不是 2 小时手动整理

### 4.2 整体架构设计（MCP + Cron + 多Agent）

这个系统采用多 Agent 协作架构，每个 Agent 负责一个环节，像流水线一样串联工作。

```
┌─────────────────────────────────────────────────────────────┐
│                    AI 投研流水线架构                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Cron 触发（每天 8:00）                                     │
│         │                                                   │
│         ▼                                                   │
│   ┌──────────────┐                                         │
│   │  数据收集 Agent │ ◄─── 多源并行抓取                       │
│   └──────┬───────┘                                         │
│          │ 原始数据                                         │
│          ▼                                                  │
│   ┌──────────────┐                                         │
│   │  数据清洗 Agent │ ──── 去重、过滤、格式化                  │
│   └──────┬───────┘                                         │
│          │ 清洗后数据                                       │
│          ▼                                                  │
│   ┌──────────────┐                                         │
│   │  分析 Agent   │ ──── 趋势提取、情感分析、关联挖掘          │
│   └──────┬───────┘                                         │
│          │ 分析结果                                         │
│          ▼                                                  │
│   ┌──────────────┐                                         │
│   │  报告生成 Agent│ ──── 格式化、图表、摘要                   │
│   └──────┬───────┘                                         │
│          │ 最终报告                                         │
│          ▼                                                  │
│   ┌──────────────┐                                         │
│   │  推送 Agent   │ ──── Telegram/飞书/邮件                  │
│   └──────────────┘                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**为什么用多 Agent 而不是单个 Agent？**

1. **关注点分离**：每个 Agent 只做好一件事，逻辑清晰，便于调试
2. **并行优化**：数据收集阶段可以同时抓取多个源
3. **成本优化**：不同环节可以用不同模型（收集用便宜模型，分析用强模型）
4. **容错隔离**：一个环节出问题不影响其他环节
5. **可替换性**：想换一个数据源？只改收集 Agent 就行

> **类比**
>
> 多 Agent 流水线就像一家报社的编辑部。有记者（收集 Agent）去各地采访，有编辑（清洗 Agent）整理稿件，有评论员（分析 Agent）写深度分析，有排版员（报告生成 Agent）做成报纸，有发行员（推送 Agent）送到读者手中。每个人专精一项，效率最高。

### 4.3 数据源配置（Hacker News、GitHub Trending、Twitter/X、Reddit）

**Hacker News 数据抓取**

```yaml
# data_sources/hacker_news.yaml
name: "hacker_news_source"
description: "从 Hacker News 抓取热门内容"

# 使用 Hermes 内置的 Web 搜索工具 + HN API
```

## 5. 实战案例：内容自动化工厂

### 5.1 需求分析：自动生成社交媒体内容

自媒体运营有一个永恒的痛点：**内容生产速度追不上发布频率的要求**。

假设你运营着 3 个平台：微信公众号、小红书和抖音。每个平台的风格不同：
- 公众号：深度长文，1500-3000 字，注重逻辑和论据
- 小红书：短平快，300-800 字，多用 emoji，注重视觉感
- 抖音：文案要口语化，适合配音，强调"钩子"和情绪

如果你每天要为每个平台生产内容，光是写作就要花 3-4 小时，还没算选题和排版的时间。

**解决方案**：搭建一个"内容自动化工厂"，让 Hermes 的多个 Agent 协作完成从选题到发布的全流程。

**预期效果**：
- 每天早上自动生成 5 篇 AI 快讯（不同平台版本）
- 选题基于实时热点，确保内容的时效性
- 每篇内容都经过审校 Agent 检查，避免事实错误
- 发布时间优化（根据各平台流量高峰）
- 你只需要花 10 分钟做最终确认和微调

### 5.2 选题 Agent（热点追踪）

选题是内容生产的第一步，也是最重要的一步。选题 Agent 的任务是：从海量信息中筛选出适合你账号定位的热点话题。

```yaml
# agents/topic_selector.yaml
name: "topic_selector"
description: "从热点信息中筛选适合的内容选题"

# 账号定位（用于筛选匹配的话题）
account_profile:
  name: "佚名"
  niche: "AI 技术解读与行业观察"
  target_audience: "AI 从业者、开发者、投资人"
  content_style: "深度分析 + 通俗解释"
  # 偏好话题权重
  topic_weights:
    "大模型技术": 1.5
    "AI Agent": 1.5
    "开源项目": 1.3
    "创业融资": 1.2
    "产品发布": 1.0
    "行业八卦": 0.3  # 低权重，不太关注八卦

# 热点数据源
sources:
  - "hacker_news"
  - "github_trending"
  - "weibo_hot"       # 微博热搜
  - "zhihu_hot"       # 知乎热榜
  - "twitter_trending"

# 选题策略
selection_strategy:
  # 策略1：时效性优先（新发生的事）
  - name: "timeliness"
    weight: 0.3
    criteria: "published_within_24h"
  
  # 策略2：热度优先（讨论的人多）
  - name: "popularity"
    weight: 0.25
    criteria: "engagement_score > threshold"
  
  # 策略3：匹配度优先（符合账号定位）
  - name: "relevance"
    weight: 0.3
    criteria: "semantic_similarity_to_profile > 0.7"
  
  # 策略4：差异化优先（别人没讲过的角度）
  - name: "differentiation"
    weight: 0.15
    criteria: "novelty_score > 0.6"

# 每日选题数量
daily_topics: 5

# 输出格式
output:
  - "title"           # 选题标题
  - "angle"           # 切入角度
  - "source_url"      # 参考链接
  - "priority"        # 优先级（1-5）
  - "estimated_reads" # 预估阅读量
```

**选题 Agent 的工作流程**

```yaml
workflow:
  name: "daily_topic_selection"
  steps:
    # 步骤1：抓取热点
    - action: "fetch_trending"
      sources: "{{sources}}"
      
    # 步骤2：初步过滤（排除明显不相关的）
    - action: "filter"
      rules:
        - "language in ['zh', 'en']"
        - "not contains: [色情, 赌博, 政治]"
    
    # 步骤3：计算匹配度
    - action: "score_topics"
      strategy: "{{selection_strategy}}"
      
    # 步骤4：去重（避免连续多天讲同一话题）
    - action: "dedup_with_history"
      history_days: 7
      threshold: 0.8
      
    # 步骤5：生成选题列表
    - action: "generate_topic_list"
      count: "{{daily_topics}}"
```

> **类比**
>
> 选题 Agent 就像一个敏锐的编辑部主任。每天早上，他浏览上百份报纸和网站，从中挑出最值得报道的 5 个话题。他不是随便挑——他要考虑：这个话题我们的读者关心吗？我们能讲出跟别人不一样的角度吗？这个话题现在讲是不是最好的时机？

**批量生成工作流**


### 5.4 审校 Agent（质量把控）

内容生成后，不能直接发布，必须经过审校。审校 Agent 负责检查事实准确性、语言表达、合规性等问题。

```yaml
# agents/editor.yaml
name: "content_editor"
description: "审校生成的内容，确保质量"

# 审校维度
check_dimensions:
  # 维度1：事实核查
  - name: "fact_check"
    model: "gpt-4o"
    prompt: |
      请检查以下内容中的事实性陈述，标记可能存在的问题：
      
      内容：{{content}}
      
      检查要点：
      1. 数据是否准确（是否有明显错误的数据）
      2. 人名/公司名是否拼写正确
      3. 时间/日期是否合理
      4. 引用是否标注来源
      5. 是否存在明显的主观臆断被包装成事实
      
      输出格式：
      - 问题位置（段落/句子）
      - 问题描述
      - 建议修改
    severity: "high"  # 事实错误必须修正
  
  # 维度2：语言表达
  - name: "language_quality"
    model: "gpt-4o-mini"
    prompt: |
      请检查以下内容的语言表达：
      
      内容：{{content}}
      平台：{{platform}}
      
      检查要点：
      1. 是否有语病或不通顺的句子
      2. 是否符合 {{platform}} 的语言风格
      3. 是否有重复啰嗦的表述
      4. 标点符号使用是否规范
      5. 段落划分是否合理
    severity: "medium"
  
  # 维度3：合规性检查
  - name: "compliance"
    model: "gpt-4o-mini"
    prompt: |
      请检查以下内容是否存在合规风险：
      
      内容：{{content}}
      
      检查要点：
      1. 是否涉及敏感话题（政治、色情、暴力等）
      2. 是否存在未经授权的引用/抄袭
      3. 是否涉及不实医疗/投资建议
      4. 是否侵犯他人隐私或名誉
      5. 广告法相关：是否使用了违禁词（"最"、"第一"、"国家级"等）
    severity: "critical"  # 合规问题一票否决
  
  # 维度4：平台适配检查
  - name: "platform_fit"
    model: "gpt-4o-mini"
    prompt: |
      请检查以下内容是否符合 {{platform}} 的平台规范：
      
      内容：{{content}}
      
      检查要点：
      1. 字数是否在合理范围内
      2. 格式是否符合平台习惯
      3. 是否有平台禁止的内容类型
      4. 标题是否符合平台算法偏好
    severity: "medium"

# 审校结果处理
result_handling:
  # 如果没有问题，直接通过
  auto_pass: "no_issues_found"
  
  # 如果有中低风险问题，自动修正
  auto_fix: "severity <= medium"
  
  # 如果有高风险问题，标记为需人工审核
  manual_review: "severity >= high"
  
  # 如果有严重合规问题，直接拒绝
  reject: "severity == critical"
```

## 6. 实战案例：代码质量守护者

### 6.1 需求分析：自动化代码审查与安全扫描

代码审查是软件开发中至关重要但也非常耗时的环节。一个典型的代码审查可能需要：
- 30 分钟理解代码变更的背景和目的
- 20 分钟检查代码风格和命名规范
- 20 分钟分析潜在的安全漏洞
- 15 分钟检查测试覆盖度
- 15 分钟给出改进建议

总计：每次 PR 审查约 100 分钟。如果团队每天产生 5-10 个 PR，光是代码审查就要占用一个人一整天的时间。

**解决方案**：让 Hermes 担任"代码质量守护者"，自动完成大部分机械性的审查工作，把人类审查者的精力解放出来，专注于架构设计和业务逻辑。

**预期效果**：
- 每个 PR 在提交后 1 分钟内得到初步审查反馈
- 自动拦截明显的安全漏洞和代码风格问题
- 生成结构化的审查报告，便于人类审查者快速浏览
- 与 CI/CD 管道集成，不达标的代码无法合并

### 6.2 Git Hook 集成

Git Hook 是在 Git 操作的特定时机自动执行的脚本。我们可以通过 Hook 在代码提交或推送时触发 Hermes 的审查。

**pre-commit Hook（提交前检查）**

```bash
#!/bin/bash
# .git/hooks/pre-commit
# 在每次 git commit 前执行

echo "🔍 Hermes 正在检查你的代码..."

# 获取暂存区的文件列表
STAGED_FILES=$(git diff --cached --name-only --diff-filter=ACM | grep '\.(py|js|ts|java|go)$')

if [ -z "$STAGED_FILES" ]; then
    echo "✅ 没有需要检查的代码文件"
    exit 0
fi

# 调用 Hermes 进行提交前检查
hermes review \
    --mode pre-commit \
    --files "$STAGED_FILES" \
    --config ./hermes-code-review.yaml \
    --output /tmp/hermes-pre-commit.json

# 读取检查结果
RESULT=$(cat /tmp/hermes-pre-commit.json | jq '.passed')

if [ "$RESULT" = "false" ]; then
    echo "❌ 代码检查未通过，请修复以下问题："
    cat /tmp/hermes-pre-commit.json | jq '.issues[] | "  - \(.severity): \(.message) (\(.file):\(.line))"'
    echo ""
    echo "如果你确定要提交，使用 git commit --no-verify 跳过检查（不推荐）"
```

## 7. 实战案例：个人知识管理系统

### 7.1 需求分析：打造第二大脑

现代人的信息摄入量是巨大的。每天你要浏览几十篇文章、看几十个视频、读几十条社交媒体动态、收到几十封邮件。但问题是：**你真正记住并能够调用的信息，可能不到 1%。**

传统的笔记方法（随手记、收藏夹、文件夹）有几个问题：
1. **存而不读**：收藏了 1000 篇文章，但再也没看过
2. **找而不得**：需要某个信息时，想不起来存在哪里
3. **信息孤岛**：笔记之间没有关联，无法形成知识网络
4. **缺乏回顾**：没有定期回顾机制，知识随着时间遗忘

**解决方案**：用 Hermes 搭建一个"第二大脑"——自动收集、智能分类、构建知识图谱、定期回顾，让你的知识资产真正产生复利。

**预期效果**：
- 自动收集你浏览过的文章、邮件、书签
- 智能分类和标签化，无需手动整理
- 知识图谱可视化，发现信息之间的隐藏关联
- 定期生成回顾报告，强化记忆
- 需要时可以通过自然语言查询快速找到相关信息

## 8. 实战案例：客服机器人

### 8.1 需求分析：7x24 智能客服

对于电商、SaaS、在线教育等业务，客服是一个刚需但成本很高的环节：
- 人工客服需要轮班，7×24 覆盖成本高
- 80% 的问题都是重复性的（"怎么退款"、"密码忘了怎么办"）
- 高峰期（双 11、活动期间）客服压力巨大
- 培训新客服成本高，且服务质量不稳定

**解决方案**：用 Hermes 搭建一个智能客服机器人，处理常见问题，复杂问题再转人工。

**预期效果**：
- 解决 70-80% 的常见问题，无需人工介入
- 7×24 在线，响应时间 < 3 秒
- 多语言支持（如果业务有出海需求）
- 满意度收集和分析，持续优化
- 大幅降低客服成本

### 10.2 数据库查询助手（接入公司 DB）

让非技术人员也能通过自然语言查询数据库，而不需要写 SQL。

```yaml
# enterprise/db_assistant.yaml
name: "database_query_assistant"
description: "自然语言数据库查询助手"

# 数据库连接
databases:
  - name: "production"
    type: "postgresql"
    host: "${DB_HOST}"
    port: 5432
    database: "analytics"
    username: "${DB_USER}"
    password: "${DB_PASSWORD}"
    # 只读权限
    read_only: true

# Schema 描述（帮助 AI 理解表结构）
schema_description:
  tables:
    - name: "orders"
      description: "订单表，存储所有交易订单"
      columns:
        - name: "id"
          description: "订单 ID"
        - name: "user_id"
          description: "用户 ID"
        - name: "total_amount"
          description: "订单总金额"
        - name: "status"
          description: "订单状态：pending, paid, shipped, completed"
        - name: "created_at"
          description: "创建时间"
    
    - name: "users"
      description: "用户表"
      columns:
        - name: "id"
          description: "用户 ID"
        - name: "email"
          description: "邮箱"
        - name: "registration_date"
          description: "注册日期"

# 安全限制
security:
  # 只允许 SELECT
  allowed_operations: ["SELECT"]
  
  # 禁止查询敏感表
  forbidden_tables:
    - "passwords"
    - "api_keys"
    - "salary_info"
  
  # 最大返回行数
  max_rows: 1000
  
  # 查询超时
  query_timeout: 30
  
  # 数据脱敏
  data_masking:
    - column_pattern: "email"
      mask: "partial"  # 显示为 a***@example.com
    - column_pattern: "phone"
      mask: "full"     # 完全隐藏

# 查询示例（用于 few-shot 提示）
examples:
  - question: "上个月销售额是多少"
    sql: |
      SELECT SUM(total_amount) as total_sales
      FROM orders
      WHERE status = 'completed'
        AND created_at >= DATE_TRUNC('month', CURRENT_DATE - INTERVAL '1 month')
        AND created_at < DATE_TRUNC('month', CURRENT_DATE)
  
  - question: "哪个地区的用户最多"
    sql: |
      SELECT region, COUNT(*) as user_count
      FROM users
      GROUP BY region
      ORDER BY user_count DESC
      LIMIT 10
```

## 11. 高阶技巧合集

### 11.1 技巧1：模型热切换（不同任务用不同模型）

不是所有任务都需要最强的模型。学会"按需选模"，能帮你省下一大笔 API 费用。

```yaml
# tips/model_routing.yaml
name: "smart_model_routing"
description: "智能模型路由配置"

# 模型路由规则
routing:
  # 简单任务用便宜模型
  - task_type: "summarize"
    model: "gpt-4o-mini"
    reason: "摘要任务对模型能力要求不高"
  
  - task_type: "classify"
    model: "gpt-4o-mini"
    reason: "分类是模型的基础能力"
  
  - task_type: "translate"
    model: "gpt-4o-mini"
    reason: "翻译任务 GPT-4o-mini 已足够优秀"
  
  # 复杂任务用强模型
  - task_type: "analyze"
    model: "gpt-4o"
    reason: "分析需要推理能力"
  
  - task_type: "code"
    model: "gpt-4o"
    reason: "代码生成需要高精度"
  
  - task_type: "creative"
    model: "gpt-4o"
    temperature: 0.8
    reason: "创意写作需要更强的理解和生成能力"
  
  # 超长文本用 Claude
  - condition: "input_tokens > 100000"
    model: "claude-sonnet"
    reason: "Claude 的上下文窗口更大"
  
  # 推理任务用 o1
  - task_type: "reasoning"
    model: "o1-mini"
    reason: "o1 在推理任务上表现更好"

# 动态降级
fallback:
  # 如果主模型超时或失败，降级到备选
  enabled: true
  chain:
    - "gpt-4o"
    - "gpt-4o-mini"
    - "claude-haiku"
```

**命令行快速切换**

```bash
# 查看当前模型
$ hermes config get model

# 临时切换模型（单次对话）
$ hermes chat --model gpt-4o "分析这份报告"

# 切换默认模型
$ hermes config set model gpt-4o-mini

# 查看各模型成本对比
$ hermes model compare --task summarize --input-length 1000
```

### 11.2 技巧2：记忆压缩与归档策略

随着时间推移，Hermes 的记忆库会越来越庞大。如果不管理，检索会变慢，成本会增加。

```yaml
# tips/memory_management.yaml
name: "memory_optimization"
description: "记忆压缩与归档策略"

# 自动归档
auto_archive:
  enabled: true
  
  # 归档规则
  rules:
    # 规则1：30 天前的会话记忆归档
    - condition: "age > 30d AND access_count < 3"
      action: "archive"
      destination: "cold_storage"
    
    # 规则2：90 天未访问的记忆归档
    - condition: "last_access > 90d"
      action: "archive"
      destination: "cold_storage"
    
    # 规则3：重复或冗余记忆合并
    - condition: "similarity_to_other > 0.95"
      action: "merge"
      keep: "most_detailed"

# 记忆压缩
compression:
  enabled: true
  
  # 压缩策略
  strategies:
    # 摘要压缩
    - type: "summarize"
      # 长对话压缩为摘要
      trigger: "conversation_length > 20"
      model: "gpt-4o-mini"
    
    # 向量化压缩
    - type: "embedding"
      # 用向量表示替代原始文本
      trigger: "storage_size > 100MB"
      model: "text-embedding-3-small"

# 分层存储
tiered_storage:
  # 热存储：最近 7 天，内存中
  hot:
    retention: "7d"
    storage: "memory"
    max_size: "50MB"
  
  # 温存储：7-90 天，本地 SQLite
  warm:
    retention: "90d"
    storage: "sqlite"
    path: "./memory/warm.db"
  
  # 冷存储：90 天以上，压缩归档
  cold:
    storage: "file"
    path: "./memory/archive/"
    compression: "gzip"
    # 需要时可以从冷存储恢复
    recoverable: true
```

### 11.3 技巧3：技能链式调用

```yaml
# tips/skill_chaining.yaml

# 链式工作流
chains:

      - skill: "web_search"

      - skill: "summarize"

      - skill: "write_article"
        input: "{{summary}}"

          style: "blog"
        output_var: "article"
      - skill: "proofread"
        input: "{{article}}"
        output_var: "final_article"
      - skill: "publish_to_wordpress"
        input: "{{final_article}}"
  
  # 链2：数据分析 -> 可视化 -> 报告
  data_to_report:
    name: "数据到报告"
    steps:
      - skill: "run_sql"
        query: "SELECT * FROM sales WHERE date > NOW() - INTERVAL '30 days'"
        output_var: "data"
      - skill: "analyze_data"
        input: "{{data}}"
        output_var: "insights"
      - skill: "generate_chart"
        input: "{{data}}"
        chart_type: "line"
        output_var: "chart_path"
      - skill: "generate_report"
        data: "{{data}}"
        insights: "{{insights}}"
        charts: ["{{chart_path}}"]
        output_var: "report"

# 条件分支
conditional_chains:
  - name: "智能客服处理"
    steps:
      - skill: "intent_detect"
        output_var: "intent"
      - branch:
          - condition: "intent == 'order_query'"
            chain: "order_lookup_chain"
          - condition: "intent == 'refund'"
            chain: "refund_chain"
          - condition: "intent == 'human'"
            chain: "escalate_chain"
```

### 11.4 技巧4：并行任务加速

当多个任务之间没有依赖关系时，可以并行执行，大幅缩短总时间。

```yaml
# tips/parallel_execution.yaml
name: "parallel_tasks"
description: "并行任务执行配置"

# 并行配置
parallel:
  # 默认并行度
  default_workers: 4
  
  # 最大并发数
  max_concurrent: 10

# 并行工作流示例
workflows:
  # 同时收集多个数据源
  multi_source_collection:
    steps:
      - parallel:
          - skill: "fetch_hacker_news"
            output_var: "hn_data"
          - skill: "fetch_reddit"
            output_var: "reddit_data"
          - skill: "fetch_github"
            output_var: "github_data"
          - skill: "fetch_twitter"
            output_var: "twitter_data"
      - skill: "merge_data"
        inputs:
          - "{{hn_data}}"
          - "{{reddit_data}}"
          - "{{github_data}}"
          - "{{twitter_data}}"
        output_var: "all_data"
  
  # 同时为多个平台生成内容
  multi_platform_generation:
    steps:
      - skill: "select_topic"
        output_var: "topic"
      - parallel:
          - skill: "write_for_wechat"
            input: "{{topic}}"
            output_var: "wechat_article"
          - skill: "write_for_xiaohongshu"
            input: "{{topic}}"
            output_var: "xhs_article"
          - skill: "write_for_douyin"
            input: "{{topic}}"
```

### 11.5 技巧5：自定义工具开发

Hermes 内置了 47 个工具，但你可能需要开发自己的工具。

```python
# custom_tools/my_tool.py
# 自定义工具开发示例

from hermes.tools import Tool, ToolParameter

class MyCustomTool(Tool):
    """
    自定义工具示例：汇率转换器
    """
    
    name = "currency_converter"
    description = "转换货币汇率"
    
    parameters = [
        ToolParameter(
            name="amount",
            type="number",
            description="金额",
            required=True
        ),
        ToolParameter(
            name="from_currency",
            type="string",
            description="源货币",
            required=True
        ),
        ToolParameter(
            name="to_currency",
            type="string",
            description="目标货币",
            required=True
        )
    ]
    
    async def execute(self, amount: float, from_currency: str, to_currency: str):
        """执行汇率转换"""
        # 调用汇率 API
        import aiohttp
        
        url = f"https://api.exchangerate-api.com/v4/latest/{from_currency}"
        
        async with aiohttp.ClientSession() as session:
            async with session.get(url) as response:
                data = await response.json()
                rate = data["rates"][to_currency]
                converted = amount * rate
                
                return {
                    "original_amount": amount,
                    "original_currency": from_currency,
                    "converted_amount": round(converted, 2),
                    "target_currency": to_currency,
                    "rate": rate,
                    "date": data["date"]
                }

# 注册工具
def register():
    return [MyCustomTool]
```
