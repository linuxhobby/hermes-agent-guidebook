# 第八册：技能系统完全指南

## 1. 技能系统：让 Hermes 从"聊天机器"变成"全能助手"

### 1.1 什么是 Skill（技能）

到了第二周，你正打算再教一次，实习生却说：「放心，上周您教我的流程我已经记下来了，以后每周五下午我自己就能搞定。」

这就是 Skill（技能）的本质——**把完成某项任务的标准化流程沉淀下来，让 Agent 能够自主复用**。

在 Hermes 中，Skill 是一个存放在 `~/.hermes/skills/<name>/SKILL.md` 路径下的 Markdown 文件。它不是一个简单的「提示词模板」，而是一个完整的「任务执行程序」，包含：

- **触发条件**：什么情况下应该调用这个技能
- **执行步骤**：具体的操作流程（可以包含工具调用、代码执行、文件操作等）
- **所需上下文**：执行这个技能需要哪些前置信息
- **示例输入输出**：让 Agent 理解期望的结果格式

> **重要区别：Skill vs 传统 Prompt 模板**
>
> 传统 Prompt 模板就像一份「菜谱」——上面写着「番茄炒蛋：先炒蛋，再炒番茄，混合在一起」。它只告诉你「做什么」，但不帮你「做」。
>
> Skill 就像一个「自动炒菜机程序」——它不仅知道菜谱，还能自动开煤气、倒油、打鸡蛋、控制火候。它包含的是**可执行的工作流**，而不仅仅是文字说明。

### 1.2 Skill 与传统 Prompt 模板的区别

为了更清晰地理解这个区别，我们来看一个具体的对比。

**场景：让 AI 帮你「搜索最新 AI 新闻并生成摘要」**

**传统 Prompt 模板的方式：**

```
请帮我搜索今天关于 AI 的最新新闻，然后生成一份摘要。
要求：
1. 搜索至少 5 条新闻
2. 每条新闻包含标题、来源、关键内容
3. 最后给出整体趋势判断
```

每次你需要这个服务时，都要把上面的提示词重新发一遍。而且如果结果不满意，你还得手动调整提示词。

**Hermes Skill 的方式：**

你第一次执行这个任务时，Agent 会：

1. 接收你的自然语言指令
2. 调用 Web Search 工具搜索新闻
3. 调用 LLM 生成摘要
4. 调用 File Write 把摘要保存为 Markdown
5. **检测到这是一个可复用的模式，自动生成 SKILL.md**

下次你只需说：「给我今天的 AI 新闻摘要」，Agent 就会自动加载对应的 Skill，按照既定流程执行——不需要你重复描述要求，不需要你手动调用工具，甚至连输出格式都会和上次保持一致。

这个区别的本质在于：**Prompt 模板是「描述」，Skill 是「程序」**。

正如 Hermes Atlas 官方文档所强调的：「Claude Code 的 memory 存储关于你的事实——偏好要点。而 Hermes 存储的是程序——产生你想要的要点的研究-过滤-格式化工作流。事实告知模型，程序直接被复用。」

这句话揭示了 Hermes 技能系统的独特价值：**它不仅让 Agent 记住「你喜欢什么」，更让 Agent 掌握「如何为你做到」**。前者是静态的信息，后者是动态的能力。

### 1.3 技能系统是 Hermes 最实用的功能

在与大量 Hermes 用户的交流中，我发现一个有趣的现象：当被问到「Hermes 最打动你的功能是什么」时，答案出奇地一致——**技能系统**。

为什么技能系统如此受欢迎？因为它解决了 AI 应用中最核心的痛点：**重复劳动**。

在没有技能系统之前，使用 AI 的工作流是这样的：

```
有需求 → 想提示词 → 写详细 Prompt → 执行 → 检查结果 → 不满意 → 改 Prompt → 再执行 → ...
        ↑___________________________________________________________↓
                              （每次都要重复）
```

有了技能系统之后，工作流变成了：

```
第一次：有需求 → 自然语言描述 → Agent 自动执行 → 自动生成 Skill
第二次及以后：有需求 → Agent 自动加载 Skill → 自动执行 → 结果满意
```

把重复的思考成本从人类转移给了机器，这就是技能系统的魔力。

根据社区实测数据，一个典型的 Hermes 用户在使用 1-2 周后：

- 重复任务的成功率提升 **30% 以上**
- 平均每个任务的交互轮次减少 **40%**
- 用户对 Agent 的满意度提升显著（因为结果更可预测、更一致）

这些数字的背后，是技能系统在默默发挥作用——它把「每次都从头教」变成了「一次教会，终身复用」。

### 1.4 技能系统的三大特点：自动创建、自我进化、跨平台复用

Hermes 的技能系统之所以强大，是因为它在三个维度上同时做到了业界领先：

**特点一：自动创建（Auto-Generation）**

这是 Hermes 最独门、最让其他框架羡慕的绝技。大多数 Agent 框架的技能（或类似的「工作流」「插件」）都需要开发者手动编写。而 Hermes 能够在检测到重复模式后，**自动生成 Skill 文件**。

触发自动创建的典型条件包括：

- 完成复杂任务（通常涉及 5+ 次工具调用）且成功
- 用户在错误后纠正了做法，Agent 学到了正确的路径
- 发现了非显而易见的工作流组合
- 用户明确说「记住这个流程，下次还要用」

这种自动创建不是简单的「保存聊天记录」，而是 Agent 主动提炼工作流、抽象通用步骤、生成可复用的 SKILL.md 文件。这个过程涉及对任务本质的理解——哪些步骤是通用的、哪些是特定于本次输入的、哪些参数应该暴露给用户调整。

**特点二：自我进化（Self-Evolution）**

技能不是一经创建就固定不变的。每次使用 Skill 时，Agent 都会收集反馈：

- 这次执行成功了吗？
- 用户有没有做修改或纠正？
- 输出结果与预期是否一致？
- 有没有更高效的方式可以完成？

基于这些反馈，Skill 会自动更新版本，优化执行步骤，修正错误。就像一位厨师，每次做菜后都会微调配方，越做越好。

进化可以是显式的（Agent 主动告诉你「Skill 已更新到 v1.2」），也可以是隐式的（后台默默优化，你发现结果越来越好了）。

**特点三：跨平台复用（Cross-Platform Reuse）**

Hermes 的 Skill 采用纯 Markdown 格式，不绑定任何特定平台或模型。这意味着：

- 你可以把 Skill 从 Claude Code 迁移到 Hermes
- 你可以把 Skill 从 Cursor 迁移到 Hermes
- 你可以把 Skill 分享给社区的其他用户
- 你可以在 Skills Hub 上找到别人分享的 Skill 并一键安装

Skill 的可移植性让它不仅仅是个人的效率工具，更成为社区协作的载体。目前 Skills Hub 上已经聚集了 **643 个社区技能**（截至 2026 年 4 月 19 日），涵盖开发、设计、数据分析、内容创作等几乎所有领域。

> **Tips**
>
> 如果你是刚接触 Hermes 的新用户，我的建议是：**不要急着手动创建 Skill**。先正常使用 Hermes 1-2 周，让自动创建机制自然积累你的常用工作流。你会惊讶地发现，Agent 已经默默为你生成了十几个实用的 Skill。在此基础上，再有针对性地手动优化和补充。

## 2. 技能的内部结构

### 2.1 Skill 文件格式详解（Markdown）

Hermes 的 Skill 文件是纯 Markdown 格式，这带来了几个显著优势：

1. **人类可读**：你可以直接用任何文本编辑器打开、阅读、修改
2. **版本控制友好**：Markdown 与 Git 配合完美，diff 清晰易读
3. **工具生态丰富**：无数的 Markdown 编辑器、预览工具、lint 工具
4. **跨平台**：任何操作系统都能处理 Markdown

Skill 的标准存储路径是：

```
~/.hermes/skills/
├── <skill-name-1>/
│   └── SKILL.md
├── <skill-name-2>/
│   └── SKILL.md
└── ...
```

每个 Skill 都有自己的目录，目录名就是 Skill 的名称。目录内目前主要存放 `SKILL.md`，但未来版本可能会支持同目录下的辅助文件（如模板文件、配置文件等）。

### 2.2 一个完整 Skill 文件的剖析

让我们拆解一个真实的 Skill 文件，看看它包含哪些部分。

以下是一个「新闻抓取与摘要」Skill 的完整示例：

```markdown
# News Fetch and Summarize

## Metadata
- Name: news-fetch-summarize
- Description: Search for latest news on a given topic and generate 
  a structured summary report
- Version: 1.2.0
- Tags: news, research, web-search, summary
- Author: auto-generated
- Created: 2026-04-15
- Last Updated: 2026-04-18

## Trigger Conditions
- User asks for "news" or "latest" about a topic
- User requests a "summary" of current events
- User mentions "what's happening" in a domain

## Required Context
- Topic: the subject to search for
- Time range: default to "today", user may specify (week/month)
- Output format: default to markdown, user may request (bullet/table)

## Execution Steps

### Step 1: Search
Use `web_search` tool to find at least 5 relevant news articles.
Query format: `<topic> news <time_range>`

### Step 2: Filter and Rank
Filter results by:
- Relevance to the topic (exclude off-topic results)
- Source credibility (prefer established news outlets)
- Recency (within the specified time range)

Rank by: relevance > credibility > recency

### Step 3: Extract Key Information
For each selected article, extract:
- Title
- Source name and URL
- Publication date
- 2-3 sentence summary of key content
- Any quoted statements or statistics

### Step 4: Generate Summary Report
Structure the output as:

```markdown
# <Topic> News Summary (<Date>)

## Overview
[2-3 sentence overall trend assessment]

## Key Stories

### 1. [Title]
- **Source**: [Name] ([URL])
- **Date**: [Publication date]
- **Summary**: [2-3 sentences]

### 2. [Title]
...

## Trend Analysis
[Overall pattern observation and forward-looking insight]

## Sources
- [Source 1]
- [Source 2]
- ...
```

### Step 5: Save (Optional)
If user requests, save the report to a file:
- Filename: `<topic>-news-<date>.md`
- Location: current working directory or user-specified

## Error Handling
- If web_search returns no results: expand query synonyms and retry
- If sources are all low-credibility: note this in the report
- If user topic is ambiguous: ask for clarification before proceeding

## Notes
- Always include source URLs for fact-checking
- Distinguish between facts and speculation in the summary
- For controversial topics, present multiple viewpoints
- Never fabricate information not present in sources

## Example Input
User: "Give me today's AI news summary"

## Example Output
Agent:
"I'll search for the latest AI news and generate a summary for you."
[Executes Steps 1-4]
"Here's your AI News Summary for April 21, 2026:

# AI News Summary (2026-04-21)

## Overview
Today's AI landscape sees significant advances in multimodal 
reasoning and enterprise adoption...

[Full report follows]"
```

这个文件清晰地展示了一个 Skill 的完整结构。让我们逐一分析每个部分。

#### 元数据区（Metadata）

```markdown
## Metadata
- Name: news-fetch-summarize
- Description: Search for latest news on a given topic and generate 
  a structured summary report
- Version: 1.2.0
- Tags: news, research, web-search, summary
- Author: auto-generated
- Created: 2026-04-15
- Last Updated: 2026-04-18
```

元数据区提供 Skill 的基本身份信息：

- **Name**：Skill 的唯一标识符，通常使用 kebab-case（短横线连接的小写字母）
- **Description**：一句话描述 Skill 的功能，用于搜索和展示
- **Version**：语义化版本号（主版本.次版本.修订号），方便追踪变更
- **Tags**：分类标签，便于管理和搜索
- **Author/.Created/Last Updated**：追溯信息

#### 触发条件区（Trigger Conditions）

```markdown
## Trigger Conditions
- User asks for "news" or "latest" about a topic
- User requests a "summary" of current events
- User mentions "what's happening" in a domain
```

触发条件定义了 Agent 应该在什么情况下「想到」使用这个 Skill。这就像是给 Agent 装了一个「雷达」——当用户的输入匹配这些条件时，Agent 就知道「啊，这个任务可以用 news-fetch-summarize 技能来处理」。

触发条件可以是：

- **关键词匹配**：用户输入中包含特定词汇
- **意图识别**：用户的请求属于某种意图类别
- **上下文匹配**：当前对话的上下文符合某种模式
- **工具链匹配**：完成任务需要调用特定的工具组合

#### 执行步骤区（Execution Steps）

这是 Skill 的核心部分，定义了完成任务的具体流程。每个步骤都包含：

- **步骤编号和名称**：清晰的流程结构
- **具体操作**：使用什么工具、调用什么命令、执行什么逻辑
- **参数说明**：输入参数的定义和格式
- **输出规范**：该步骤应该产生什么结果

```markdown
### Step 1: Search
Use `web_search` tool to find at least 5 relevant news articles.
Query format: `<topic> news <time_range>`
```

步骤之间可以是有顺序依赖的（Step 2 依赖 Step 1 的结果），也可以是并行的（多个步骤可以同时执行）。

#### 注意事项区（Notes / Error Handling）

```markdown
## Error Handling
- If web_search returns no results: expand query synonyms and retry
- If sources are all low-credibility: note this in the report
- If user topic is ambiguous: ask for clarification before proceeding

## Notes
- Always include source URLs for fact-checking
- Distinguish between facts and speculation in the summary
```

这个区域记录执行过程中的注意事项、边界情况、错误处理策略等。它帮助 Agent 在遇到意外情况时做出正确的判断，而不是盲目继续或报错退出。

#### 示例输入输出区（Example Input/Output）

```markdown
## Example Input
User: "Give me today's AI news summary"

## Example Output
Agent: "I'll search for the latest AI news..."
```

示例是 Skill 质量的关键指标。好的示例能让 Agent 更准确地理解用户的期望和输出的标准。示例越多、覆盖面越广，Skill 的泛化能力就越强。

### 2.3 技能文件存储位置：`~/.hermes/skills/`

所有 Skill 都存储在用户的 Hermes 配置目录下的 `skills/` 文件夹中：

```
~/.hermes/skills/
├── news-fetch-summarize/
│   └── SKILL.md
├── code-review-python/
│   └── SKILL.md
├── data-analysis-pipeline/
│   └── SKILL.md
├── docker-deploy/
│   └── SKILL.md
└── ...
```

这种目录结构的设计考虑到了未来的扩展性：

- 每个 Skill 有自己的目录，避免命名冲突
- 可以在目录中添加辅助文件（如模板、配置、测试数据）
- 便于打包、分享和版本控制

你可以直接操作这个目录：

```bash
# 查看所有技能
$ ls ~/.hermes/skills/

# 查看某个技能的内容
$ cat ~/.hermes/skills/news-fetch-summarize/SKILL.md

# 手动复制技能（分享给同事）
$ cp -r ~/.hermes/skills/news-fetch-summarize ./shared-skills/

# 手动安装技能（从同事那里收到的）
$ cp -r ./shared-skills/news-fetch-summarize ~/.hermes/skills/
```

### 2.4 技能的命名规范

好的命名让 Skill 更容易被发现和使用。Hermes 社区推荐使用以下命名规范：

**基本格式：**

```
<action>-<target>[-<context>]
```

例如：

- `search-news`：搜索新闻
- `analyze-sentiment`：分析情感
- `deploy-docker-prod`：部署 Docker 到生产环境
- `generate-report-weekly`：生成周报

**命名原则：**

1. **使用小写字母和短横线**：kebab-case 是标准，不要用空格或驼峰
2. **动作优先**：先说明「做什么」，再说明「对谁做」
3. **保持简洁**：名称不要超过 3-4 个词
4. **避免歧义**：`fix` 不如 `fix-bug` 或 `fix-lint-error` 清晰
5. **添加版本后缀**：如果同一技能有多个变体，用 `-v2`、`-lite`、`-advanced` 区分

**按分类命名（可选前缀）：**

```
web-search-*        # 网络搜索相关
code-*              # 代码相关
data-*              # 数据分析相关
devops-*            # 运维部署相关
content-*           # 内容创作相关
research-*          # 研究分析相关
```

> **Tips**
>
> 虽然 Hermes 支持中文 Skill 名称（因为文件名只是标识符），但强烈建议使用英文命名。原因有三：1）便于在 Skills Hub 上分享；2）Agent 在匹配触发条件时，英文关键词更稳定；3）与社区其他用户交流时无障碍。

## 3. 自动技能创建：Hermes 的独门绝技

### 3.1 触发条件详解

自动技能创建是 Hermes 区别于其他所有 Agent 框架的标志性功能。理解它的触发条件，能帮助你更好地「引导」Agent 为你生成有用的 Skill。

根据 Hermes Atlas 官方文档，Skill 在检测到重复模式后自动生成，典型的触发场景包括：

**场景一：完成复杂任务（5+ 工具调用）且成功**

这是最主要的触发条件。当 Agent 完成一个涉及多次工具调用的复杂任务时，系统会判断这个工作流是否值得沉淀为 Skill。

为什么是「5+ 次工具调用」？这个数字是经过大量实验验证的「甜蜜点」：

- **少于 5 次**：流程太简单，不值得专门创建 Skill（直接每次手动做就行）
- **5 次以上**：说明任务有一定复杂度，包含多个步骤的协调，复用价值高
- **过多（20+ 次）**：可能是任务粒度太大，生成的 Skill 会过于臃肿

> **官方规则：5-Tool-Call 规则**
>
> Hermes 的自动技能创建有一个不成文的「5-tool-call 规则」：当 Agent 在一次任务中成功调用了 5 个或更多工具，且任务最终成功完成，系统就会评估这个工作流是否适合生成 Skill。这个规则确保了只有足够复杂、有复用价值的工作流才会被沉淀下来。

**场景二：遇到错误最终找到可行路径**

这是一个特别有价值的触发场景。当 Agent 在执行任务时遇到了错误，经过多轮尝试（可能包括你的纠正和引导）最终找到了正确的解决方案，这个「试错→纠正→成功」的完整路径会被记录为一个 Skill。

为什么这种 Skill 特别有价值？因为它包含了「避坑指南」——不仅告诉你「怎么做是对的」，还告诉你「哪些地方容易出错」。

**场景三：用户纠正了做法**

当你说「不对，应该这样...」或者「下次记得先...再...」时，Agent 会把你的纠正作为重要反馈。如果这种纠正在类似场景中出现过多次，Agent 会推断出一个通用规则，生成 Skill 来固化正确的做法。

```bash
User: 帮我部署这个服务
Agent: [尝试直接部署... 失败了]
User: 不对，应该先检查环境变量配置好了没有
Agent: [检查环境变量... 发现问题... 修复后部署成功]

[系统检测到：用户纠正了流程，生成了 Skill "deploy-with-env-check"]
```

**场景四：发现非显而易见的工作流**

有时候 Agent 会「创造性地」组合多个工具来完成任务，这种组合可能连开发者都没有预料到。当这种「非显而易见的工作流」被验证为有效时，系统会立即生成 Skill 来记录这个「发现」。

```bash
User: 帮我找出这个项目里最近引入 Bug 的提交
Agent: [没有直接的 "find-bug-commit" 工具...]
  → 调用 git log 获取提交历史
  → 调用 git diff 查看每个提交的变更
  → 调用 code_review 工具分析变更
  → 交叉对比 Bug 报告的时间线
  → 找到最可疑的提交

[系统检测到：创新工作流组合，生成了 Skill "git-bisect-intelligent"]
```

### 3.2 创建过程详解（从任务执行到 Skill 生成）

自动创建 Skill 不是一个「瞬间完成」的操作，而是一个分阶段的过程。让我们深入了解它的内部机制。

**阶段一：模式识别（Pattern Detection）**

在任务执行过程中，系统会实时监控工具调用序列、用户输入、Agent 输出之间的关系。它寻找的是「可重复的模式」：

```
工具调用序列: [web_search] → [file_read] → [code_execute] → [file_write]
用户输入模式: "帮我分析...并生成报告"
输出模式: Markdown 报告，包含图表

→ 检测到模式：搜索-读取-分析-输出报告
```

**阶段二：通用化抽象（Generalization）**

识别到模式后，系统需要判断这个模式是「通用」的还是「特定于本次输入」的。通用化的过程包括：

- 提取可变参数（如搜索关键词、文件路径、输出格式）
- 识别固定步骤（如「先搜索再分析」的顺序）
- 确定默认值（如默认搜索 5 条结果、默认输出 Markdown）

```
特定实例:
  用户说 "分析 Python 趋势" → 搜索 Python 新闻 → 生成 Python 报告

通用模式:
  用户说 "分析 <topic> 趋势" → 搜索 <topic> 新闻 → 生成 <topic> 报告
  
→ 可变参数: <topic>
→ 固定步骤: 搜索 → 分析 → 生成报告
→ 默认: 搜索 5 条，输出 Markdown
```

**阶段三：质量评估（Quality Assessment）**

不是所有识别出的模式都值得生成 Skill。系统会对候选 Skill 进行质量评分：

| 评估维度 | 权重 | 说明 |
|---------|------|------|
| 复用潜力 | 30% | 这个模式在未来被用到的概率 |
| 执行成功率 | 25% | 按照此模式执行，任务成功的概率 |
| 复杂度适中 | 20% | 不太简单（有沉淀价值）也不太复杂（可维护） |
| 用户满意度 | 15% | 用户对这次任务结果的评价 |
| 创新性 | 10% | 是否发现了新的工具组合方式 |

总分超过阈值的候选 Skill 才会进入下一阶段。

**阶段四：SKILL.md 生成**

通过质量评估后，系统会调用 LLM 生成标准的 SKILL.md 文件。生成过程使用专门的「Skill 生成提示模板」，确保输出符合规范。

生成的 SKILL.md 包含：

1. 从任务描述中提取的元数据
2. 从用户输入历史中归纳的触发条件
3. 从工具调用链中提取的执行步骤
4. 从对话中总结的注意事项和错误处理
5. 从实际输入输出中提取的示例

**阶段五：用户确认（可选）**

根据配置，Hermes 可能会在新 Skill 生成后征求你的确认：

```bash
Agent: 我刚刚为你生成了一个新的 Skill：news-fetch-summarize
       这个 Skill 可以帮你自动搜索新闻并生成摘要。
       
       触发条件：当你要求新闻摘要时
       执行步骤：搜索 → 过滤 → 提取 → 生成报告
       
       是否保存这个 Skill？(y/n/edit)

User: y
Agent: Skill 已保存到 ~/.hermes/skills/news-fetch-summarize/SKILL.md
```

你也可以配置为「自动保存，无需确认」，适合高级用户。

### 3.3 自动提炼的算法逻辑

自动提炼的核心算法可以概括为「频率 + 重要性 + 结构性」的三维评估模型。

**维度一：频率（Frequency）**

统计某个工作流在多长时间内被重复了多少次：

```python
frequency_score = min(repeat_count / threshold, 1.0)
# threshold 通常为 2-3 次
# 即同一个工作流重复 2-3 次，频率维度就达到满分
```

**维度二：重要性（Importance）**

评估工作流对用户的价值：

```python
importance_factors = {
    'tool_call_count': len(tool_calls),           # 工具调用越多越重要
    'user_explicit_save': user_said_remember,      # 用户明确说记住
    'task_success': task_completed_successfully,   # 任务成功完成
    'error_recovery': recovered_from_error,        # 从错误中恢复
    'time_saved': estimated_time_vs_manual         # 节省的时间
}
```

**维度三：结构性（Structure）**

评估工作流是否有清晰的步骤结构：

```python
structure_factors = {
    'has_clear_steps': has_stepwise_pattern,       # 有清晰的步骤序列
    'has_parameters': has_extractable_params,      # 有可提取的参数
    'has_error_handling': handles_edge_cases,      # 有错误处理
    'has_examples': has_io_examples                 # 有输入输出示例
}
```

综合评分 = 频率 × 0.3 + 重要性 × 0.4 + 结构性 × 0.3

评分超过 0.7 的候选 Skill 会被生成。

## 4. 技能自我进化

### 4.1 进化触发机制

Skill 的自我进化不是随机发生的，而是由明确的触发机制驱动的。理解这些机制，能帮助你更好地「培养」你的 Skill。

**触发器一：执行偏差检测**

当 Skill 的实际执行路径与 SKILL.md 中定义的路径出现偏差时，进化被触发。

```
SKILL.md 定义:
  Step 1: web_search
  Step 2: file_read
  Step 3: code_execute

实际执行:
  Step 1: web_search → 返回 0 结果
  [偏差!] Agent 自动添加了 Step 1b: 扩展搜索关键词
  Step 2: file_read
  Step 3: code_execute

→ 触发进化: 在 SKILL.md 中添加错误处理分支
  "If web_search returns 0 results, expand query synonyms and retry"
```

**触发器二：用户反馈**

用户对 Skill 执行结果的评价（显式或隐式）会直接驱动进化。

显式反馈：

```bash
User: 这个报告格式不太对，应该用表格而不是列表
Agent: 收到反馈。正在更新 news-fetch-summarize Skill...
       已将默认输出格式从列表改为表格。
       Skill 已升级到 v1.2.0。
```

隐式反馈：

```bash
User: [看到 Skill 输出后，手动修改了 5 处]
Agent: [检测到用户修改了输出]
       [分析修改模式...]
       [发现用户偏好：更喜欢紧凑的表格、不喜欢冗余的解释]
       [更新 Skill 以匹配用户偏好]
```

**触发器三：成功率统计**

系统会持续跟踪每个 Skill 的成功率。当成功率低于阈值时，触发诊断和修复流程：

```
Skill: deploy-release-pipeline
成功率统计:
  最近 10 次执行: 7 成功, 3 失败
  失败原因:
    - 2 次: 测试未通过就尝试部署
    - 1 次: 版本号未更新

→ 触发进化:
  1. 在 Step 1 中添加更严格的测试检查
  2. 在 Step 2 中添加版本号自动验证
  3. 如果版本号未更新，提示用户或自动更新
```

**触发器四：环境变化**

当 Skill 依赖的外部环境发生变化时（如工具 API 更新、文件路径变更），Skill 会自动适应：

```
检测到: web_search 工具返回格式已变更
  旧格式: {"results": [{"title": "...", "url": "..."}]}
  新格式: {"items": [{"title": "...", "link": "..."}]}

→ 触发进化:
  更新 Skill 中所有引用 web_search 结果的步骤
  将 "url" 改为 "link"
  将 "results" 改为 "items"
```

### Step 1: Access Setup
- Add to GitHub organization
- Grant VPN access
- Create Slack account

### Step 2: Environment
- Clone repo: git@github.com:company/project.git

- Verify: make test

这种流程在项目开始前就定义好了，自动创建无法预知。

**场景二：高度定制化的需求**

自动创建的 Skill 是「通用化」的，可能缺少你的特殊要求。比如你公司内部的代码规范非常严格，自动生成的 code-review Skill 不会包含这些规范。

**场景三：集成内部系统**

如果你的 Skill 需要调用公司内部 API（如 JIRA、Confluence、自研平台），自动创建无法知道这些内部系统的存在。

**场景四：从其他平台迁移**

你从 Claude Code 或 Cursor 迁移过来，想把已有的 Skill/规则迁移到 Hermes。

**场景五：追求极致优化**

自动创建的 Skill 是「70 分方案」。如果你想达到「95 分」，手动调整和优化是必要的。

### 5.2 手动创建的完整教程

手动创建一个 Skill 只需要三步：创建目录、编写 SKILL.md、验证测试。

**第一步：创建 Skill 目录**

```bash
# 确定 Skill 名称（使用 kebab-case）
SKILL_NAME="my-custom-skill"

# 创建目录
mkdir -p ~/.hermes/skills/$SKILL_NAME

# 创建 SKILL.md 文件
touch ~/.hermes/skills/$SKILL_NAME/SKILL.md
```

**第二步：编写 SKILL.md**

用你喜欢的编辑器打开文件，按照以下模板编写：

```markdown
# <Skill 名称>

## Metadata
- Name: <skill-name>
- Description: <一句话描述>
- Version: 1.0.0
- Tags: <tag1>, <tag2>
- Author: <你的名字>
- Created: <日期>

## Trigger Conditions
- <触发条件 1>
- <触发条件 2>

## Required Context
- <需要的上下文 1>
- <需要的上下文 2>

## Execution Steps

### Step 1: <步骤名称>
<具体描述>

### Step 2: <步骤名称>
<具体描述>

## Error Handling
- <错误情况 1>: <处理方式>
- <错误情况 2>: <处理方式>

## Notes
- <注意事项 1>
- <注意事项 2>

## Example Input
<User 输入示例>

## Example Output
<Agent 输出示例>
```

**第三步：验证 Skill**

```bash
# 验证 Skill 格式
$ hermes skills validate my-custom-skill

# 测试执行
$ hermes skills test my-custom-skill --input "测试输入"

# 查看 Skill 详情
$ hermes skills show my-custom-skill
```

### 5.4 编写高质量 Skill 的 10 个技巧

经过社区大量实践，我们总结出了编写高质量 Skill 的 10 个核心技巧：

**技巧 1：触发条件要具体**

差的触发条件：
```markdown
## Trigger Conditions
- User wants to do something with data
```

好的触发条件：
```markdown
## Trigger Conditions
- User asks to "convert", "transform", or "reformat" data
- User mentions changing file format (CSV, JSON, Excel, etc.)
- User provides data and asks for different output format
```

**技巧 2：步骤要原子化**

每个步骤只做一件事，步骤之间用明确的输入输出连接：

```markdown
### Step 1: Read Source
Input: file path or raw data
Output: parsed data structure

### Step 2: Validate
Input: parsed data
Output: validation report (pass/fail with details)

### Step 3: Transform
Input: validated data
Output: transformed data
```

**技巧 3：错误处理要全面**

思考所有可能的失败情况：

```markdown
## Error Handling
- File not found: ask user for correct path
- Permission denied: suggest checking file permissions
- Invalid format: show expected vs actual format
- Empty result: inform user and suggest alternatives
- Network timeout: retry with exponential backoff
```

**技巧 4：示例要真实**

使用真实的输入输出示例，不要用占位符：

```markdown
## Example Input
User: "Convert this CSV to JSON: name,age,city\nAlice,30,NYC"

## Example Output
Agent: "Here's the JSON:\n[\n  {\"name\": \"Alice\", \"age\": 30, \"city\": \"NYC\"}\n]"
```

**技巧 5：版本号要管理**

从 v1.0.0 开始，每次有意义的更新都升级版本号。

**技巧 6：标签要准确**

使用社区共识的标签体系，便于搜索和分类：

```markdown
- Tags: web-search, data-analysis, python, api, documentation
```

**技巧 7：上下文要声明**

明确说明执行这个 Skill 需要什么前置信息：

```markdown
## Required Context
- API key for external service (stored in env var or MEMORY.md)
- Input data file path
- Target output directory
```

**技巧 8：输出格式要规范**

定义清晰的输出格式，让结果可预测：

```markdown
## Output Format
- Status: success / partial / failed
- Summary: one-line description
- Details: structured data
- Next steps: recommended actions
```

**技巧 9：考虑边界情况**

问自己：输入为空怎么办？数据量太大怎么办？网络断了怎么办？

**技巧 10：保持简洁**

SKILL.md 不是论文，不需要长篇大论。用简洁的语言描述清晰的步骤。如果 Skill 太复杂，考虑拆分成多个子 Skill。

### 5.5 测试与验证技能

创建 Skill 后，必须进行测试。Hermes 提供了多种测试方式：

### 7.2 四个信任等级体系

Hermes 建立了一套严格的信任等级体系，帮助用户评估 Skill 的安全性和可靠性。根据 Hermes Atlas 官方文档，所有 Skill 分为四个信任等级：

| 等级 | 名称 | 标识 | 说明 | 安全审计 |
|------|------|------|------|---------|
| 1 | 内置 (Built-in) | 🏠 | 随 Hermes 发行版一同发布 | 完整安全审计 |
| 2 | 官方 (Official) | ✅ | Nous Research 团队发布 | 完整安全审计 |
| 3 | 可信 (Trusted) | 👤 | 经验证的社区成员发布 | 代码扫描 + 人工抽查 |
| 4 | 社区 (Community) | 🌐 | 任何社区成员发布 | 自动扫描（建议查看源码） |

**等级详解：**

**内置（Built-in）**

内置技能随 Hermes 安装包一起分发，是系统核心功能的一部分。它们经过了最严格的安全审计和性能测试，是最值得信赖的 Skill。

内置技能的特点是：
- 代码完全透明，可以在 GitHub 上查看源码
- 更新频率与 Hermes 主版本同步
- 不依赖外部网络服务（纯本地执行）
- 有专门的维护团队负责

**官方（Official）**

官方技能由 Nous Research 团队或官方认证的贡献者开发和维护。它们不随 Hermes 一起安装，需要通过 Skills Hub 单独安装，但同样经过了完整的安全审计。

官方技能的特点是：
- 有官方团队背书
- 定期更新和维护
- 文档完整、示例丰富
- 与 Hermes 核心功能深度集成

**可信（Trusted）**

可信技能来自经验证的社区成员。这些开发者已经通过了社区的身份验证和技能质量审核，他们发布的 Skill 被标记为「可信」。

可信技能的特点是：
- 开发者身份已验证
- 代码经过自动扫描（检测恶意代码、敏感信息泄露等）
- 人工抽查通过
- 有明确的维护承诺

**社区（Community）**

社区技能是 Skills Hub 上数量最多的类别。任何注册开发者都可以发布社区技能。这些技能经过了自动扫描，但没有人工审计。

社区技能的特点是：
- 数量庞大（截至 2026 年 4 月 19 日，Skills Hub 上共有 **643 个社区技能**）
- 创新速度快，覆盖领域广
- 质量参差不齐
- **建议安装前查看源码**

> **安全建议**
>
> 安装 Skill 时，优先选择高等级的 Skill。对于社区等级的 Skill，安装前务必查看 SKILL.md 的内容，确认：1）它不会执行危险的系统命令；2）不会向外部发送你的敏感数据；3）代码逻辑合理、无恶意行为。

## 8. 跨平台技能复用

### 8.1 从 Claude Code 迁移技能

Claude Code 是 Anthropic 推出的 AI 编程助手，它也有一套「记忆」和「规则」系统。但正如 Hermes Atlas 官方文档所指出的：Claude Code 存储的是「事实」（如用户的偏好要点），而 Hermes 存储的是「程序」（可执行的工作流）。这种本质区别意味着迁移需要一些转换工作。

**Claude Code 的记忆格式：**

Claude Code 的记忆通常以简单的文本片段存储，记录用户的偏好：

```
# Claude Code Memory (示例)
- User prefers TypeScript over JavaScript
- User likes concise responses
- User's project uses Next.js and Prisma
- Always run tests before committing
```

这些「事实」类记忆可以迁移到 Hermes 的 USER.md 中，直接就能用。

**Claude Code 的规则格式：**

Claude Code 的 `.claude/rules/` 目录下的规则文件，更接近 Hermes 的 Skill：

```markdown
# .claude/rules/code-style.md

When writing code:
1. Use TypeScript with strict mode
2. Prefer functional programming patterns
3. Add JSDoc comments for all public APIs
4. Run eslint and prettier before finishing
```

**迁移方法：**

```bash
# 1. 找到 Claude Code 的规则目录
$ ls ~/.claude/rules/
code-style.md  testing.md  deploy.md

# 2. 转换为 Hermes Skill 格式
# 可以手动转换，或使用迁移工具
$ hermes skills migrate --from claude-code ~/.claude/rules/

正在迁移 Claude Code 规则...
✓ code-style.md → ~/.hermes/skills/code-style/SKILL.md
✓ testing.md → ~/.hermes/skills/testing/SKILL.md
⚠ deploy.md → 需要手动调整（包含 Claude 特有语法）

迁移完成！
请检查生成的 SKILL.md 文件，根据需要调整触发条件和执行步骤。
```

**迁移注意事项：**

1. **触发条件需要补充**：Claude Code 的规则是「全局生效」的，而 Hermes 的 Skill 需要明确的触发条件。迁移后要手动添加触发条件。
2. **工具调用需要转换**：Claude Code 的某些命令（如 `/commit`）在 Hermes 中对应不同的工具调用。
3. **测试验证**：迁移后务必测试每个 Skill，确保在 Hermes 环境下正常工作。

### 8.2 从 Cursor 迁移技能

Cursor 是另一款流行的 AI 编程编辑器，它的「Rules」和「Project Rules」功能与 Hermes 的 Skill 有一定对应关系。

# Code Style

- Type hints are mandatory

# Project Structure

- Tests in tests/

### 8.3 从 OpenClaw 迁移技能

如果你是从 OpenClaw 迁移到 Hermes 的用户，技能迁移是最平滑的部分之一。因为 Hermes 本身就是从 OpenClaw 演进而来，两者的 Skill 格式高度兼容。

**OpenClaw 技能格式：**

```yaml
# .claude/skills/my-skill.yaml
name: my-skill
description: Do something useful
triggers:
  - pattern: "do something"
steps:
  - type: tool
    tool: web_search
    args:
      query: "{{input}}"
  - type: process
    action: summarize
```

**迁移方法：**

```bash
# OpenClaw 到 Hermes 的迁移非常简单
$ hermes skills migrate --from openclaw ~/.claude/skills/

正在迁移 OpenClaw 技能...
发现 5 个技能:
  ✓ my-skill → 自动转换格式
  ✓ data-fetch → 自动转换格式
  ✓ code-review → 自动转换格式
  ⚠ deploy-flow → 需要手动检查（使用了 OpenClaw 特有语法）
  ✓ test-gen → 自动转换格式

4/5 个技能自动迁移成功！
```

**OpenClaw → Hermes 格式映射：**

| OpenClaw | Hermes | 说明 |
|---------|--------|------|
| `.claude/skills/*.yaml` | `~/.hermes/skills/*/SKILL.md` | 路径不同 |
| `name` | `Metadata.Name` | 相同 |
| `description` | `Metadata.Description` | 相同 |
| `triggers[].pattern` | `Trigger Conditions` | 合并为列表 |
| `steps[].type: tool` | `Execution Steps` | 转为 Markdown 描述 |
| `steps[].type: process` | `Execution Steps` | 转为 Markdown 描述 |

### 8.4 技能格式兼容性说明

不同平台的技能系统虽然名称不同，但核心概念是相通的。以下是主要平台的兼容性对照：

| 平台 | 技能名称 | 文件格式 | 与 Hermes 兼容性 |
|------|---------|---------|----------------|
| Hermes | Skill | Markdown (SKILL.md) | 原生格式 |
| Claude Code | Rule | Markdown | 高（需手动调整触发条件） |
| Cursor | Rule | Markdown (.cursorrules) | 中（全局规则需转触发条件） |
| OpenClaw | Skill | YAML | 高（自动迁移） |
| AutoGPT | Skill | JSON/Python | 低（需重写） |
| LangChain | Chain | Python | 低（概念不同） |

**通用迁移原则：**

1. **提取核心逻辑**：无论原格式如何，提取「触发条件」和「执行步骤」这两个核心要素
2. **转换为 Markdown**：Hermes 使用 Markdown，这是最通用的文档格式
3. **补充触发条件**：其他平台的规则可能是全局的，需要为 Hermes 定义明确的触发条件
4. **测试验证**：迁移后必须测试，不同平台的工具名称和 API 可能有差异

### 8.5 迁移后的调整与优化

迁移完成后，建议进行以下优化：

**优化一：调整触发条件**

迁移后的 Skill 触发条件可能过于宽泛或过于狭窄。根据实际使用场景调整：

```markdown
# 迁移后（过于宽泛）
## Trigger Conditions
- User mentions code

# 优化后（更精确）
## Trigger Conditions
- User asks to "review" or "check" code
- User mentions "lint", "format", or "style" issues
- User requests "code quality" improvement
```

**优化二：补充错误处理**

其他平台的规则可能缺少错误处理，迁移后补充：

```markdown
## Error Handling
- If file not found: ask user for correct path
- If syntax error in code: show error location and suggest fix
- If tool execution timeout: suggest breaking task into smaller steps
```

**优化三：添加示例**

为迁移后的 Skill 添加输入输出示例，帮助 Agent 理解期望的行为：

```markdown
## Example Input
User: "Review this Python function for style issues"
[attached: function.py]

## Example Output
Agent: "I've reviewed function.py. Found 3 style issues:
       1. Line 15: Missing type hints
       2. Line 22: Line too long (120 chars)
       3. Line 30: Unused import
       
       Suggested fixes: [shown inline]"
```

**优化四：版本标记**

在 Metadata 中标注这是迁移来的 Skill：

```markdown
## Metadata
- Name: my-migrated-skill
- Version: 1.0.0-hermes
- Original Platform: Cursor
- Migration Date: 2026-04-21
- Notes: Migrated from .cursorrules, adjusted trigger conditions
```

## 10. 高阶技能开发

### 10.1 复合技能（一个技能调用多个子技能）

复合技能（Composite Skill）是指一个 Skill 在执行过程中调用其他 Skill，形成「技能链」。这就像乐高积木——单个积木功能简单，但组合起来可以搭出复杂结构。

**复合技能的结构：**

```markdown
# Fullstack Project Setup

## Metadata
- Name: fullstack-project-setup
- Description: Complete setup for a new fullstack project
- Version: 1.0.0

## Trigger Conditions
- User says "create a new project" or "setup project"
- User mentions "fullstack" + "scaffold" or "template"

## Execution Steps

### Step 1: Tech Stack Selection
**Call Skill**: gstack
Input: user requirements
Output: recommended tech stack

### Step 2: Backend Setup
**Call Skill**: boilerplate-generator
Input: backend framework from Step 1
Output: backend project scaffold

### Step 3: Frontend Setup
**Call Skill**: boilerplate-generator
Input: frontend framework from Step 1
Output: frontend project scaffold

### Step 4: Docker Configuration
**Call Skill**: docker-deploy
Input: project structure from Steps 2-3
Output: Dockerfile + docker-compose.yml

### Step 5: API Documentation
**Call Skill**: api-doc-generator
Input: backend code from Step 2
Output: OpenAPI specification

### Step 6: Initial Tests
**Call Skill**: test-automation
Input: generated code
Output: initial test suite

### Step 7: Security Scan
**Call Skill**: security-audit
Input: entire project
Output: security report + fixes
```

**复合技能的优势：**

1. **模块化**：每个子 Skill 独立维护，复用性高
2. **可替换**：如果某个子 Skill 有更好的替代品，只需替换调用
3. **可配置**：用户可以选择跳过某些步骤

### 10.2 条件分支技能

条件分支技能根据输入或环境的不同，执行不同的流程。

```markdown
# Smart Deploy

## Execution Steps

### Step 1: Detect Environment
Check current environment:
- If `.github/workflows/` exists → GitHub Actions
- If `.gitlab-ci.yml` exists → GitLab CI
- If `Jenkinsfile` exists → Jenkins
- If none → ask user

### Step 2: Branch by Environment

**If GitHub Actions:**
- Call Skill: github-actions-deploy

**If GitLab CI:**
- Call Skill: gitlab-ci-deploy

**If Jenkins:**
- Call Skill: jenkins-deploy

**If Manual:**
- Generate deployment script
- Provide step-by-step instructions

### Step 3: Verify Deployment
Regardless of path:
- Check service health
- Run smoke tests
- Monitor for 5 minutes
```

### 10.4 技能的错误处理

健壮的错误处理是高质量 Skill 的标志。

**错误处理策略模板：**

```markdown
## Error Handling

### Network Errors
- **Timeout**: Retry with exponential backoff (max 3 retries)
- **Connection Refused**: Check service status, suggest alternatives
- **DNS Error**: Verify URL, suggest checking network

### Data Errors
- **Invalid Format**: Show expected format, provide converter
- **Missing Required Field**: List missing fields, ask user to provide
- **Data Too Large**: Suggest pagination or streaming

### Permission Errors
- **Access Denied**: Ask for credentials or elevated permissions
- **Rate Limited**: Wait and retry, suggest upgrading plan
- **Quota Exceeded**: Inform user, suggest alternatives

### Tool Errors
- **Tool Not Found**: Check installation, provide setup guide
- **Tool Version Mismatch**: Detect version, suggest update/downgrade
- **Tool Execution Failed**: Capture error output, suggest fixes

### Fallback Strategy
If all else fails:
1. Report what was attempted
2. Explain why it failed
3. Provide manual workaround
4. Offer to create a simpler version
```

### 10.5 技能的性能优化

Skill 的执行效率直接影响用户体验。以下是优化技巧：

**技巧一：减少不必要的工具调用**

```markdown
# 优化前（4次调用）
Step 1: web_search → 获取结果
Step 2: file_read → 读取模板
Step 3: web_search → 获取更多信息
Step 4: file_write → 写入输出

# 优化后（2次调用）
Step 1: web_search ×2 (并行) → 获取所有信息
Step 2: file_write → 直接写入（模板内联到 Skill 中）
```

**技巧二：缓存中间结果**

```markdown
## Notes
- Cache API responses for 5 minutes to avoid repeated calls
- Store intermediate results in MEMORY.md if reusable
```

**技巧三：批量处理**

```markdown
# 优化前（逐条处理）
For each item:
  Call API
  Process result

# 优化后（批量处理）
Batch items into groups of 100
Call API with batch
Process results
```

**技巧四：异步执行**

```markdown
## Execution Steps
### Step 1: Parallel Data Fetch
- Fetch data from Source A (async)
- Fetch data from Source B (async)
- Wait for both to complete

### Step 2: Merge and Process
- Combine results
- Process merged data
```

**技巧五：提前退出**

```markdown
## Execution Steps
### Step 1: Validate Input
If input is invalid:
  → Exit early with clear error message
  → Don't proceed to expensive steps

### Step 2: Check Cache
If result already exists and is fresh:
  → Return cached result
  → Skip computation
```

## 11. 技能系统最佳实践

### 11.1 技能命名规范

统一的命名规范让 Skill 更容易被发现和管理。

**命名结构：**

```
[domain]-[action]-[target]

示例:
  web-search-news        # Web 领域 - 搜索 - 新闻
  code-review-python     # 代码领域 - 审查 - Python
  data-clean-csv         # 数据领域 - 清洗 - CSV
  deploy-docker-prod     # 部署领域 - Docker - 生产环境
```

**命名原则：**

1. 全部小写，用短横线连接
2. 动作动词用原形（search, review, clean, deploy）
3. 避免缩写（用 docker 而非 dk，用 kubernetes 而非 k8s）
4. 版本用目录或后缀区分（`my-skill-v2/` 或 `my-skill-advanced/`）

### 11.3 定期审计与清理

Skill 会不断积累，定期审计能保持系统高效。

**月度审计清单：**

```bash
# 1. 查看使用统计
$ hermes skills list --stats

# 2. 找出从未使用的 Skill
$ hermes skills list --unused --older-than 30d

# 3. 找出重复功能的 Skill
$ hermes skills list --duplicates

# 4. 检查过时的 Skill（很久没有更新）
$ hermes skills list --outdated

# 5. 清理
$ hermes skills archive unused-skill    # 归档（不删除）
$ hermes skills delete --unused         # 删除未使用的
```

**审计后的处理：**

- **高频使用**：确保是最新版本，考虑优化
- **偶尔使用**：保持现状
- **从未使用**：归档或删除
- **功能重复**：合并或保留最好的一个
- **已过时**：更新或替换
